# Мотиватор — фронтенд (v1.0 / MVP)

Минимальная сборка для деплоя на **Vercel**: React (Vite), Tailwind, Supabase Auth, клиентское шифрование vault (AES-GCM + PBKDF2 по seed).

**Целевой объём MVP для разработки** (релиз продукта **v1.0.0**) зафиксирован в [`obsidian-motivator/16-TZ-MVP-v1.0.md`](../obsidian-motivator/16-TZ-MVP-v1.0.md); поэтапный план внедрения — [`obsidian-motivator/17-План-реализации-MVP.md`](../obsidian-motivator/17-План-реализации-MVP.md).

Версия пакета в [`package.json`](./package.json) (**`version`**) — semver приложения (сейчас **1.0.0** для ветки MVP).

Подробная спецификация раннего запуска и деплоя: [`obsidian-motivator/14-V1-минимальный-запуск-Vercel.md`](../obsidian-motivator/14-V1-минимальный-запуск-Vercel.md).

## Возможности (сводка)

| Область | Реализация |
|--------|-------------|
| **Локализация** | **i18next** + **react-i18next**, локали `src/i18n/locales/` (`ru`, `en`), язык по умолчанию `ru`, ключ в `localStorage`: **`motivator_lang`**. Переключатель на экране настроек. |
| **Vault** | Один JSON на пользователя, шифруется целиком; версия схемы **`schemaVersion: 4`**. Миграции v1→v2→v3→v4 на клиенте: **`src/vault/normalize.ts`**. |
| **Вкладки `/app`** | **День** — план на дату + бэклог; **Неделя** — таймблокинг (понедельник–воскресенье, сетка часов, слоты по времени начала/конца и оценке); **Месяц** — календарная сетка и счётчик задач по дням (клик по дню открывает вкладку «День»). |
| **Фильтры вида** | Группа; включение/выключение уровней приоритета 1–5; заглушка «Повторы» до отдельной фазы; текстовый **информер** активных условий. |
| **Пересечения слотов** | При изменении даты плана, оценки или времени в модалке задачи — если интервал пересекается с другой незавершённой задачей на тот же день, показывается подтверждение (`lib/timeblocking.ts`, ключ `app.overlapConfirm`). |
| **Задача** | Заголовок, выполнено, группа, цвет (палитра), **приоритет 1–5**, чек-лист («план работы»), дата плана **или** только бэклог, оценка времени (мин), **время начала XOR время завершения** (или без времени). Родитель нельзя закрыть, пока открыты пункты чек-листа. Удаление с непустым чек-листом — подтверждение (`app.confirmDeleteTaskWithChecklist`). |
| **Настройки** | Пять **названий уровней приоритета** (хранятся в vault), группы как раньше. |
| **Группы** | Список групп в vault, фильтр по группе на `/app`, CRUD групп в настройках (группу по умолчанию нельзя удалить). |

Сборка: в **`vite.config.ts`** задан `build.chunkSizeWarningLimit` (ослабление предупреждения Vite о размере одного чанка).

## Supabase

1. Создайте проект в [Supabase](https://supabase.com/).
2. В SQL Editor выполните миграцию [`supabase/migrations/001_user_vault.sql`](./supabase/migrations/001_user_vault.sql).
3. Authentication → включите Email / пароль (если ещё не включено для вашего проекта).
4. Для разработки без писем: отключите обязательное подтверждение email (или подключите SMTP). Встроенная почта Supabase имеет rate limit.

## Seed и повторный вход

- После **выхода** из аккаунта seed удаляется из браузера. Чтобы открыть те же задачи, на экране **«Ключ шифрования»** выберите **«У меня уже есть seed»** и вставьте сохранённый base64 + тот же пароль деривации (если использовали).
- Новый сгенерированный seed после выхода **не** расшифрует старый vault на сервере.

## Vercel

1. Импортируйте репозиторий, **Root Directory** → `web`.
2. Environment Variables: те же `VITE_SUPABASE_*`, что в `.env.local`.
3. Build: `npm run build`, Output: `dist` (по умолчанию для Vite).

Файл [`vercel.json`](./vercel.json) отправляет все пути на `index.html` для SPA.

## PWA (установка на домашний экран)

В production-сборке подключены **manifest** и **service worker** (`vite-plugin-pwa`): кэшируются статические ассеты приложения, для SPA настроен **navigate fallback** на `/index.html`.

Иконки `public/pwa-192.png` и `public/pwa-512.png` можно перегенерировать скриптом:

`powershell -ExecutionPolicy Bypass -File scripts/generate-pwa-icons.ps1`

В режиме `npm run dev` SW **выключен** (`devOptions.enabled: false`), чтобы не мешать разработке.

## Синхронизация vault

До завершения первой загрузки с Supabase поля задач **неактивны** (нет гонки с созданием пустого vault). Строка статуса под заголовком: загрузка / сохранение / время последнего успешного сохранения. При ошибке расшифровки показывается сообщение, редактирование блокируется (флаг **`decryptFailed`**).

## Контракт vault (актуальная схема)

Хранится как один зашифрованный JSON. Актуальная версия — **4** (см. `src/vault/types.ts`). Кратко:

- **`priorityLabels`**: пользовательские строки для уровней **1–5**.
- **`groups`**: проекты/группы.
- **`tasks`**: для каждой задачи — **`priorityRank`** (1–5), **`scheduledLocalDate`** (`YYYY-MM-DD` или `null` = только бэклог), **`checklist`**, **`estimatedMinutes`**, **`timeMode`** (`none` | `start` | `end`), **`timeMinutesFromMidnight`**, плюс группа, цвет, заголовок, выполнено.

Цепочка миграций: снимки **v1**, **v2**, **v3** (старые поля уровней 1–3 и Эйзенхауэр) при открытии приводятся к **v4** через **`normalizeVault`**.

### Миграция v3 → v4 (поля)

| Было (v3) | Стало (v4) |
|-----------|------------|
| `prioritySystem` + `priorityLevel` / `eisenhowerQuadrant` | одно поле **`priorityRank`** (эвристика: уровни 1–3 → ранги 1–3; квадранты q1…q4 → ранги 1,2,4,5; Inbox → 3) |
| `subtasks` | **`checklist`** (тот же массив объектов) |
| — | **`scheduledLocalDate`**: для всех мигрированных задач **`null`** (бэклог); пользователь заново разносит по дням |
| — | **`estimatedMinutes`**, **`timeMode`**, **`timeMinutesFromMidnight`**: по умолчанию пусто / `none` / `null` |
