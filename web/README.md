# Мотиватор — веб-клиент

Клиент для деплоя на **Vercel**: React (Vite), Tailwind, Supabase Auth, клиентское шифрование vault (AES-GCM + PBKDF2 по seed). Номер версии **`0.*.*`** не означает «только черновик» — см. [версионирование](#версионирование).

**Целевой объём MVP для разработки** (релиз продукта **v1.0.0**) зафиксирован в [`obsidian-motivator/16-TZ-MVP-v1.0.md`](../obsidian-motivator/16-TZ-MVP-v1.0.md); поэтапный план внедрения — [`obsidian-motivator/17-План-реализации-MVP.md`](../obsidian-motivator/17-План-реализации-MVP.md).

**Текущая версия веб-клиента** в репозитории: **`0.7.3`** ([`package.json`](./package.json)) — см. правила ниже. Краткий обзор модалки **«Краткая сводка»** — в [отдельном разделе](#краткая-сводка); источник данных — [`web/src/data/productRoadmap.ts`](./src/data/productRoadmap.ts).

### Версионирование

**После `1.0.0`** ориентируемся на классический **SemVer** `MAJOR.MINOR.PATCH`:

| Часть | Когда повышать | Пример |
|--------|----------------|--------|
| **MAJOR** | Обратно несовместимые изменения: ломающая миграция данных/контракта, новая архитектура, удаление опубликованных возможностей | `1.0.0` → `2.0.0` |
| **MINOR** | Новая функциональность, совместимая с предыдущей версией | `1.3.0` → `1.4.0` |
| **PATCH** | Исправления ошибок и мелкие улучшения без новых фич | `1.4.5` → `1.4.6` |

**До `1.0.0`:** в [`package.json`](./package.json) хранится строка **`0.x.y`** — три числа, как требует **SemVer 2.0** и npm (четвёртое число в одном поле **нельзя** записать без выхода из спецификации). Смысл для команды — **«ноль и три уровня»** **`0 · x · y`**, плюс **четвёртый идентификатор** в виде **метаданных сборки**:

| Уровень | В `package.json` | Смысл |
|---------|------------------|--------|
| **0** | первая цифра всегда **0** до релиза **`1.0.0`** | Это **эпоха нумерации**, а **не** оценка качества. Приложение на **`0.10.x`** может быть уже **полнофункциональным, стабильным и «продуктовым»** — ограничивает только принятое правило версий, а не зрелость UX или объём фич. **Не путать** с ярлыком «только MVP / прототип». |
| **x** | **MINOR** (у нас **`N`**) | Номер **последней закрытой** фазы из [`obsidian-motivator/17-План-реализации-MVP.md`](../obsidian-motivator/17-План-реализации-MVP.md). В дорожной карте приложения — фазы **0…13** до **`1.0.0`**; после закрытия фазы поднимают **`x`** (например после фазы 7 → **`0.7.0`**). |
| **y** | **PATCH** | Правки **внутри** текущего **`x`**: багфиксы, полировка, документация без смены фазы. |
| **z** | не отдельное число в JSON | В production-сборке к semver добавляется **`+<git short SHA>`** (см. `vite.config.ts`, `APP_VERSION`). Полная метка билда: **`0.x.y+abc1234`** — это и есть **четвёртый слой** для различения коммитов без bump **`y`** на каждый push. |

Переход на **`1.0.0`** — когда закрыта **фаза 13** (последняя в дорожной карте: **дизайн и адаптивность**; перед ней — домен **11**, ветки **12**, чеклист **10**) по ТЗ/Scope и пройдена регрессия: формально «первая стабильная линия» и публичный контракт версий, **не** обязательно первый день, когда приложение стало хорошим (оно могло быть таким ещё на **`0.x.y`**).

Ориентировочная шкала (не заменяет правил выше):

| Тип | Версия | Когда | Пример |
|-----|--------|--------|--------|
| Начало разработки | `0.1.0` | Первый осмысленный билд | Старт MVP |
| Фича / закрытая фаза | `0.2.0`, … | Milestone по фазам плана | Закрыли фазу 2 |
| Баги без новой фазы | `0.2.1`, … | После тестирования внутри той же фазы | Фиксы таймблокинга |
| Pre-release | `0.3.0-beta.1` | По необходимости до 1.0.0 | Опционально |
| Первый публичный | **`1.0.0`** | Готовность по Scope/TЗ | Официальный запуск |

Пакет **`@motivator/core`** имеет **собственный** `version` в `packages/motivator-core/package.json`: там MINOR/PATCH завязаны на **совместимость API и схемы vault**, а не обязаны совпадать с версией веб-клиента.

**Сборка:** при **`vite build`** к semver добавляется **`+<git short SHA>`** (из корня репозитория). Строка в интерфейсе (`APP_VERSION` в `src/version.ts`) различает сборки без обязательного bump **`PATCH`** на каждый коммит.

**Текущее значение** в репозитории берётся из `package.json`; его поддерживают вручную при закрытии фазы или при серии патчей.

Подробная спецификация раннего запуска и деплоя: [`obsidian-motivator/14-V1-минимальный-запуск-Vercel.md`](../obsidian-motivator/14-V1-минимальный-запуск-Vercel.md).

## Краткая сводка

**«Краткая сводка»** доступна **всем залогиненным**: с вкладки **«Общие»** на **`/settings`**; для **admin** — также страница **`/admin/roadmap`** (пункт **`ShellAdminNav`**, без модалки в nav). Для остальных ролей — модалка (**`ProductRoadmapModal`**) по кнопке в Общих. Контент панели — **[`ProductRoadmapPanel.tsx`](./src/components/ProductRoadmapPanel.tsx)** (общий для страницы и модалки). Данные — **[`productRoadmap.ts`](./src/data/productRoadmap.ts)**; «Открытые вопросы» — **i18n** `settings.roadmapOpenQuestion*`.

### Содержимое модалки (шесть блоков сверху вниз)

1. **Верх:** кольцевая диаграмма прогресса до **MVP 1.0.0** — доля закрытых фаз **0–6** из **0–13** и подпись с процентом. Источник чисел — те же константы, что у блоков **«Уже реализовано»** и **«План до 1.0.0»** (без дублирования «магических» процентов в разметке).
2. **Уже реализовано** — массив **`IMPLEMENTED_MVP_PHASES`**: фазы **0–6**, у каждой краткое описание и раскрывающийся список **«Подробнее»** (`detailBullets`). Назначение: зафиксировать **фактическое** поведение текущей сборки; по возможности согласовать формулировки с пунктами **`RELEASE_NOTES_BLOCKS`** за соответствующие периоды.
3. **План до 1.0.0** — **`MVP_PHASES_PLANNED`**: фазы **7–13** (порядок: **7** настройки/аккаунт → **8** монетизация → **9** offline-first → **10** чеклист → **11** домен → **12** ветки `dev`→`main` → **13** дизайн); смысл — [`obsidian-motivator/17-План-реализации-MVP.md`](../obsidian-motivator/17-План-реализации-MVP.md). У фазы может быть **`plain`** и **`detailBullets`**.
4. **Релиз-ноты** — **`RELEASE_NOTES_BLOCKS`**: хронология по **календарным датам** внесения изменений в продукт (`dateLabel`, **`YYYY-MM-DD`**). В модалке блоки дат идут **по убыванию** (самый свежий календарный день выше). Внутри одной даты — один или несколько выпусков **`items`**; у выпуска обязательно **`releasedInVersion`** (semver из [`web/package.json`](./package.json) на момент выпуска, **без** суффикса **`+git`**), массив **`changes`**, опционально **`plainBullets`** под раскрывашкой **«Подробности простым языком»**. Сюда же попадают **заметные для пользователя** ограничения планировщика (например невозможность выбрать прошлую **плановую дату** или время на **сегодня** раньше текущего момента) — дублирующая сводка по поведению есть в таблице **«Возможности (сводка)»** выше. Правила ведения массива — в **комментарии над `RELEASE_NOTES_BLOCKS`** в `productRoadmap.ts`, в [правиле репозитория](../.cursor/rules/pre-commit-docs-roadmap.mdc) и в скилле **[`../.cursor/skills/pre-commit-docs-roadmap/SKILL.md`](../.cursor/skills/pre-commit-docs-roadmap/SKILL.md)** (перед коммитом агент проходит скилл целиком, краткий `.mdc` его **не заменяет**).
5. **Идеи на потом** — **`IDEAS_LATER_ENTRIES`**: черновой backlog после **1.0.0**; в модалке **раскрываемые тематические группы** с **числом идей** в заголовке секции и группы (`groupIdeasLaterForDisplay`); полный пул идей вне модалки — [`obsidian-motivator/15-Идеи-для-развития.md`](../obsidian-motivator/15-Идеи-для-развития.md).
6. **Открытые вопросы** — продуктовые заметки без срока: строки **`settings.roadmapOpenQuestion*`** в **`web/src/i18n/locales/ru.json`** и **`en.json`** (заголовок секции — **`settings.roadmapOpenQuestions`**, подсказка — **`settings.roadmapOpenQuestionsHint`**); разметка списка в **`ProductRoadmapModal.tsx`**.

## Seed, вход и восстановление ключа

**Три разных секрета:** (1) **пароль входа** — Supabase, только сессия; (2) **seed (base64)** + (3) **пароль KDF** (может быть пустым) — шифрование vault в браузере. Смена пароля входа в `/settings` **не** меняет seed/KDF.

**Сейчас в сборке (`/onboarding`, DR-014 / 7a.1):** после `/login` — редирект на `/app`; `RequireVault` отправляет на `/onboarding`, если локального seed нет. Режим выбирается по наличию строки **`user_vault`** на сервере: **восстановление** (только импорт seed+KDF) или **первичная настройка** (генерация + чекбокс «сохранил seed»). Экспорт seed — `/settings` (копия, `.txt`, QR, DR-006). При `decryptFailed` на `/app` и в настройках — баннер и **«Ввести seed заново»** без выхода из аккаунта. Выход — с подтверждением (seed удаляется из браузера). Регистрация — согласие на ПД; вход — «Забыли пароль?» (Supabase reset).

**Зафиксировано продуктом ([[DR-014]](../obsidian-motivator/12-Журнал-решений.md#dr-014--онбординг-seed-два-экрана-и-восстановление-ключа-фаза-7)); в плане дальше:**

| После успешного входа | Экран | Действия пользователя |
|------------------------|-------|------------------------|
| Seed+KDF уже в браузере | — | Сразу `/app` |
| На сервере **есть** vault, локального seed нет | **Восстановление ключа** | Только вставка seed + KDF; **без** генерации нового seed в основном потоке |
| Vault на сервере **нет** | **Первичная настройка** | Генерация seed + подтверждение сохранения (DR-006) до `/app` |

**Восстановление** = пользователь вводит **сохранённую** пару seed+KDF (не восстановление поддержкой). Резервная копия — экспорт в настройках. После **выхода** seed удаляется из браузера (`lock()`); снова нужен импорт той же пары. При ошибке расшифровки (`decryptFailed`) — баннер и форма повторного ввода seed+KDF на месте. Новый seed при существующем vault на сервере **не** открывает старые данные.

**Полная ротация seed/KDF (DR-014, план 7c):** в `/settings` при успешной расшифровке — новый seed и/или новый пароль KDF, локальная перешифровка vault, замена ciphertext на сервере; обязательный бэкап **нового** seed (DR-006). Старый seed не открывает актуальные данные. **Не в 7a/7a.1:** удаление аккаунта + 30 дней (7b).

## Vercel

Монорепозиторий: задайте **корень репозитория** как корень проекта в Vercel (не только `web`), иначе workspace-пакет `@motivator/core` не попадёт в сборку.

1. **Root Directory** → корень репозитория (`planner/`).
2. **Install Command**: `npm install` (дублируется в корневом [`vercel.json`](../vercel.json), чтобы совпадало с CI).
3. **Build Command**: `npm run build -w web` (там же в **`vercel.json`**).
4. **Output Directory**: **`web/dist`** — не **`dist`** в корне: Vite пишет сборку в **`web/dist`**. В **`vercel.json`** задано **`outputDirectory`**: **`web/dist`**, иначе Vercel ищет **`dist`** и падает с *No Output Directory named "dist"*.
5. Environment Variables: те же `VITE_SUPABASE_*`, что в `.env.local`, плюс **`VITE_VAPID_PUBLIC_KEY`**; для минутного **`send-due`** — см. [«Минутный вызов send-due»](#минутный-вызов-send-due) (**`CRON_SECRET`**, **`SUPABASE_SEND_DUE_URL`**, **`SUPABASE_CRON_ANON_KEY`** на Vercel + внешний cron, например [cron-job.org](https://cron-job.org)).

Если по историческим причинам корень проекта в Vercel остаётся только `web`, перед сборкой нужно вручную обеспечить доступ к `packages/motivator-core` (например, скопировать структуру монорепозитория или собирать из полного clone с установкой из родительской папки).

Сборка рассчитана на **Vite 7** (Rollup): в корне репозитория в `package.json` задан **`overrides`** на `vite@7.3.3`, а `@vitejs/plugin-react` — ветка **5.x**, совместимая с Vite 7. Так на Vercel/Linux не подтягивается Vite 8 с нативным **Rolldown**, из‑за которого часто возникает `MODULE_NOT_FOUND` при загрузке нативного binding Rolldown.

Файл [`vercel.json`](../vercel.json) в **корне** репозитория (`planner/`) задаёт **`installCommand`**, **`buildCommand`** и **`outputDirectory`**: **`web/dist`**, отправляет все пути на `index.html` для SPA и **короткий HTTP-кэш** для **`sw.js`**, **`manifest.webmanifest`**, **`workbox-*.js`**. Минутный вызов **`send-due`** — через serverless [`api/send-due-cron.js`](../api/send-due-cron.js); ежедневная очистка черновиков дефектов — опционально через [`api/defect-attachments-cleanup-cron.js`](../api/defect-attachments-cleanup-cron.js) (см. [ниже](#прокси-defect-attachments-cleanup-vercel)). Оба сценария — **внешний** cron или **Pro**-`crons` (см. [«Минутный вызов send-due»](#минутный-вызов-send-due)); на **Hobby** встроенный Cron **не** даёт минутного расписания / `* * * * *` может **ломать деплой**.

## Доступ из РФ (без VPN)

**Симптом (GitHub #37):** в РФ приложение и **Web Push** часто работают только через VPN: не открывается SPA, не проходит вход, не синхронизируется vault, не приходят напоминания.

**Почему:** клиент завязан на **два внешних контура**, оба могут быть недоступны или нестабильны из российских сетей:

| Контур | Сейчас | Что ломается без доступа |
|--------|--------|---------------------------|
| **Фронт + PWA + cron-прокси** | **Vercel** (`*.vercel.app`, прод: `https://planner-tawny-omega.vercel.app`) | Не грузится UI, service worker, `/api/send-due-cron` |
| **Auth, vault, push, Edge** | **Supabase** (`https://ntpkveicqetjjvlnfrwc.supabase.co`) | Нет входа, нет ciphertext, нет `send-due` / подписок |

Один **свой домен** на Vercel (фаза 12) **не гарантирует** доступ из РФ, если блокируется сам хостинг или маршрут до Supabase. Нужно осознанное решение по инфраструктуре.

**Варианты (по возрастанию объёма работ):**

1. **Временно VPN** — для команды/беты; не продуктовое решение для ру-сегмента.
2. **Свой домен + HTTPS на Vercel** (фаза 12) + **Custom Domain для Supabase API** — иногда улучшает доступность; проверить с реальных сетей РФ до обещаний пользователям.
3. **Обратный прокси в РФ** (Selectel, VK Cloud, Yandex Cloud и т.п.): один origin для SPA (статика из `web/dist` или прокси на Vercel) и **отдельный** прокси/host для API Supabase; cron **`send-due`** — с узла в РФ или через тот же прокси. Секреты и TLS — по чеклисту команды.
4. **Зеркало статики в РФ** (Object Storage + CDN) с тем же билдом, **`VITE_SUPABASE_URL`** на доступный из РФ endpoint (п. 2 или 3) — UI без VPN, данные всё равно через API.
5. **Перенос бэкенда** (Postgres + Auth + Edge-аналог на российском облаке) — максимальный объём, после MVP; см. идею в «Краткой сводке» (*Доступность для РФ*).

**В приложении (0.6.63+):** при типичных сетевых ошибках (`Failed to fetch` и т.п.) на `/app` и входе показывается подсказка со ссылкой на этот раздел.

**Следующий шаг команды:** выбрать вариант **2–4** для пилота, зафиксировать DNS/прокси в runbook и закрыть #37 после выката; до выката issue остаётся **открытым** с комментарием и ссылкой на PR с доками/UI.

## PWA (установка на домашний экран)

В production-сборке подключены **manifest** и **service worker** (`vite-plugin-pwa`): кэшируются статические ассеты приложения, для SPA настроен **navigate fallback** на `/index.html`.

Регистрация SW выполняется из кода — **`initPwaServiceWorker()`** в [`src/lib/pwaServiceWorker.ts`](./src/lib/pwaServiceWorker.ts) (импорт в [`src/main.tsx`](./src/main.tsx)); в **`vite.config.ts`** у плагина **`injectRegister: null`**, чтобы не было **двойной** регистрации (раньше в `index.html` дополнительно подключался `registerSW.js`). Режим **`registerType: 'autoUpdate'`** по-прежнему активирует новый worker и перезагружает страницу, когда на сервере появилась новая сборка. Плюс явно вызывается **`registration.update()`** при возврате на вкладку / в окно и **раз в час**, чтобы мобильный PWA не ждал только внутреннего расписания браузера.

Иконки `public/pwa-192.png` и `public/pwa-512.png` можно перегенерировать скриптом:

`powershell -ExecutionPolicy Bypass -File scripts/generate-pwa-icons.ps1`

В режиме `npm run dev` SW **выключен** (`devOptions.enabled: false`), чтобы не мешать разработке.

## Уведомления (Web Push)

- **Режим** хранится в vault: **`notificationPreferences.deliveryMode`** — **`off`** (по умолчанию), **`hybrid`** (на Supabase синхронизируются время, тип события и идентификатор задачи; **без** названия задачи в открытом виде; тексты push на сервере **различают** начало окна и конец), **`full`** (на сервер могут попадать **названия задач** в открытом виде). Настройки и кнопки — экран **`/settings`**.
- **Таблицы Supabase** (миграция [`web/supabase/migrations/002_notifications.sql`](./supabase/migrations/002_notifications.sql)): **`push_subscriptions`**, **`notification_fire_requests`** (RLS: только свой `user_id`). После правок vault клиент с **debounce** пересчитывает окна срабатываний по задачам с **`timeMode`** `start` | `end` и **вхождением в план на день** (разовые даты и **повторы** — `taskOccursOnDate`, как на вкладке «День») в горизонте 14 дней, добавляет при необходимости **одно** срабатывание **`eod_reminder`** на **сегодня** (если включён ритуал EOD, задано время напоминания, день ещё не в **`eodCompletedLocalDates`**, режим не **`off`**, момент в будущем), и заменяет будущие строки со статусом **`scheduled`**. Для значения **`kind` = `eod_reminder`** в проде нужна миграция [`web/supabase/migrations/004_notification_fire_kind_eod.sql`](./supabase/migrations/004_notification_fire_kind_eod.sql) (расширение `CHECK` на колонке **`kind`**).

[«Прокси defect-attachments-cleanup»](#прокси-defect-attachments-cleanup-vercel)); **`admin-motivator-roles`** — POST с JWT, список пользователей Auth и смена **`motivator_role`** только для вызывающего с ролью **`admin`** (те же базовые секреты Supabase, см. [Edge **`admin-motivator-roles`**](#edge-admin-motivator-roles)).
- **Cron / минутный вызов `send-due`:** без периодического POST на Edge **`send-due`** запланированные строки не превращаются в push. **Рекомендуемый путь на Vercel Hobby:** эндпоинт [`api/send-due-cron.js`](../api/send-due-cron.js) + **внешний** HTTP-cron ([cron-job.org](https://cron-job.org) и аналоги) на **`https://<ваш-домен>/api/send-due-cron`** раз в минуту (см. ### Минутный вызов send-due

Serverless в корне репозитория: [`api/send-due-cron.js`](../api/send-due-cron.js). В [`vercel.json`](../vercel.json) **нет** секции **`crons`**: на тарифе **Vercel Hobby** встроенный Cron **не** даёт расписание «раз в минуту» (до **раза в сутки**), а выражение **`* * * * *`** может **ломать деплой**. Минутный тик настраивается **внешним** HTTP-cron.

**Задокументированный прод-URL Vercel для этого тика:** **`https://planner-tawny-omega.vercel.app/api/send-due-cron`** — прокси только на Edge **`send-due`** (переменные **`SUPABASE_SEND_DUE_URL`** / **`SUPABASE_CRON_ANON_KEY`** на стороне Vercel). Очистка черновиков дефектов — **отдельное** задание: прямой POST на **`https://ntpkveicqetjjvlnfrwc.supabase.co/functions/v1/defect-attachments-cleanup`** или тот же домен Vercel через **[`api/defect-attachments-cleanup-cron.js`](../api/defect-attachments-cleanup-cron.js)** (см. [«Прокси defect-attachments-cleanup»](#прокси-defect-attachments-cleanup-vercel)).

#### 1. Переменные на Vercel (как минимум Production)

| Переменная | Значение |
|------------|----------|
| **`CRON_SECRET`** | Случайная длинная строка. Любой вызов **`/api/send-due-cron`** должен передавать **`Authorization: Bearer <CRON_SECRET>`** (cron-job.org и ручной `curl`). |
| **`SUPABASE_SEND_DUE_URL`** | `https://ntpkveicqetjjvlnfrwc.supabase.co/functions/v1/send-due` (для этого проекта; иначе подставьте свой **`<REF>`**) |
| **`SUPABASE_CRON_ANON_KEY`** | Legacy **anon** `public` (JWT `eyJ…`) из Supabase **Settings → API** — тот же смысл, что **`VITE_SUPABASE_ANON_KEY`**. |

После сохранения — **Redeploy**.

#### 2. [cron-job.org](https://cron-job.org) (или аналог)

1. Зарегистрироваться / войти. **Create cronjob**.  
2. **URL:** **`https://planner-tawny-omega.vercel.app/api/send-due-cron`** (текущий прод-деплой) или `https://<ваш-проект>.vercel.app/api/send-due-cron` / кастомный домен на Vercel.  
3. **Schedule:** каждую минуту (например `* * * * *` в расширенном режиме, если сервис предлагает).  
4. **Request method:** **GET** или **POST** (оба поддерживает `send-due-cron.js`).  
5. **Headers:** добавить **`Authorization`** со значением **`Bearer <ваш CRON_SECRET>`** (ровно тот же секрет, что в Vercel).  
6. При необходимости **Body** для POST: `{}` и **Content-Type:** `application/json`.  
7. Сохранить и **включить** задачу. Проверка: логи задания на cron-job.org и ответ **`{"processed":…,"sent":…}`** либо ошибка в теле.

На **Pro** можно снова добавить в **`vercel.json`** массив **`crons`** с путём **`/api/send-due-cron`** и расписанием **`* * * * *`** — тогда Vercel сам будет слать запрос с **`Authorization: Bearer <CRON_SECRET>`** ([документация](https://vercel.com/docs/cron-jobs)).

#### Ручной вызов (PowerShell)

```powershell

```

3. **Schedule:** **раз в сутки** (или реже; функция удаляет черновики старше **24 ч**).  
4. **Request method:** **GET** или **POST**.  
5. **Headers:** **`Authorization: Bearer <CRON_SECRET>`** (как на Vercel).  
6. Для POST при необходимости **Body:** `{}`, **Content-Type:** `application/json`.  
7. Ожидаемый ответ: **`{"removed_rows":…,"removed_files":…}`**.

## Синхронизация vault

До завершения первой загрузки с Supabase поля задач **неактивны** (нет гонки с созданием пустого vault). Строка статуса под заголовком: загрузка / сохранение / время последнего успешного сохранения. При ошибке расшифровки показывается сообщение, редактирование блокируется (флаг **`decryptFailed`**).

Запись ciphertext на сервер **отложена** (debounce, см. `VAULT_REMOTE_SAVE_DEBOUNCE_MS` в пакете `@motivator/core`): локальное состояние обновляется сразу, сеть — после паузы; перед выходом из аккаунта выполняется принудительный flush.

## Контракт vault и криптография

Спецификация формата ciphertext, PBKDF2/AES-GCM и золотой вектор KDF для паритета с нативными клиентами: [`packages/motivator-core/docs/VAULT_CRYPTO_CONTRACT.md`](../packages/motivator-core/docs/VAULT_CRYPTO_CONTRACT.md).

## Контракт vault (актуальная схема)

Хранится как один зашифрованный JSON. Актуальная версия — **8** (типы в `@motivator/core`, модуль `vault/types`). Кратко:

- **`priorityLabels`**: пользовательские строки для уровней **1–5**.
- **`groups`**: проекты/группы.
- **`tasks`**: для каждой задачи — **`priorityRank`** (1–5), **`scheduledLocalDate`** (`YYYY-MM-DD` или `null`), **`checklist`** (массив в порядке отображения сверху вниз; новые пункты добавляются в конец), **`estimatedMinutes`**, **`timeMode`** (`none` | `start` | `end`), **`timeMinutesFromMidnight`**, **`recurrence`** / **`recurrenceAnchorLocalDate`** (правило повтора: каждый день / каждые N дней / дни недели, и якорь серии), **`completedOccurrenceLocalDates`** — для повторов список дней, на которые отмечено выполнение вхождения (глобальный **`done`** при повторе по-прежнему может архивировать серию целиком). Идентификатор серии для отчётов (**DR-008**) совпадает с **`id`** задачи. Плюс группа, цвет, заголовок, **`done`**, опционально **`includeInEodRitual`** (участие в ритуале EOD). **DR-004:** опционально **`doubleConfirmEnabled`**, **`doubleConfirmIntervalMinutes`**, **`doubleConfirmGraceMinutes`**, **`doubleConfirmPending`** (ожидание второго подтверждения за конкретный календарный день).
- **`drafts`**: черновики формы создания задачи; запись выполняется только при закрытии модалки без сохранения и при **фактических** изменениях формы относительно состояния при открытии (см. раздел «Создание задачи» в таблице возможностей).
- **`eodCompletedLocalDates`**, **`eodPreferences`**: ритуал End-of-Day (схема **v6+**).
- **`notificationPreferences`** (**v8+**): **`deliveryMode`** — **`off`** | **`hybrid`** | **`full`** (Web Push; см. раздел **«Уведомления (Web Push)»** в этом README).

Цепочка миграций: снимки **v1** … **v7** при открытии приводятся к **v8** через **`normalizeVault`** (например старые vault без повтора получают **`recurrence: null`**, **`drafts: []`**; до **v6** добавляются поля EOD по умолчанию; **v7** — поля двойного подтверждения **DR-004**; **v8** — **`notificationPreferences`** с режимом доставки push по умолчанию **`off`**).

### Миграция v3 → v4 (поля)

| Было (v3) | Стало (v4) |
|-----------|------------|
| `prioritySystem` + `priorityLevel` / `eisenhowerQuadrant` | одно поле **`priorityRank`** (эвристика: уровни 1–3 → ранги 1–3; квадранты q1…q4 → ранги 1,2,4,5; Inbox → 3) |
| `subtasks` | **`checklist`** (тот же массив объектов) |
| — | **`scheduledLocalDate`**: для всех мигрированных задач **`null`** (бэклог); пользователь заново разносит по дням |
| — | **`estimatedMinutes`**, **`timeMode`**, **`timeMinutesFromMidnight`**: по умолчанию пусто / `none` / `null` |
