# Мотиватор — веб-клиент

Клиент для деплоя на **Vercel**: React (Vite), Tailwind, Supabase Auth, клиентское шифрование vault (AES-GCM + PBKDF2 по seed). Номер версии **`0.*.*`** не означает «только черновик» — см. [версионирование](#версионирование).

**Целевой объём MVP для разработки** (релиз продукта **v1.0.0**) зафиксирован в [`obsidian-motivator/16-TZ-MVP-v1.0.md`](../obsidian-motivator/16-TZ-MVP-v1.0.md); поэтапный план внедрения — [`obsidian-motivator/17-План-реализации-MVP.md`](../obsidian-motivator/17-План-реализации-MVP.md).

**Текущая версия веб-клиента** в репозитории: **`0.6.7`** ([`package.json`](./package.json)) — см. правила ниже. По дорожной карте в приложении (**модалка «Краткая сводка»** в настройках, `web/src/data/productRoadmap.ts`): **реализовано** — фазы **0–6** (включая ритуал End-of-Day, **DR-004** и анимации мини-карточки); **запланировано до 1.0.0** — фазы **7–11**; отдельно — **релиз-ноты**, **«Идеи на потом»**, блок **«Открытые вопросы»**.

### Версионирование

**После `1.0.0`** ориентируемся на классический **SemVer** `MAJOR.MINOR.PATCH`:

| Часть | Когда повышать | Пример |
|--------|----------------|--------|
| **MAJOR** | Обратно несовместимые изменения: ломающая миграция данных/контракта, новая архитектура, удаление опубликованных возможностей | `1.0.0` → `2.0.0` |
| **MINOR** | Новая функциональность, совместимая с предыдущей версией | `1.3.0` → `1.4.0` |
| **PATCH** | Исправления ошибок и мелкие улучшения без новых фич | `1.4.5` → `1.4.6` |

**До `1.0.0`:** в [`package.json`](./package.json) хранится строка **`0.x.y`** — три числа, как требует **SemVer 2.0** и npm (четвёртое число в одном поле **нельзя** записать без выхода из спецификации). Смысл для команды — **«ноль и три уровня»** **`0 · x · y`**, плюс **четвёртый идентификатор** в виде **метаданных сборки**:

| Уровень | В `package.json` | Смысл |
|---------|------------------|--------|
| **0** | первая цифра всегда **0** до релиза **`1.0.0`** | Это **эпоха нумерации**, а **не** оценка качества. Приложение на **`0.10.x`** может быть уже **полнофункциональным, стабильным и «продуктовым»** — ограничивает только принятое правило версий, а не зрелость UX или объём фич. **Не путать** с ярлыком «только MVP / прототип». |
| **x** | **MINOR** (у нас **`N`**) | Номер **последней закрытой** фазы из [`obsidian-motivator/17-План-реализации-MVP.md`](../obsidian-motivator/17-План-реализации-MVP.md). В дорожной карте приложения — фазы **0…11** до **`1.0.0`**; после закрытия фазы поднимают **`x`** (например после фазы 7 → **`0.7.0`**). |
| **y** | **PATCH** | Правки **внутри** текущего **`x`**: багфиксы, полировка, документация без смены фазы. |
| **z** | не отдельное число в JSON | В production-сборке к semver добавляется **`+<git short SHA>`** (см. `vite.config.ts`, `APP_VERSION`). Полная метка билда: **`0.x.y+abc1234`** — это и есть **четвёртый слой** для различения коммитов без bump **`y`** на каждый push. |

Переход на **`1.0.0`** — когда закрыта **фаза 11** (релизный чеклист) по ТЗ/Scope и пройдена регрессия: формально «первая стабильная линия» и публичный контракт версий, **не** обязательно первый день, когда приложение стало хорошим (оно могло быть таким ещё на **`0.x.y`**).

Ориентировочная шкала (не заменяет правил выше):

| Тип | Версия | Когда | Пример |
|-----|--------|--------|--------|
| Начало разработки | `0.1.0` | Первый осмысленный билд | Старт MVP |
| Фича / закрытая фаза | `0.2.0`, … | Milestone по фазам плана | Закрыли фазу 2 |
| Баги без новой фазы | `0.2.1`, … | После тестирования внутри той же фазы | Фиксы таймблокинга |
| Pre-release | `0.3.0-beta.1` | По необходимости до 1.0.0 | Опционально |
| Первый публичный | **`1.0.0`** | Готовность по Scope/TЗ | Официальный запуск |

Пакет **`@motivator/core`** имеет **собственный** `version` в `packages/motivator-core/package.json`: там MINOR/PATCH завязаны на **совместимость API и схемы vault**, а не обязаны совпадать с версией веб-клиента.

**Сборка:** при **`vite build`** к semver добавляется **`+<git short SHA>`** (из корня репозитория). Строка в интерфейсе (`APP_VERSION` в `src/version.ts`) различает сборки без обязательного bump **`PATCH`** на каждый коммит.

**Текущее значение** в репозитории берётся из `package.json`; его поддерживают вручную при закрытии фазы или при серии патчей.

Подробная спецификация раннего запуска и деплоя: [`obsidian-motivator/14-V1-минимальный-запуск-Vercel.md`](../obsidian-motivator/14-V1-минимальный-запуск-Vercel.md).

## Возможности (сводка)

| Область | Реализация |
|--------|-------------|
| **Локализация** | **i18next** + **react-i18next**, локали `src/i18n/locales/` (`ru`, `en`), язык по умолчанию `ru`, ключ в `localStorage`: **`motivator_lang`**. Переключатель на экране настроек. |
| **Vault** | Один JSON на пользователя, шифруется целиком; актуальная версия схемы **`schemaVersion: 7`** (**DR-004**: поля двойного подтверждения на задаче; ритуал EOD — поля с **v6**). Миграции v1→…→**v7** на клиенте: пакет **`@motivator/core`** (`normalizeVault`). |
| **Вкладки `/app`** | **День** — план на дату + бэклог; справа от списка (на широком экране) — **кольцо прогресса**: **`plannedDayCompletionWeights`** — знаменатель = **число задач в плане**; каждая задача без чек-листа даёт 0 или 1 по главной галочке дня; задача с чек-листом даёт **долю выполненных пунктов** (четыре пункта → один выполненный = **0,25** этой задачи). Список плана и расчёт кольца берут один набор задач — **`tasksScheduledForPlannerDay`** в **`@motivator/core`** (то же, что фильтр «есть вхождение на этот календарный день», без чистого бэклога). В UI — **процент** и строка вроде **«X из Y задач»** (доля выполнения по сумме вкладов задач); **`DayPlanDonut`**. На узком экране кольцо **выше** списка (`flex-col-reverse`). Выполненные остаются в тех же списках **ниже** невыполненных (приоритет и название внутри группы). **Неделя** — таймблокинг (понедельник–воскресенье, сетка часов, слоты по времени начала/конца и оценке); справа от сетки (на широком экране) — **кольцо «план недели»**: **`plannedPeriodProgress`** суммирует по дням недели ту же метрику; учитываются только дни с **датой ≤ сегодня**; действуют **те же фильтры**, что и у задач на вкладке; если в охвате **нет** ни одной задачи в плане — блок кольца **скрыт**. На узком экране кольцо **выше** сетки (`flex-col-reverse`). Таблица недели **без горизонтального скролла**: семь дней делят ширину контейнера (`minmax(0,1fr)`); область с часами при ограничении высоты прокручивается **по вертикали**. Текст-пояснение про слоты закреплён **под** сеткой (не внутри блока прокрутки по времени). **Месяц** — календарная сетка и счётчик задач по дням (клик по дню открывает вкладку «День»); **кольцо «план месяца»** — те же правила по дням текущего месяца до сегодня, фильтры, скрытие при нуле охвата; контейнер **`max-w-5xl`** по центру. Ссылка **«Отчёты»** в шапке ведёт на **`/app/reports`**. |
| **Отчёты (фаза 5 MVP)** | Считаются **на клиенте** после расшифровки vault (**DR-007**): окно **7 / 30** последних календарных дней (график — по полным дням до вчера); KPI «дни подряд» — **стрик DR-013** с учётом завершённого ритуала End-of-Day (`vault.eodCompletedLocalDates`, см. ниже); доля дней с ≥1 отметкой, сумма отметок; столбчатая диаграмма по дням; таблицы **часто проваленных** — повторы по **серии** (`task.id`, **DR-008**) и до **5 разовых** с просроченным планом в окне. У блоков KPI, гистограммы и таблиц — **иконки подсказок** (`ReportHint`, нативный `title` при наведении). Логика — **`@motivator/core`** (`lib/reporting/vaultAnalytics.ts`). |
| **Фаза 6: End-of-Day, DR-004, анимации** | **EOD:** в шапке `/app` кнопка **«Завершить день»**; после отметки завершённого ритуала за календарный день — **«Отчёт за сегодня»** (то же окно сводки). Модалка **DR-002** (`EndOfDayModal`): круговая диаграмма доли закрытых задач по плану на день; **`eodCompletedLocalDates`**, **`eodPreferences`**, глобальный вкл/выкл в настройках, чекбокс задачи «участвует в ритуале». Списки «сделано / не закрыто» — только по **плану на календарный день** (`tasksPlannedForLocalDay` в `@motivator/core`); при «не закрыто (0)» блок подсвечивается зелёным; бэклог — отдельный мягкий блок. Продуктовые «открытые вопросы» — в модалке **«Краткая сводка»**. **DR-004:** второе подтверждение «сделано» на мини-карточке и в модалках создания/редактирования; истечение окна ожидания на клиенте (`VaultProvider`); стили **`animate-dc-pending`**, **`animate-task-done-success`**, **`animate-task-soft-miss`** в `index.css`. Задел под роли **`user` / `beta_tester` / `admin`**: `motivatorRole.ts`, **`app_metadata.motivator_role`** (ограничения UI пока не активны). |
| **Фильтры вида** | Кнопка раскрывает панель (**группа**, **приоритеты**, **цвет**, **повторы**). Опции подбираются из задач **текущей вкладки** (день: план выбранного дня + бэклог; неделя: попадание в любой день недели; месяц: в любой день месяца), с перекрёстным учётом остальных фильтров (например список цветов — только те, что остаются среди задач после группы/приоритета/повторов). **Информер** активных условий — когда панель **свернута**. Логика области видимости — `web/src/lib/plannerFilterScope.ts`. |
| **Пересечения слотов** | При изменении даты плана, оценки или времени в модалке задачи — если интервал пересекается с другой незавершённой задачей на тот же день, показывается подтверждение (`@motivator/core` / timeblocking, ключ `app.overlapConfirm`). |
| **Задача** | Заголовок, выполнено, группа, **цвет** в одной раскрывающейся секции: кружки (недавние + палитра), **HEX** и **нативная пипетка** синхронизированы; **приоритет 1–5**, чек-лист («план работы»): новые пункты попадают **в конец** списка (сверху вниз — порядок добавления), отметки на **мини-карточке** дня/бэклога; дата плана **или** только бэклог; оценка времени в форме **часы + минуты (0–59)** (в vault **`estimatedMinutes`**); для задачи **с датой в плане** оценка обязательна. **Время начала XOR время завершения** (или без времени) — блок «Время» в раскрывашке: два селекта **часы** и **минуты** в стиле интерфейса (без нативного `<input type="time">`). На **мини-карточке** и в **недельной сетке** левая полоска цвета задаётся через опорный **HEX** палитры (`TASK_COLOR_HEX`), чтобы цвет не терялся из‑за JIT Tailwind; текст времени суток — в формате «часы и минуты» (`app.miniCardTimeHm`). Редактирование (`TaskEditModal`) использует те же правила расписания и якоря повтора, что и создание задачи (`taskScheduleValidation`). Родитель нельзя закрыть, пока открыты пункты чек-листа. Удаление с непустым чек-листом — подтверждение (`app.confirmDeleteTaskWithChecklist`). |
| **Создание задачи** | В модалке показывается подсказка **обязательных для «Сохранить»** полей (название; при задаче с датой в плане — ещё оценка), оформленная в **янтарном** стиле блока черновиков на `/app`. Снимок формы инициализируется **только при открытии** модалки (пока окно открыто, смена выбранного дня или фильтра группы **не** сбрасывает форму и выбранный цвет). Закрытие без успешного «Сохранить» (крестик, «Отмена», клик по затемнению, **Escape**) открывает **собственный** диалог подтверждения в стиле приложения (не `window.confirm`). **Черновик** в vault попадает только если форма **реально менялась** относительно начального состояния (каноническое сравнение полей в `CreateTaskModal`, без ложной «грязи» от HEX/пробелов); при закрытии без изменений задача и черновик **не создаются**. Успешное сохранение создаёт задачу и при необходимости удаляет черновик по `removeDraftId`. На **`/app`** черновики из vault: при **одном** черновике — компактная секция со списком; при **нескольких** — кнопка **«Черновики»** с числом и **модальное окно** со списком (продолжить / удалить). Санитизация ввода полей — модуль `web/src/lib/fieldSanitize.ts` (оценка, «каждые N дней», заголовок, HEX палитры и т.д.). Проверки даты/времени/оценки/якоря повтора для сохранения задачи — **`web/src/lib/taskScheduleValidation.ts`** (черновик без этих проверок). |
| **Настройки** | Пять **названий уровней приоритета** (хранятся в vault), **смена пароля входа** (Supabase `signInWithPassword` + `updateUser`), группы как раньше. Кнопка **«Краткая сводка»** открывает модалку для **всех** залогиненных пользователей (ограничения по ролям — в плане, не включены в UI). Задел под роли: **`web/src/lib/motivatorRole.ts`**, **`app_metadata.motivator_role`**. Вверху модалки — **кольцевая диаграмма**: доля закрытых фаз дорожной карты до **MVP 1.0.0** (фазы **0–6** из **0–11**) и подпись с **процентом**. **Пять** раскрывающихся блоков по умолчанию **свёрнуты** при открытии: **уже реализовано**, **план до 1.0.0**, **релиз-ноты** (список изменений по датам), **идеи после релиза**, **открытые вопросы** (строки в локалях). Дорожная карта и **changelog** — **`web/src/data/productRoadmap.ts`** (`IMPLEMENTED_MVP_PHASES`, `MVP_PHASES_PLANNED`, `RELEASE_NOTES_BLOCKS`, `IDEAS_LATER_ENTRIES`). |
| **Группы** | Список групп в vault, фильтр по группе на `/app`, CRUD групп в настройках (группу по умолчанию нельзя удалить). |

Сборка: в **`vite.config.ts`** задан `build.chunkSizeWarningLimit` (ослабление предупреждения Vite о размере одного чанка).

### Перед коммитом (документация и «Краткая сводка»)

Имеет смысл выполнять **перед каждым коммитом**, который меняет продукт или публичный контент:

1. **`web/README.md`** — таблица «Возможности», версия из [`package.json`](./package.json), при необходимости разделы ниже (локально, Vercel, vault).
2. **`web/src/data/productRoadmap.ts`** — модалка **«Краткая сводка»** в настройках:
   - **`RELEASE_NOTES_BLOCKS`** — при пользовательски заметных изменениях добавить блок с **новой датой сверху** (формат как у существующих записей).
   - **`IMPLEMENTED_MVP_PHASES`** / **`MVP_PHASES_PLANNED`** — если изменился реализованный или планируемый scope.
   - **`IDEAS_LATER_ENTRIES`** — при добавлении идей «после релиза»; держать в согласовании с **`obsidian-motivator/15-Идеи-для-развития.md`** (в модалке пользователь видит не все заголовки Obsidian одновременно — только то, что перенесено в массив).
   - Проверить в локалях строки секции **«Открытые вопросы»** в модалке (`settings.roadmapOpenQuestion*` в `src/i18n/locales/*.json`), если трогали продуктовые формулировки.

Контент дорожной карты и релиз-нотов не обязан совпадать с каждым микрокоммитом, но **не должен противоречить** текущей сборке.

## Локально

Монорепозиторий: зависимости и линк `@motivator/core` ставятся из **корня** репозитория (`planner/`), затем запуск фронта из `web/`.

```bash
cd ..   # из web/: перейти в корень planner
npm install
cd web
cp .env.example .env.local
# заполните VITE_SUPABASE_URL и VITE_SUPABASE_ANON_KEY
npm run dev
```

## Supabase

1. Создайте проект в [Supabase](https://supabase.com/).
2. В SQL Editor выполните миграцию [`supabase/migrations/001_user_vault.sql`](./supabase/migrations/001_user_vault.sql).
3. Authentication → включите Email / пароль (если ещё не включено для вашего проекта).
4. Для разработки без писем: отключите обязательное подтверждение email (или подключите SMTP). Встроенная почта Supabase имеет rate limit.

## Seed и повторный вход

- После **выхода** из аккаунта seed удаляется из браузера. Чтобы открыть те же задачи, на экране **«Ключ шифрования»** выберите **«У меня уже есть seed»** и вставьте сохранённый base64 + тот же пароль деривации (если использовали).
- Новый сгенерированный seed после выхода **не** расшифрует старый vault на сервере.

## Vercel

Монорепозиторий: задайте **корень репозитория** как корень проекта в Vercel (не только `web`), иначе workspace-пакет `@motivator/core` не попадёт в сборку.

1. **Root Directory** → корень репозитория (`planner/`).
2. **Install Command**: `npm install`
3. **Build Command**: `npm run build -w web`
4. **Output Directory**: `web/dist`
5. Environment Variables: те же `VITE_SUPABASE_*`, что в `.env.local`.

Если по историческим причинам корень проекта в Vercel остаётся только `web`, перед сборкой нужно вручную обеспечить доступ к `packages/motivator-core` (например, скопировать структуру монорепозитория или собирать из полного clone с установкой из родительской папки).

Сборка рассчитана на **Vite 7** (Rollup): в корне репозитория в `package.json` задан **`overrides`** на `vite@7.3.3`, а `@vitejs/plugin-react` — ветка **5.x**, совместимая с Vite 7. Так на Vercel/Linux не подтягивается Vite 8 с нативным **Rolldown**, из‑за которого часто возникает `MODULE_NOT_FOUND` при загрузке нативного binding Rolldown.

Файл [`vercel.json`](./vercel.json) отправляет все пути на `index.html` для SPA.

## PWA (установка на домашний экран)

В production-сборке подключены **manifest** и **service worker** (`vite-plugin-pwa`): кэшируются статические ассеты приложения, для SPA настроен **navigate fallback** на `/index.html`.

Иконки `public/pwa-192.png` и `public/pwa-512.png` можно перегенерировать скриптом:

`powershell -ExecutionPolicy Bypass -File scripts/generate-pwa-icons.ps1`

В режиме `npm run dev` SW **выключен** (`devOptions.enabled: false`), чтобы не мешать разработке.

## Синхронизация vault

До завершения первой загрузки с Supabase поля задач **неактивны** (нет гонки с созданием пустого vault). Строка статуса под заголовком: загрузка / сохранение / время последнего успешного сохранения. При ошибке расшифровки показывается сообщение, редактирование блокируется (флаг **`decryptFailed`**).

Запись ciphertext на сервер **отложена** (debounce, см. `VAULT_REMOTE_SAVE_DEBOUNCE_MS` в пакете `@motivator/core`): локальное состояние обновляется сразу, сеть — после паузы; перед выходом из аккаунта выполняется принудительный flush.

Спецификация формата ciphertext, PBKDF2/AES-GCM и золотой вектор KDF для паритета с нативными клиентами: [`packages/motivator-core/docs/VAULT_CRYPTO_CONTRACT.md`](../packages/motivator-core/docs/VAULT_CRYPTO_CONTRACT.md).

## Контракт vault (актуальная схема)

Хранится как один зашифрованный JSON. Актуальная версия — **7** (типы в `@motivator/core`, модуль `vault/types`). Кратко:

- **`priorityLabels`**: пользовательские строки для уровней **1–5**.
- **`groups`**: проекты/группы.
- **`tasks`**: для каждой задачи — **`priorityRank`** (1–5), **`scheduledLocalDate`** (`YYYY-MM-DD` или `null`), **`checklist`** (массив в порядке отображения сверху вниз; новые пункты добавляются в конец), **`estimatedMinutes`**, **`timeMode`** (`none` | `start` | `end`), **`timeMinutesFromMidnight`**, **`recurrence`** / **`recurrenceAnchorLocalDate`** (правило повтора: каждый день / каждые N дней / дни недели, и якорь серии), **`completedOccurrenceLocalDates`** — для повторов список дней, на которые отмечено выполнение вхождения (глобальный **`done`** при повторе по-прежнему может архивировать серию целиком). Идентификатор серии для отчётов (**DR-008**) совпадает с **`id`** задачи. Плюс группа, цвет, заголовок, **`done`**, опционально **`includeInEodRitual`** (участие в ритуале EOD). **DR-004:** опционально **`doubleConfirmEnabled`**, **`doubleConfirmIntervalMinutes`**, **`doubleConfirmGraceMinutes`**, **`doubleConfirmPending`** (ожидание второго подтверждения за конкретный календарный день).
- **`drafts`**: черновики формы создания задачи; запись выполняется только при закрытии модалки без сохранения и при **фактических** изменениях формы относительно состояния при открытии (см. раздел «Создание задачи» в таблице возможностей).
- **`eodCompletedLocalDates`**, **`eodPreferences`**: ритуал End-of-Day (схема **v6+**).

Цепочка миграций: снимки **v1** … **v6** при открытии приводятся к **v7** через **`normalizeVault`** (например старые vault без повтора получают **`recurrence: null`**, **`drafts: []`**; до **v6** добавляются поля EOD по умолчанию; **v7** — поля двойного подтверждения **DR-004**).

### Миграция v3 → v4 (поля)

| Было (v3) | Стало (v4) |
|-----------|------------|
| `prioritySystem` + `priorityLevel` / `eisenhowerQuadrant` | одно поле **`priorityRank`** (эвристика: уровни 1–3 → ранги 1–3; квадранты q1…q4 → ранги 1,2,4,5; Inbox → 3) |
| `subtasks` | **`checklist`** (тот же массив объектов) |
| — | **`scheduledLocalDate`**: для всех мигрированных задач **`null`** (бэклог); пользователь заново разносит по дням |
| — | **`estimatedMinutes`**, **`timeMode`**, **`timeMinutesFromMidnight`**: по умолчанию пусто / `none` / `null` |
