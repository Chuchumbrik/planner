# Мотиватор — веб-клиент

Клиент для деплоя на **Vercel**: React (Vite), Tailwind, Supabase Auth, клиентское шифрование vault (AES-GCM + PBKDF2 по seed). Номер версии **`0.*.*`** не означает «только черновик» — см. [версионирование](#версионирование).

**Целевой объём MVP для разработки** (релиз продукта **v1.0.0**) зафиксирован в [`obsidian-motivator/16-TZ-MVP-v1.0.md`](../obsidian-motivator/16-TZ-MVP-v1.0.md); поэтапный план внедрения — [`obsidian-motivator/17-План-реализации-MVP.md`](../obsidian-motivator/17-План-реализации-MVP.md).

**Текущая версия веб-клиента** в репозитории: **`0.6.39`** ([`package.json`](./package.json)) — см. правила ниже. Краткий обзор модалки **«Краткая сводка»** — в [отдельном разделе](#краткая-сводка); источник данных — [`web/src/data/productRoadmap.ts`](./src/data/productRoadmap.ts).

### Версионирование

**После `1.0.0`** ориентируемся на классический **SemVer** `MAJOR.MINOR.PATCH`:

| Часть | Когда повышать | Пример |
|--------|----------------|--------|
| **MAJOR** | Обратно несовместимые изменения: ломающая миграция данных/контракта, новая архитектура, удаление опубликованных возможностей | `1.0.0` → `2.0.0` |
| **MINOR** | Новая функциональность, совместимая с предыдущей версией | `1.3.0` → `1.4.0` |
| **PATCH** | Исправления ошибок и мелкие улучшения без новых фич | `1.4.5` → `1.4.6` |

**До `1.0.0`:** в [`package.json`](./package.json) хранится строка **`0.x.y`** — три числа, как требует **SemVer 2.0** и npm (четвёртое число в одном поле **нельзя** записать без выхода из спецификации). Смысл для команды — **«ноль и три уровня»** **`0 · x · y`**, плюс **четвёртый идентификатор** в виде **метаданных сборки**:

| Уровень | В `package.json` | Смысл |
|---------|------------------|--------|
| **0** | первая цифра всегда **0** до релиза **`1.0.0`** | Это **эпоха нумерации**, а **не** оценка качества. Приложение на **`0.10.x`** может быть уже **полнофункциональным, стабильным и «продуктовым»** — ограничивает только принятое правило версий, а не зрелость UX или объём фич. **Не путать** с ярлыком «только MVP / прототип». |
| **x** | **MINOR** (у нас **`N`**) | Номер **последней закрытой** фазы из [`obsidian-motivator/17-План-реализации-MVP.md`](../obsidian-motivator/17-План-реализации-MVP.md). В дорожной карте приложения — фазы **0…13** до **`1.0.0`**; после закрытия фазы поднимают **`x`** (например после фазы 7 → **`0.7.0`**). |
| **y** | **PATCH** | Правки **внутри** текущего **`x`**: багфиксы, полировка, документация без смены фазы. |
| **z** | не отдельное число в JSON | В production-сборке к semver добавляется **`+<git short SHA>`** (см. `vite.config.ts`, `APP_VERSION`). Полная метка билда: **`0.x.y+abc1234`** — это и есть **четвёртый слой** для различения коммитов без bump **`y`** на каждый push. |

Переход на **`1.0.0`** — когда закрыта **фаза 13** (последняя в дорожной карте до публичного контракта: после релизного чеклиста фазы **11** — домен и процесс веток **12–13**) по ТЗ/Scope и пройдена регрессия: формально «первая стабильная линия» и публичный контракт версий, **не** обязательно первый день, когда приложение стало хорошим (оно могло быть таким ещё на **`0.x.y`**).

Ориентировочная шкала (не заменяет правил выше):

| Тип | Версия | Когда | Пример |
|-----|--------|--------|--------|
| Начало разработки | `0.1.0` | Первый осмысленный билд | Старт MVP |
| Фича / закрытая фаза | `0.2.0`, … | Milestone по фазам плана | Закрыли фазу 2 |
| Баги без новой фазы | `0.2.1`, … | После тестирования внутри той же фазы | Фиксы таймблокинга |
| Pre-release | `0.3.0-beta.1` | По необходимости до 1.0.0 | Опционально |
| Первый публичный | **`1.0.0`** | Готовность по Scope/TЗ | Официальный запуск |

Пакет **`@motivator/core`** имеет **собственный** `version` в `packages/motivator-core/package.json`: там MINOR/PATCH завязаны на **совместимость API и схемы vault**, а не обязаны совпадать с версией веб-клиента.

**Сборка:** при **`vite build`** к semver добавляется **`+<git short SHA>`** (из корня репозитория). Строка в интерфейсе (`APP_VERSION` в `src/version.ts`) различает сборки без обязательного bump **`PATCH`** на каждый коммит.

**Текущее значение** в репозитории берётся из `package.json`; его поддерживают вручную при закрытии фазы или при серии патчей.

Подробная спецификация раннего запуска и деплоя: [`obsidian-motivator/14-V1-минимальный-запуск-Vercel.md`](../obsidian-motivator/14-V1-минимальный-запуск-Vercel.md).

В центре кольца (**`PlanProgressRing`**) — **процент** выполнения (расчёт долей без изменений); **`DayPlanDonut`** / **`PeriodPlanDonut`**. На узком экране кольцо **выше** списка (`flex-col-reverse`). Выполненные остаются в тех же списках **ниже** невыполненных (приоритет и название внутри группы). **Неделя** — таймблокинг (понедельник–воскресенье, сетка часов, слоты по времени начала/конца и оценке); справа от сетки (на широком экране) — **кольцо «план недели»**: **`plannedPeriodProgress`** суммирует по дням недели ту же метрику; учитываются только дни с **датой ≤ сегодня**; действуют **те же фильтры**, что и у задач на вкладке; если в охвате **нет** ни одной задачи в плане — блок кольца **скрыт**. На узком экране кольцо **выше** сетки (`flex-col-reverse`). Таблица недели **без горизонтального скролла**: семь дней делят ширину контейнера (`minmax(0,1fr)`); заголовки дней и сетка слотов на **одной CSS Grid** с общими колонками (`subgrid` для строки времени), чтобы вертикальные линии совпадали; вертикальная прокрутка — тёмная полоса (`week-grid-v-scroll`); высота часа в сетке задаётся **`HOUR_HEIGHT_PX`** (комфортнее тачу). Текст-пояснение про слоты закреплён **под** сеткой (не внутри блока прокрутки по времени). **Месяц** — календарная сетка и счётчик задач по дням (клик по дню открывает вкладку «День»); **кольцо «план месяца»** — те же правила по дням текущего месяца до сегодня, фильтры, скрытие при нуле охвата; контейнер **`max-w-5xl`** по центру. **Период «Неделя»/«Месяц»:** рядом с **`PeriodPlanDonut`** — столбчатый разбивочный график (**Recharts**, **`PeriodPlanBreakdownChart`**) по **группам** или **цветам** слотов (`plannedPeriodSlotsByGroupId` / `plannedPeriodSlotsByColorKey` в **`@motivator/core`**); навигация недели — компактные **иконки** и короткий текст диапазона дат. **Каркас `/app`:** **`min-h-dvh`**, центральная колонка **`flex-1 min-h-0`** — чтобы сетка недели и нижняя панель **не обрезались** на коротком экране. В шапке — **меню аккаунта** (иконка пользователя): **«Отчёты»** (`/app/reports`); рядом с **«Фильтры»** на вкладке **«День»** при включённом EOD — **«Завершить день»** / **«Отчёт за сегодня»** (если в календаре выбран **сегодня**) или **«Отчёт за день»** (если выбрана **прошлая** дата); в меню аккаунта — ритуал за **сегодня** (**«Завершить день»** / **«Отчёт за сегодня»**); **«Краткая сводка»** (модалка дорожной карты и релиз-нот); переход в **`/settings`**; **выход**. | **Задача** | Заголовок, выполнено; на мини-карточке — полоска **цвета** по палитре; в модалках создания/редактирования выбор **группы** и **цвета** (**кружки**, **HEX**, пипетка) — в раскрывающемся блоке **«Дополнительные настройки»**; **приоритет 1–5**, чек-лист («план работы»): новые пункты попадают **в конец** списка (сверху вниз — порядок добавления); **галочки «выполнено» по пунктам** на мини-карточке дня/бэклога и в **`TaskEditModal`** — только в **календарный сегодня**; у строки чек-листа на карточке — **`<label>`** на всю строку, **`items-center`**, на мобилке **`min-h-11`** (~44px) для зоны тапа, на **`md+`** компактнее (`TaskMiniCard`; тот же контекст «сегодня», что главная галочка); **добавление и удаление** строк чек-листа в модалке — при праве редактирования и на других днях; дата плана **или** только бэклог; оценка времени в форме **часы + минуты (0–59)** (в vault **`estimatedMinutes`**); для задачи **с датой в плане** оценка обязательна. **Плановая дата** и **якорь повтора** в календаре модалок — **не раньше локального сегодня** (`LocalDatePickerField`, **`minLocalDateKey`**); кнопка **«Запланировать на выбранный день»** при выбранном в планировщике **прошлом** дне — **неактивна**; при открытии формы создания и после снятия бэклога подставляется дата **не раньше сегодня**. **Время начала XOR время завершения** (или без времени) — блок «Время» в раскрывашке: два селекта **часы** и **минуты** в стиле интерфейса (без нативного `<input type="time">`); если план на **сегодня**, варианты времени **не раньше текущего** локального момента (**`TaskTimeAccordion`**, **`earliestClockMinutesFromMidnight`**; нижняя граница обновляется **раз в минуту**, пока модалка открыта). На **мини-карточке** и в **недельной сетке** левая полоска цвета задаётся через опорный **HEX** палитры (`TASK_COLOR_HEX`), чтобы цвет не терялся из‑за JIT Tailwind; текст времени суток — в формате «часы и минуты» (`app.miniCardTimeHm`). Редактирование (`TaskEditModal`) использует те же правила расписания и якоря повтора, что и создание задачи (`taskScheduleValidation`). На телефонах (WebKit) модалка монтируется в **`document.body`** (`createPortal`), на время открытия у **`body`** отключается прокрутка — касания не уходят на список задач под окном. Родитель нельзя закрыть, пока открыты пункты чек-листа. Удаление с непустым чек-листом — подтверждение (`app.confirmDeleteTaskWithChecklist`). |
| **Создание задачи** | Подвал модалки **липкий**: после **первой** попытки «Сохранить» показывается янтарный список недостающего (до этого не отображается) — **единственный** блок с текстами ошибок для этого шага, **без** дублирующей красной строки под ним; для плана на день без оценки в списке используется **`estimateRequiredWhenPlanned`**. **«Сохранить»** активен при праве редактирования, при ошибках — прокрутка к полю. Правила оценки: для задачи **с датой в плане на день** нужна **непустая корректная** оценка; для **бэклога** (без даты в плане) оценка **не обязательна**, но **некорректный ввод** в полях часы/минуты по-прежнему блокирует сохранение. Средняя часть модалки при длинной форме **прокручивается** (`overflow-y-auto` / **`min-h-0`** в flex-цепочке); в блоке **«Дополнительные настройки»** выбор **цвета** — единая типографика подписей (**`TaskColorAccordion`**). Снимок формы инициализируется **только при открытии** модалки (пока окно открыто, смена выбранного дня или фильтра группы **не** сбрасывает форму и выбранный цвет). Закрытие без успешного «Сохранить» — диалог подтверждения (не `window.confirm`). **Черновик** в vault только если форма менялась; успешное сохранение удаляет черновик по `removeDraftId`. На **`/app`** доступ к черновикам — бейдж на **«Фильтры»**, только пока есть хотя бы один черновик (см. строку «Фильтры вида»). Санитизация — `web/src/lib/fieldSanitize.ts`; проверки расписания — **`web/src/lib/taskScheduleValidation.ts`** (черновик без этих проверок). |
| **Настройки** | Страница **`/settings`** (переход из **меню аккаунта** на **`/app`**): пять **названий уровней приоритета** (хранятся в vault), **смена пароля входа** (Supabase `signInWithPassword` + `updateUser`), CRUD групп, язык интерфейса. Блок **уведомлений (Web Push):** режим **`off` / `hybrid` / `full`** в vault, запрос разрешения ОС, синхронизация расписания в Supabase, кнопка теста — см. [«Уведомления (Web Push)»](#уведомления-web-push). Блок **EOD:** глобальный вкл/выкл ритуала и **`autoCloseAtDayEnd`** (автоматическое добавление прошлых дней с планом в **`eodCompletedLocalDates`**). Модалка **«Краткая сводка»** открывается **с планировщика** (`/app`), не с экрана настроек — см. [раздел «Краткая сводка»](#краткая-сводка). Задел под роли: **`web/src/lib/motivatorRole.ts`**, **`app_metadata.motivator_role`** (ограничения UI в плане, фаза 8 дорожной карты). |
| **Группы** | Список групп в vault, фильтр по группе на `/app`, CRUD групп в настройках (группу по умолчанию нельзя удалить). |

## Краткая сводка

Модалка **«Краткая сводка»** открывается из **меню аккаунта** на странице планировщика **`/app`**: иконка пользователя в шапке → выпадающий список → пункт с текстом **`settings.roadmapTempButton`** в **`web/src/i18n/locales/`** (в коде **`AppPage.tsx`**, рядом с ссылкой на **`/settings`** и выходом). На экране **`/settings`** отдельной кнопки для этой модалки **нет** — при необходимости вернитесь на **`/app`** и откройте меню. Доступна **всем залогиненным** пользователям; скрытие дорожной карты по ролям — в плане (см. фазу **8** в **`MVP_PHASES_PLANNED`**).

Разметка и логика модалки — **[`web/src/components/ProductRoadmapModal.tsx`](./src/components/ProductRoadmapModal.tsx)**. Данные (дорожная карта, релиз-ноты, идеи) — **[`web/src/data/productRoadmap.ts`](./src/data/productRoadmap.ts)**. Текст «Открытых вопросов» в модалке дублируется из **i18n** (`settings.roadmapOpenQuestion*` в **`ru.json`** / **`en.json`**), чтобы переводы жили рядом с остальными строками приложения.

### Содержимое модалки (шесть блоков сверху вниз)

1. **Верх:** кольцевая диаграмма прогресса до **MVP 1.0.0** — доля закрытых фаз **0–6** из **0–13** и подпись с процентом. Источник чисел — те же константы, что у блоков **«Уже реализовано»** и **«План до 1.0.0»** (без дублирования «магических» процентов в разметке).
2. **Уже реализовано** — массив **`IMPLEMENTED_MVP_PHASES`**: фазы **0–6**, у каждой краткое описание и раскрывающийся список **«Подробнее»** (`detailBullets`). Назначение: зафиксировать **фактическое** поведение текущей сборки; по возможности согласовать формулировки с пунктами **`RELEASE_NOTES_BLOCKS`** за соответствующие периоды.
3. **План до 1.0.0** — **`MVP_PHASES_PLANNED`**: фазы **7–13**; смысл фаз — [`obsidian-motivator/17-План-реализации-MVP.md`](../obsidian-motivator/17-План-реализации-MVP.md). У фазы может быть **`plain`** («простыми словами») и **`detailBullets`** («Подробнее»).
4. **Релиз-ноты** — **`RELEASE_NOTES_BLOCKS`**: хронология по **календарным датам** внесения изменений в продукт (`dateLabel`, **`YYYY-MM-DD`**). В модалке блоки дат идут **по убыванию** (самый свежий календарный день выше). Внутри одной даты — один или несколько выпусков **`items`**; у выпуска обязательно **`releasedInVersion`** (semver из [`web/package.json`](./package.json) на момент выпуска, **без** суффикса **`+git`**), массив **`changes`**, опционально **`plainBullets`** под раскрывашкой **«Подробности простым языком»**. Сюда же попадают **заметные для пользователя** ограничения планировщика (например невозможность выбрать прошлую **плановую дату** или время на **сегодня** раньше текущего момента) — дублирующая сводка по поведению есть в таблице **«Возможности (сводка)»** выше. Правила ведения массива — в **комментарии над `RELEASE_NOTES_BLOCKS`** в `productRoadmap.ts`, в [правиле репозитория](../.cursor/rules/pre-commit-docs-roadmap.mdc) и в скилле **[`../.cursor/skills/pre-commit-docs-roadmap/SKILL.md`](../.cursor/skills/pre-commit-docs-roadmap/SKILL.md)** (перед коммитом агент проходит скилл целиком, краткий `.mdc` его **не заменяет**).
5. **Идеи на потом** — **`IDEAS_LATER_ENTRIES`**: черновой backlog после **1.0.0**; полный список идей вне модалки — [`obsidian-motivator/15-Идеи-для-развития.md`](../obsidian-motivator/15-Идеи-для-развития.md); в модалке только перенесённые карточки.
6. **Открытые вопросы** — продуктовые заметки без срока: строки **`settings.roadmapOpenQuestion*`** в **`web/src/i18n/locales/ru.json`** и **`en.json`** (заголовок секции — **`settings.roadmapOpenQuestions`**, подсказка — **`settings.roadmapOpenQuestionsHint`**); разметка списка в **`ProductRoadmapModal.tsx`**.

Контент **`productRoadmap.ts`** не должен противоречить текущей сборке и **`README`**; при правке «Открытых вопросов» обновляют **обе** локали.

### Перед коммитом (сводка и README)

Полный чеклист — **следующий подраздел** («Перед коммитом (документация и «Краткая сводка») — **обязательно**») и скилл **[`../.cursor/skills/pre-commit-docs-roadmap/SKILL.md`](../.cursor/skills/pre-commit-docs-roadmap/SKILL.md)**. В коммите с изменениями кода или UX почти всегда нужны правки **`web/README.md`** (таблица **«Возможности»**, строка **версии** у шапки README, при необходимости этот раздел **«Краткая сводка»**) и **`web/src/data/productRoadmap.ts`** (как минимум **`RELEASE_NOTES_BLOCKS`** при пользовательски заметном эффекте).

### Перед коммитом (документация и «Краткая сводка») — **обязательно**

**Жёсткое правило репозитория:** перед **любым** `git commit` (включая мелкие и «технические») нужно пройти чеклист и **включить результат в тот же коммит**. Иначе документация и модалка **«Краткая сводка»** отстают от кода. Для агентов Cursor: правило **`.cursor/rules/pre-commit-docs-roadmap.mdc`** (`alwaysApply`) **плюс** обязательный скилл **`.cursor/skills/pre-commit-docs-roadmap/SKILL.md`** — пошаговая процедура; краткий `.mdc` **не заменяет** чтение скилла перед коммитом.

1. **`web/README.md`** — таблица «Возможности», **версия =** [`package.json`](./package.json), при необходимости разделы ниже (локально, Vercel, vault).
2. **`web/src/data/productRoadmap.ts`** — данные модалки **«Краткая сводка»** (открытие с **`/app`**, см. выше):
   - **`RELEASE_NOTES_BLOCKS`** — см. `.cursor/rules/pre-commit-docs-roadmap.mdc`: дата = день работ; новый день — новый `dateLabel`; несколько `items` за день; у каждого элемента **`releasedInVersion`** (semver без `+git`); `plainBullets` для сути без кода; при внутренних правках без UI — краткий пункт или пометка «техн. / внутреннее».
   - **`IMPLEMENTED_MVP_PHASES`** / **`MVP_PHASES_PLANNED`** — если изменился реализованный или планируемый scope.
   - **`IDEAS_LATER_ENTRIES`** — при добавлении идей «после релиза»; держать в согласовании с **`obsidian-motivator/15-Идеи-для-развития.md`** (в модалке пользователь видит не все заголовки Obsidian одновременно — только то, что перенесено в массив).
   - Проверить в локалях строки секции **«Открытые вопросы»** в модалке (`settings.roadmapOpenQuestion*` в `src/i18n/locales/*.json`), если трогали продуктовые формулировки.

Контент дорожной карты **не должен противоречить** текущей сборке.

Так на Vercel/Linux не подтягивается Vite 8 с нативным **Rolldown**, из‑за которого часто возникает `MODULE_NOT_FOUND` при загрузке нативного binding Rolldown.

## PWA (установка на домашний экран)

В production-сборке подключены **manifest** и **service worker** (`vite-plugin-pwa`): кэшируются статические ассеты приложения, для SPA настроен **navigate fallback** на `/index.html`.

Регистрация SW выполняется из кода — **`initPwaServiceWorker()`** в [`src/lib/pwaServiceWorker.ts`](./src/lib/pwaServiceWorker.ts) (импорт в [`src/main.tsx`](./src/main.tsx)); в **`vite.config.ts`** у плагина **`injectRegister: null`**, чтобы не было **двойной** регистрации (раньше в `index.html` дополнительно подключался `registerSW.js`). Режим **`registerType: 'autoUpdate'`** по-прежнему активирует новый worker и перезагружает страницу, когда на сервере появилась новая сборка. Плюс явно вызывается **`registration.update()`** при возврате на вкладку / в окно и **раз в час**, чтобы мобильный PWA не ждал только внутреннего расписания браузера.

Иконки `public/pwa-192.png` и `public/pwa-512.png` можно перегенерировать скриптом:

`powershell -ExecutionPolicy Bypass -File scripts/generate-pwa-icons.ps1`

В режиме `npm run dev` SW **выключен** (`devOptions.enabled: false`), чтобы не мешать разработке.

## Уведомления (Web Push)

- **Режим** хранится в vault: **`notificationPreferences.deliveryMode`** — **`off`** (по умолчанию), **`hybrid`** (на Supabase синхронизируются только время и тип события; текст уведомления обезличенный), **`full`** (на сервер могут попадать **названия задач** в открытом виде). Настройки и кнопки — экран **`/settings`**.
- **Таблицы Supabase** (миграция [`web/supabase/migrations/002_notifications.sql`](./supabase/migrations/002_notifications.sql)): **`push_subscriptions`**, **`notification_fire_requests`** (RLS: только свой `user_id`). После правок vault клиент с **debounce** пересчитывает окна срабатываний по задачам с **`timeMode`** `start` | `end` и датой в плане в горизонте 14 дней и заменяет будущие строки со статусом **`scheduled`**.
- **Сборка фронта:** в `.env.local` / Vercel задайте **`VITE_VAPID_PUBLIC_KEY`** (пара ключа VAPID; приватный ключ хранится только в секретах Edge / сервера, **не** в `VITE_*`). Пример: [`.env.example`](./.env.example).
- **Edge Functions** (исходники в [`web/supabase/functions/`](./supabase/functions/)): **`send-due`** — POST, выбор просроченных `notification_fire_requests` и отправка через **web-push** (секреты `VAPID_*`, `SUPABASE_SERVICE_ROLE_KEY`; опционально **`CRON_SECRET`** в заголовке `Authorization: Bearer …`); **`notifications-test`** — POST с JWT пользователя, тестовый push на все сохранённые подписки устройства.
- **Cron:** настройте вызов **`send-due`** раз в минуту (например **Supabase pg_cron** + `net.http_post`, либо внешний планировщик). Без cron строки в БД накапливаются, но push не уходит.
- **iOS:** Web Push для веб-приложения с экрана «Домой» — с версий, поддерживаемых Apple; ограничения см. документацию Apple.

### Ручная настройка Supabase и Vercel (для Web Push)

**Supabase**

1. **SQL:** в SQL Editor по порядку — **`001_user_vault.sql`**, затем **`002_notifications.sql`**. В Table Editor должны появиться **`push_subscriptions`** и **`notification_fire_requests`**.
2. **Пара VAPID** (публичный и приватный ключ, формат **Base64 URL**, например `npx web-push generate-vapid-keys`). Публичный дублируется на фронте как **`VITE_VAPID_PUBLIC_KEY`**; приватный **только** на сервере.
3. **Секреты Edge Functions** (Dashboard → Edge Functions → Secrets): **`SUPABASE_URL`**, **`SUPABASE_ANON_KEY`**, **`SUPABASE_SERVICE_ROLE_KEY`**, **`VAPID_PUBLIC_KEY`** (тот же, что на Vercel), **`VAPID_PRIVATE_KEY`**, **`VAPID_SUBJECT`** (например `mailto:ваш@email`). Опционально **`CRON_SECRET`**: тогда каждый вызов **`send-due`** снаружи должен передавать заголовок **`Authorization: Bearer <CRON_SECRET>`** (без секрета функция принимает любой POST — не рекомендуется в проде).
4. **Деплой функций** из корня репозитория (с настроенным CLI и `--project-ref`):  
   `supabase functions deploy send-due --project-ref <REF>`  
   `supabase functions deploy notifications-test --project-ref <REF>`  
   Имена совпадают с вызовами **`supabase.functions.invoke`** во фронте.
5. **Cron:** раз в минуту **POST** на URL функции **`send-due`** (в Dashboard — Invoke URL; при **`CRON_SECRET`** — заголовок как в п.3). Варианты: **pg_cron** + **`net.http_post`**, GitHub Actions, другой внешний планировщик.

**Vercel**

1. Корень репозитория, **`npm run build -w web`**, **`web/dist`** — как в [разделе «Vercel»](#vercel) выше.
2. **Переменные:** **`VITE_SUPABASE_URL`**, **`VITE_SUPABASE_ANON_KEY`**, **`VITE_VAPID_PUBLIC_KEY`** (= публичный VAPID, совпадает с секретом **`VAPID_PUBLIC_KEY`** у Edge). Сервисный ключ и приватный VAPID **не** добавлять в Vercel.

## Синхронизация vault

До завершения первой загрузки с Supabase поля задач **неактивны** (нет гонки с созданием пустого vault). Строка статуса под заголовком: загрузка / сохранение / время последнего успешного сохранения. При ошибке расшифровки показывается сообщение, редактирование блокируется (флаг **`decryptFailed`**).

Запись ciphertext на сервер **отложена** (debounce, см. `VAULT_REMOTE_SAVE_DEBOUNCE_MS` в пакете `@motivator/core`): локальное состояние обновляется сразу, сеть — после паузы; перед выходом из аккаунта выполняется принудительный flush.

## Контракт vault и криптография

Спецификация формата ciphertext, PBKDF2/AES-GCM и золотой вектор KDF для паритета с нативными клиентами: [`packages/motivator-core/docs/VAULT_CRYPTO_CONTRACT.md`](../packages/motivator-core/docs/VAULT_CRYPTO_CONTRACT.md).

Общая доменная логика и крипто вынесены в workspace-пакет **`@motivator/core`** (`packages/motivator-core`). Установка зависимостей из **корня** монорепозитория: `npm install`.

## Контракт vault (актуальная схема)

Хранится как один зашифрованный JSON. Актуальная версия — **8** (типы в `@motivator/core`, модуль `vault/types`). Кратко:

- **`priorityLabels`**: пользовательские строки для уровней **1–5**.
- **`groups`**: проекты/группы.
- **`tasks`**: для каждой задачи — **`priorityRank`** (1–5), **`scheduledLocalDate`** (`YYYY-MM-DD` или `null`), **`checklist`** (массив в порядке отображения сверху вниз; новые пункты добавляются в конец), **`estimatedMinutes`**, **`timeMode`** (`none` | `start` | `end`), **`timeMinutesFromMidnight`**, **`recurrence`** / **`recurrenceAnchorLocalDate`** (правило повтора: каждый день / каждые N дней / дни недели, и якорь серии), **`completedOccurrenceLocalDates`** — для повторов список дней, на которые отмечено выполнение вхождения (глобальный **`done`** при повторе по-прежнему может архивировать серию целиком). Идентификатор серии для отчётов (**DR-008**) совпадает с **`id`** задачи. Плюс группа, цвет, заголовок, **`done`**, опционально **`includeInEodRitual`** (участие в ритуале EOD). **DR-004:** опционально **`doubleConfirmEnabled`**, **`doubleConfirmIntervalMinutes`**, **`doubleConfirmGraceMinutes`**, **`doubleConfirmPending`** (ожидание второго подтверждения за конкретный календарный день).
- **`drafts`**: черновики формы создания задачи; запись выполняется только при закрытии модалки без сохранения и при **фактических** изменениях формы относительно состояния при открытии (см. раздел «Создание задачи» в таблице возможностей).
- **`eodCompletedLocalDates`**, **`eodPreferences`**: ритуал End-of-Day (схема **v6+**).
- **`notificationPreferences`** (**v8+**): **`deliveryMode`** — **`off`** | **`hybrid`** | **`full`** (Web Push; см. раздел **«Уведомления (Web Push)»** в этом README).

Цепочка миграций: снимки **v1** … **v7** при открытии приводятся к **v8** через **`normalizeVault`** (например старые vault без повтора получают **`recurrence: null`**, **`drafts: []`**; до **v6** добавляются поля EOD по умолчанию; **v7** — поля двойного подтверждения **DR-004**; **v8** — **`notificationPreferences`** с режимом доставки push по умолчанию **`off`**).

### Миграция v3 → v4 (поля)

| Было (v3) | Стало (v4) |
|-----------|------------|
| `prioritySystem` + `priorityLevel` / `eisenhowerQuadrant` | одно поле **`priorityRank`** (эвристика: уровни 1–3 → ранги 1–3; квадранты q1…q4 → ранги 1,2,4,5; Inbox → 3) |
| `subtasks` | **`checklist`** (тот же массив объектов) |
| — | **`scheduledLocalDate`**: для всех мигрированных задач **`null`** (бэклог); пользователь заново разносит по дням |
| — | **`estimatedMinutes`**, **`timeMode`**, **`timeMinutesFromMidnight`**: по умолчанию пусто / `none` / `null` |
