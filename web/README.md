# Мотиватор — веб-клиент

Клиент для деплоя на **Vercel**: React (Vite), Tailwind, Supabase Auth, клиентское шифрование vault (AES-GCM + PBKDF2 по seed). Номер версии **`0.*.*`** не означает «только черновик» — см. [версионирование](#версионирование).

**Целевой объём MVP для разработки** (релиз продукта **v1.0.0**) зафиксирован в [`obsidian-motivator/16-TZ-MVP-v1.0.md`](../obsidian-motivator/16-TZ-MVP-v1.0.md); поэтапный план внедрения — [`obsidian-motivator/17-План-реализации-MVP.md`](../obsidian-motivator/17-План-реализации-MVP.md).

**Текущая версия веб-клиента** в репозитории: **`0.6.27`** ([`package.json`](./package.json)) — см. правила ниже. Краткий обзор модалки **«Краткая сводка»** в настройках — в [отдельном разделе](#краткая-сводка); источник данных — [`web/src/data/productRoadmap.ts`](./src/data/productRoadmap.ts).

### Версионирование

**После `1.0.0`** ориентируемся на классический **SemVer** `MAJOR.MINOR.PATCH`:

| Часть | Когда повышать | Пример |
|--------|----------------|--------|
| **MAJOR** | Обратно несовместимые изменения: ломающая миграция данных/контракта, новая архитектура, удаление опубликованных возможностей | `1.0.0` → `2.0.0` |
| **MINOR** | Новая функциональность, совместимая с предыдущей версией | `1.3.0` → `1.4.0` |
| **PATCH** | Исправления ошибок и мелкие улучшения без новых фич | `1.4.5` → `1.4.6` |

**До `1.0.0`:** в [`package.json`](./package.json) хранится строка **`0.x.y`** — три числа, как требует **SemVer 2.0** и npm (четвёртое число в одном поле **нельзя** записать без выхода из спецификации). Смысл для команды — **«ноль и три уровня»** **`0 · x · y`**, плюс **четвёртый идентификатор** в виде **метаданных сборки**:

| Уровень | В `package.json` | Смысл |
|---------|------------------|--------|
| **0** | первая цифра всегда **0** до релиза **`1.0.0`** | Это **эпоха нумерации**, а **не** оценка качества. Приложение на **`0.10.x`** может быть уже **полнофункциональным, стабильным и «продуктовым»** — ограничивает только принятое правило версий, а не зрелость UX или объём фич. **Не путать** с ярлыком «только MVP / прототип». |
| **x** | **MINOR** (у нас **`N`**) | Номер **последней закрытой** фазы из [`obsidian-motivator/17-План-реализации-MVP.md`](../obsidian-motivator/17-План-реализации-MVP.md). В дорожной карте приложения — фазы **0…13** до **`1.0.0`**; после закрытия фазы поднимают **`x`** (например после фазы 7 → **`0.7.0`**). |
| **y** | **PATCH** | Правки **внутри** текущего **`x`**: багфиксы, полировка, документация без смены фазы. |
| **z** | не отдельное число в JSON | В production-сборке к semver добавляется **`+<git short SHA>`** (см. `vite.config.ts`, `APP_VERSION`). Полная метка билда: **`0.x.y+abc1234`** — это и есть **четвёртый слой** для различения коммитов без bump **`y`** на каждый push. |

Переход на **`1.0.0`** — когда закрыта **фаза 13** (последняя в дорожной карте до публичного контракта: после релизного чеклиста фазы **11** — домен и процесс веток **12–13**) по ТЗ/Scope и пройдена регрессия: формально «первая стабильная линия» и публичный контракт версий, **не** обязательно первый день, когда приложение стало хорошим (оно могло быть таким ещё на **`0.x.y`**).

Ориентировочная шкала (не заменяет правил выше):

| Тип | Версия | Когда | Пример |
|-----|--------|--------|--------|
| Начало разработки | `0.1.0` | Первый осмысленный билд | Старт MVP |
| Фича / закрытая фаза | `0.2.0`, … | Milestone по фазам плана | Закрыли фазу 2 |
| Баги без новой фазы | `0.2.1`, … | После тестирования внутри той же фазы | Фиксы таймблокинга |
| Pre-release | `0.3.0-beta.1` | По необходимости до 1.0.0 | Опционально |
| Первый публичный | **`1.0.0`** | Готовность по Scope/TЗ | Официальный запуск |

Пакет **`@motivator/core`** имеет **собственный** `version` в `packages/motivator-core/package.json`: там MINOR/PATCH завязаны на **совместимость API и схемы vault**, а не обязаны совпадать с версией веб-клиента.

**Сборка:** при **`vite build`** к semver добавляется **`+<git short SHA>`** (из корня репозитория). Строка в интерфейсе (`APP_VERSION` в `src/version.ts`) различает сборки без обязательного bump **`PATCH`** на каждый коммит.

**Текущее значение** в репозитории берётся из `package.json`; его поддерживают вручную при закрытии фазы или при серии патчей.

Подробная спецификация раннего запуска и деплоя: [`obsidian-motivator/14-V1-минимальный-запуск-Vercel.md`](../obsidian-motivator/14-V1-минимальный-запуск-Vercel.md).

## Возможности (сводка)

| Область | Реализация |
|--------|-------------|
| **Локализация** | **i18next** + **react-i18next**, локали `src/i18n/locales/` (`ru`, `en`), язык по умолчанию `ru`, ключ в `localStorage`: **`motivator_lang`**. Переключатель на экране настроек. |
| **Vault** | Один JSON на пользователя, шифруется целиком; актуальная версия схемы **`schemaVersion: 7`** (**DR-004**: поля двойного подтверждения на задаче; ритуал EOD — поля с **v6**). Миграции v1→…→**v7** на клиенте: пакет **`@motivator/core`** (`normalizeVault`). Ошибка удалённой загрузки/гидрации — баннер с понятным текстом и действием **повторить** (`VaultProvider`). |
| **Вкладки `/app`** | **День** — план на дату + бэклог; справа от списка (на широком экране) — **кольцо прогресса**: **`plannedDayCompletionWeights`** — знаменатель = **число задач в плане**; каждая задача без чек-листа даёт 0 или 1 по главной галочке дня; задача с чек-листом даёт **долю выполненных пунктов** (четыре пункта → один выполненный = **0,25** этой задачи). Список плана и расчёт кольца берут один набор задач — **`tasksScheduledForPlannerDay`** в **`@motivator/core`** (то же, что фильтр «есть вхождение на этот календарный день», без чистого бэклога). В центре кольца (**`PlanProgressRing`**) — **процент** выполнения (расчёт долей без изменений); **`DayPlanDonut`** / **`PeriodPlanDonut`**. На узком экране кольцо **выше** списка (`flex-col-reverse`). Выполненные остаются в тех же списках **ниже** невыполненных (приоритет и название внутри группы). **Неделя** — таймблокинг (понедельник–воскресенье, сетка часов, слоты по времени начала/конца и оценке); справа от сетки (на широком экране) — **кольцо «план недели»**: **`plannedPeriodProgress`** суммирует по дням недели ту же метрику; учитываются только дни с **датой ≤ сегодня**; действуют **те же фильтры**, что и у задач на вкладке; если в охвате **нет** ни одной задачи в плане — блок кольца **скрыт**. На узком экране кольцо **выше** сетки (`flex-col-reverse`). Таблица недели **без горизонтального скролла**: семь дней делят ширину контейнера (`minmax(0,1fr)`); заголовки дней и сетка слотов на **одной CSS Grid** с общими колонками (`subgrid` для строки времени), чтобы вертикальные линии совпадали; вертикальная прокрутка — тёмная полоса (`week-grid-v-scroll`); высота часа в сетке задаётся **`HOUR_HEIGHT_PX`** (комфортнее тачу). Текст-пояснение про слоты закреплён **под** сеткой (не внутри блока прокрутки по времени). **Месяц** — календарная сетка и счётчик задач по дням (клик по дню открывает вкладку «День»); **кольцо «план месяца»** — те же правила по дням текущего месяца до сегодня, фильтры, скрытие при нуле охвата; контейнер **`max-w-5xl`** по центру. В шапке — меню аккаунта (отчёты, на вкладке «День» — EOD, настройки, выход); рядом компактная иконка состояния синхронизации (намеренно менее контрастная, чем кнопка аккаунта); ссылка **«Отчёты»** ведёт на **`/app/reports`**. |
| **Отчёты (фаза 5 MVP)** | Считаются **на клиенте** после расшифровки vault (**DR-007**): окно **7 / 30** последних календарных дней (график — по полным дням до вчера); KPI «дни подряд» — **стрик DR-013** с учётом завершённого ритуала End-of-Day (`vault.eodCompletedLocalDates`, см. ниже); доля дней с ≥1 отметкой, сумма отметок; столбчатая диаграмма по дням; таблицы **часто проваленных** — повторы по **серии** (`task.id`, **DR-008**) и до **5 разовых** с просроченным планом в окне. У блоков KPI, гистограммы и таблиц — **иконки подсказок** (`ReportHint`, нативный `title` при наведении). Логика — **`@motivator/core`** (`lib/reporting/vaultAnalytics.ts`). |
| **Фаза 6: End-of-Day, DR-004, анимации** | **EOD:** на вкладке **«День»** кнопка **«Завершить день»** в одной строке с полем создания задачи (дублируется в меню аккаунта в шапке); после отметки завершённого ритуала за календарный день — **«Отчёт за сегодня»** (то же окно сводки). Модалка **DR-002** (`EndOfDayModal`): кольцо прогресса с **процентом внутри** по доле закрытых задач плана на день; **`eodCompletedLocalDates`**, **`eodPreferences`**, глобальный вкл/выкл в настройках, чекбокс задачи «участвует в ритуале». Списки «сделано / не закрыто» — только по **плану на календарный день** (`tasksPlannedForLocalDay` в `@motivator/core`); при «не закрыто (0)» блок подсвечивается зелёным; бэклог — отдельный мягкий блок. Продуктовые «открытые вопросы» — в модалке **«Краткая сводка»**. **DR-004:** второе подтверждение «сделано» на мини-карточке и в модалках создания/редактирования; истечение окна ожидания на клиенте (`VaultProvider`); стили **`animate-dc-pending`**, **`animate-task-done-success`**, **`animate-task-soft-miss`** в `index.css`. Задел под роли **`user` / `beta_tester` / `admin`**: `motivatorRole.ts`, **`app_metadata.motivator_role`** (ограничения UI пока не активны). |
| **Фильтры вида** | Если есть черновики — кнопка **«Черновики»** рядом с фильтром (счётчик), список в модалке; если черновиков нет — кнопка **не показывается**. Кнопка фильтра открывает панель (**группа**, **приоритеты**, **цвет**, **повторы**); на узком экране — полноэкранный лист с затемнением. Опции подбираются из задач **текущей вкладки** (день: план выбранного дня + бэклог; неделя: попадание в любой день недели; месяц: в любой день месяца), с перекрёстным учётом остальных фильтров. Когда панель свернута и выбрано не состояние по умолчанию — строка **чипов** активных условий (снять по одному) и **«Сбросить всё»**. Логика области видимости — `web/src/lib/plannerFilterScope.ts`. |
| **Пересечения слотов** | При изменении даты плана, оценки или времени в модалке задачи — если интервал пересекается с другой незавершённой задачей на тот же день, показывается подтверждение (`@motivator/core` / timeblocking, ключ `app.overlapConfirm`). |
| **Задача** | Заголовок, выполнено; на мини-карточке — полоска **цвета** по палитре; в модалках создания/редактирования выбор **группы** и **цвета** (**кружки**, **HEX**, пипетка) — в раскрывающемся блоке **«Дополнительные настройки»**; **приоритет 1–5**, чек-лист («план работы»): новые пункты попадают **в конец** списка (сверху вниз — порядок добавления); **галочки «выполнено» по пунктам** на мини-карточке дня/бэклога и в **`TaskEditModal`** — только в **календарный сегодня** (тот же контекст, что главная галочка задачи); **добавление и удаление** строк чек-листа в модалке — при праве редактирования и на других днях; дата плана **или** только бэклог; оценка времени в форме **часы + минуты (0–59)** (в vault **`estimatedMinutes`**); для задачи **с датой в плане** оценка обязательна. **Время начала XOR время завершения** (или без времени) — блок «Время» в раскрывашке: два селекта **часы** и **минуты** в стиле интерфейса (без нативного `<input type="time">`). На **мини-карточке** и в **недельной сетке** левая полоска цвета задаётся через опорный **HEX** палитры (`TASK_COLOR_HEX`), чтобы цвет не терялся из‑за JIT Tailwind; текст времени суток — в формате «часы и минуты» (`app.miniCardTimeHm`). Редактирование (`TaskEditModal`) использует те же правила расписания и якоря повтора, что и создание задачи (`taskScheduleValidation`). На телефонах (WebKit) модалка монтируется в **`document.body`** (`createPortal`), на время открытия у **`body`** отключается прокрутка — касания не уходят на список задач под окном. Родитель нельзя закрыть, пока открыты пункты чек-листа. Удаление с непустым чек-листом — подтверждение (`app.confirmDeleteTaskWithChecklist`). |
| **Создание задачи** | Подвал модалки **липкий**: после **первой** попытки «Сохранить» показывается янтарный список недостающего (до этого не отображается); **«Сохранить»** активен при праве редактирования, при ошибках — прокрутка к полю. Правила оценки: для задачи **с датой в плане на день** нужна **непустая корректная** оценка; для **бэклога** (без даты в плане) оценка **не обязательна**, но **некорректный ввод** в полях часы/минуты по-прежнему блокирует сохранение. Снимок формы инициализируется **только при открытии** модалки (пока окно открыто, смена выбранного дня или фильтра группы **не** сбрасывает форму и выбранный цвет). Закрытие без успешного «Сохранить» — диалог подтверждения (не `window.confirm`). **Черновик** в vault только если форма менялась; успешное сохранение удаляет черновик по `removeDraftId`. На **`/app`** доступ к черновикам — кнопка рядом с фильтром, только пока есть хотя бы один черновик (см. строку «Фильтры вида»). Санитизация — `web/src/lib/fieldSanitize.ts`; проверки расписания — **`web/src/lib/taskScheduleValidation.ts`** (черновик без этих проверок). |
| **Настройки** | Пять **названий уровней приоритета** (хранятся в vault), **смена пароля входа** (Supabase `signInWithPassword` + `updateUser`), CRUD групп. Кнопка **«Краткая сводка»** — см. [раздел «Краткая сводка»](#краткая-сводка). Задел под роли: **`web/src/lib/motivatorRole.ts`**, **`app_metadata.motivator_role`** (ограничения UI в плане, фаза 8 дорожной карты). |
| **Группы** | Список групп в vault, фильтр по группе на `/app`, CRUD групп в настройках (группу по умолчанию нельзя удалить). |

## Краткая сводка

Модалка **«Краткая сводка»** открывается из **настроек** (`/app` → настройки → кнопка с текстом из `settings.roadmapTempButton` в `web/src/i18n/locales/`). Доступна **всем залогиненным** пользователям; скрытие дорожной карты по ролям — в плане (см. фазу **8** в `MVP_PHASES_PLANNED`).

### Содержимое модалки

1. **Верх:** кольцевая диаграмма прогресса до **MVP 1.0.0** — доля закрытых фаз **0–6** из **0–13** и подпись с процентом (данные из тех же констант, что и блоки ниже).
2. **Уже реализовано** — массив **`IMPLEMENTED_MVP_PHASES`** в [`productRoadmap.ts`](./src/data/productRoadmap.ts): фазы **0–6** с кратким описанием и раскрывающимся списком **«Подробнее»** (фактическое поведение сборки; по возможности согласовано с релиз-нотами).
3. **План до 1.0.0** — **`MVP_PHASES_PLANNED`**: фазы **7–13** (источник смысла — [`obsidian-motivator/17-План-реализации-MVP.md`](../obsidian-motivator/17-План-реализации-MVP.md)); у фаз есть блок **«Простыми словами»** (`plain`) и **«Подробнее»**.
4. **Релиз-ноты** — **`RELEASE_NOTES_BLOCKS`**: хронология по **календарным датам** работ (`dateLabel`, формат `YYYY-MM-DD`); в модалке дни отсортированы **по убыванию** (новее выше). Внутри одного дня — несколько элементов **`items`** (несколько выпусков за день); у каждого — **`releasedInVersion`** (semver из `web/package.json` на момент выпуска, **без** суффикса `+git`), список **`changes`**, опционально **`plainBullets`** под раскрывашкой **«Подробности простым языком»**. Правила ведения — в комментарии над константой в `productRoadmap.ts` и в [правиле репозитория](../.cursor/rules/pre-commit-docs-roadmap.mdc).
5. **Идеи на потом** — **`IDEAS_LATER_ENTRIES`**: черновой backlog после **1.0.0**; расширение согласовать с [`obsidian-motivator/15-Идеи-для-развития.md`](../obsidian-motivator/15-Идеи-для-развития.md) (в модалке только перенесённые карточки).
6. **Открытые вопросы** — продуктовые заметки без срока: строки **`settings.roadmapOpenQuestion*`** в **`web/src/i18n/locales/ru.json`** и **`en.json`** (заголовок секции — `settings.roadmapOpenQuestions` / `roadmapOpenQuestionsHint`).

Контент **`productRoadmap.ts`** не должен противоречить текущей сборке; при смене продуктовых формулировок в «Открытых вопросах» правятся обе локали.

### Перед коммитом (документация и «Краткая сводка») — **обязательно**

**Жёсткое правило репозитория:** перед **любым** `git commit` (включая мелкие и «технические») нужно пройти чеклист и **включить результат в тот же коммит**. Иначе документация и модалка **«Краткая сводка»** отстают от кода. Правило для Cursor агентов: `.cursor/rules/pre-commit-docs-roadmap.mdc` (`alwaysApply`).

1. **`web/README.md`** — таблица «Возможности», **версия =** [`package.json`](./package.json), при необходимости разделы ниже (локально, Vercel, vault).
2. **`web/src/data/productRoadmap.ts`** — модалка **«Краткая сводка»** в настройках:
   - **`RELEASE_NOTES_BLOCKS`** — см. `.cursor/rules/pre-commit-docs-roadmap.mdc`: дата = день работ; новый день — новый `dateLabel`; несколько `items` за день; у каждого элемента **`releasedInVersion`** (semver без `+git`); `plainBullets` для сути без кода; при внутренних правках без UI — краткий пункт или пометка «техн. / внутреннее».
   - **`IMPLEMENTED_MVP_PHASES`** / **`MVP_PHASES_PLANNED`** — если изменился реализованный или планируемый scope.
   - **`IDEAS_LATER_ENTRIES`** — при добавлении идей «после релиза»; держать в согласовании с **`obsidian-motivator/15-Идеи-для-развития.md`** (в модалке пользователь видит не все заголовки Obsidian одновременно — только то, что перенесено в массив).
   - Проверить в локалях строки секции **«Открытые вопросы»** в модалке (`settings.roadmapOpenQuestion*` в `src/i18n/locales/*.json`), если трогали продуктовые формулировки.

Контент дорожной карты **не должен противоречить** текущей сборке.

Сборка: в **`vite.config.ts`** задан `build.chunkSizeWarningLimit` (ослабление предупреждения Vite о размере одного чанка).

## Supabase

1. Создайте проект в [Supabase](https://supabase.com/).
2. В SQL Editor выполните миграцию [`supabase/migrations/001_user_vault.sql`](./supabase/migrations/001_user_vault.sql).

## Seed и повторный вход

- После **выхода** из аккаунта seed удаляется из браузера. Чтобы открыть те же задачи, на экране **«Ключ шифрования»** выберите **«У меня уже есть seed»** и вставьте сохранённый base64 + тот же пароль деривации (если использовали).
- Новый сгенерированный seed после выхода **не** расшифрует старый vault на сервере.

## Vercel

Монорепозиторий: задайте **корень репозитория** как корень проекта в Vercel (не только `web`), иначе workspace-пакет `@motivator/core` не попадёт в сборку.

1. **Root Directory** → корень репозитория (`planner/`).
2. **Install Command**: `npm install`
3. **Build Command**: `npm run build -w web`
4. **Output Directory**: `web/dist`
5. Environment Variables: те же `VITE_SUPABASE_*`, что в `.env.local`.

Если по историческим причинам корень проекта в Vercel остаётся только `web`, перед сборкой нужно вручную обеспечить доступ к `packages/motivator-core` (например, скопировать структуру монорепозитория или собирать из полного clone с установкой из родительской папки).

Сборка рассчитана на **Vite 7** (Rollup): в корне репозитория в `package.json` задан **`overrides`** на `vite@7.3.3`, а `@vitejs/plugin-react` — ветка **5.x**, совместимая с Vite 7. Так на Vercel/Linux не подтягивается Vite 8 с нативным **Rolldown**, из‑за которого часто возникает `MODULE_NOT_FOUND` при загрузке нативного binding Rolldown.

Файл [`vercel.json`](./vercel.json) отправляет все пути на `index.html` для SPA.

## Синхронизация vault

До завершения первой загрузки с Supabase поля задач **неактивны** (нет гонки с созданием пустого vault). Строка статуса под заголовком: загрузка / сохранение / время последнего успешного сохранения. При ошибке расшифровки показывается сообщение, редактирование блокируется (флаг **`decryptFailed`**).

Запись ciphertext на сервер **отложена** (debounce, см. `VAULT_REMOTE_SAVE_DEBOUNCE_MS` в пакете `@motivator/core`): локальное состояние обновляется сразу, сеть — после паузы; перед выходом из аккаунта выполняется принудительный flush.

## Контракт vault (актуальная схема)

Хранится как один зашифрованный JSON. Актуальная версия — **7** (типы в `@motivator/core`, модуль `vault/types`). Кратко:

- **`priorityLabels`**: пользовательские строки для уровней **1–5**.
- **`groups`**: проекты/группы.
- **`tasks`**: для каждой задачи — **`priorityRank`** (1–5), **`scheduledLocalDate`** (`YYYY-MM-DD` или `null`), **`checklist`** (массив в порядке отображения сверху вниз; новые пункты добавляются в конец), **`estimatedMinutes`**, **`timeMode`** (`none` | `start` | `end`), **`timeMinutesFromMidnight`**, **`recurrence`** / **`recurrenceAnchorLocalDate`** (правило повтора: каждый день / каждые N дней / дни недели, и якорь серии), **`completedOccurrenceLocalDates`** — для повторов список дней, на которые отмечено выполнение вхождения (глобальный **`done`** при повторе по-прежнему может архивировать серию целиком). Идентификатор серии для отчётов (**DR-008**) совпадает с **`id`** задачи. Плюс группа, цвет, заголовок, **`done`**, опционально **`includeInEodRitual`** (участие в ритуале EOD). **DR-004:** опционально **`doubleConfirmEnabled`**, **`doubleConfirmIntervalMinutes`**, **`doubleConfirmGraceMinutes`**, **`doubleConfirmPending`** (ожидание второго подтверждения за конкретный календарный день).
- **`drafts`**: черновики формы создания задачи; запись выполняется только при закрытии модалки без сохранения и при **фактических** изменениях формы относительно состояния при открытии (см. раздел «Создание задачи» в таблице возможностей).
- **`eodCompletedLocalDates`**, **`eodPreferences`**: ритуал End-of-Day (схема **v6+**).

Цепочка миграций: снимки **v1** … **v6** при открытии приводятся к **v7** через **`normalizeVault`** (например старые vault без повтора получают **`recurrence: null`**, **`drafts: []`**; до **v6** добавляются поля EOD по умолчанию; **v7** — поля двойного подтверждения **DR-004**).

### Миграция v3 → v4 (поля)

| Было (v3) | Стало (v4) |
|-----------|------------|
| `prioritySystem` + `priorityLevel` / `eisenhowerQuadrant` | одно поле **`priorityRank`** (эвристика: уровни 1–3 → ранги 1–3; квадранты q1…q4 → ранги 1,2,4,5; Inbox → 3) |
| `subtasks` | **`checklist`** (тот же массив объектов) |
| — | **`scheduledLocalDate`**: для всех мигрированных задач **`null`** (бэклог); пользователь заново разносит по дням |
| — | **`estimatedMinutes`**, **`timeMode`**, **`timeMinutesFromMidnight`**: по умолчанию пусто / `none` / `null` |
