# Приватность и безопасность

Критично для целевой аудитории ([[02-Целевая-аудитория]]).

## Модель угроз (целевая)

- Сервер и оператор хостинга **не должны** читать содержимое задач, заметок ритуала и отчётных данных в открытом виде.
- Ключ шифрования **не покидает контроль пользователя** в открытом виде при нормальной эксплуатации.

## Клиентское шифрование

- Все пользовательские данные **шифруются на клиенте** через **Web Crypto API** (**AES-GCM**).
- На сервер (Supabase) уходит **только ciphertext** и служебные метаданные, необходимые для синхронизации (идентификаторы, версии, timestamps — уточняется при проектировании схемы).

## Хранение ключа и деривация

Актуальная схема: [[12-Журнал-решений#DR-005 — Ключ шифрования экспорт QR восстановление на устройстве]].

- Генерируется **seed** (32 байта); мастер-ключ для **AES-GCM** выводится через **PBKDF2(seed + опциональный пароль пользователя)**.
- **Экспорт для пользователя** — **base64 от seed** (не от сырого ключа); файл как альтернатива тому же содержимому.
- Локальное хранение seed/ключевого материала — в **IndexedDB**; детали извлечения для экспорта реализуются без противоречия с тем, что в UI передаётся seed.

## Новое устройство

- **QR-код** переносит **тот же seed** (base64); после входа через **Supabase Auth** на новом устройстве пользователь восстанавливает ключ из seed.

## Управление данными

- **Полное удаление аккаунта** и всех данных на сервере **одним действием**.
- **Экспорт всех данных**: формат **JSON** + расшифровка **локально** на устройстве экспорта.
- Потеря ключа и сценарий сброса: [[12-Журнал-решений#DR-006 — Потеря ключа тексты и сценарии]].
