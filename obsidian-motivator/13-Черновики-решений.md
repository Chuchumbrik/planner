## DR-012 (черновик) — Offline-first: конфликты синхронизации

**Контекст:** клиентское шифрование + локальный кэш + запись на сервер при появлении сети; возможны параллельные правки одной сущности на двух устройствах.

**Что нужно решить**

- Гранулярность конфликта: задача целиком, отдельные поля, вложенные подзадачи.
- Стратегия по умолчанию: **last-write-wins (LWW)** по `updated_at`, версионирование (`version`), **операционные преобразования** / CRDT (обычно избыточно для MVP), ручное разрешение «показать два варианта».

**Варианты для обсуждения**

| Подход | Плюсы | Минусы |
|--------|--------|--------|
| LWW по серверному или локальному времени | Простота | Тихая потеря правок |
| Версия + отказ записи при mismatch + UX слияния | Меньше потерь | Сложнее UI |
| Локальная очередь операций (patch-level) | Точнее | Дольше реализовать |

**Связь:** [[08-Архитектура]], [[11-Модель-данных-концептуально]].

**Черновик гипотезы (не утверждено):** начать с **версии сущности + LWW с предупреждением** при расхождении только если появится жалобы на потерю данных в альфе.

## DR-013 (черновик) — «Запланировано на день» для стрика (уточнение к DR-001)

**Контекст:** в [[12-Журнал-решений#DR-001 — Стрик что считается успешным днём]] успешный день допускает ветку «**0 задач запланировано**». Нужно формально определить «запланировано», чтобы не было игры системой.

**Что нужно решить**

- Учитываются только задачи с **датой «на сегодня»** в календаре?
- Или всё, что попало в пользовательский вид «Сегодня» / inbox дня?
- Как **повторяющиеся задачи** и **задачи без даты**, перенесённые вручную на сегодня?
- Часовой пояс: календарный день пользователя из профиля?

**Черновик гипотезы (не утверждено):** «Запланировано на день N» = задача имеет явное поле **planned_for_date = N** (или эквивалент в модели); повторяющиеся создают вхождение на дату; задачи только в Inbox без даты **не считаются** запланированными на день.

## DR-014 (черновик) — Push-уведомления (PWA)

**Контекст:** напоминания задач, второй шаг двойного подтверждения ([[12-Журнал-решений#DR-004 — Двойное подтверждение пропуск второго шага]]), ритуал вечером.

**Что нужно решить**

- **Тихие часы:** глобально и пер-тип уведомлений?
- Лимит **push в день** / анти-спам при большом числе задач.
- Поведение при **отказе в разрешении:** только in-app / e-mail не в MVP?
- Доставка через **service worker** + политика при закрытом приложении (ограничения ОС — документировать для поддержки).

**Черновик гипотезы (не утверждено):** тихие часы 22:00–08:00 по локали пользователя по умолчанию, настраиваемо; максимум **N** напоминаний в час без группировки «батчем» на MVP.

## DR-015 (черновик) — Оплата премиума (РФ) и момент подключения

**Контекст:** цена зафиксирована ориентиром в [[12-Журнал-решений#DR-010 — Цена премиума]]; провайдер не выбран.

**Что нужно решить**

- Платёжный провайдер с юридически корректной моделью для **ИП/ООО в РФ**, чеки **54-ФЗ**, НДС при необходимости.
- Подписка vs разовый год — технически одна система или две.
- **Когда включать в roadmap:** вместе с MVP (если премиум «заглушка»), закрытая бета без оплаты, или релиз только free до готовности биллинга.

**Черновик гипотезы (не утверждено):** MVP **без реальной оплаты**: флаг премиума вручную / инвайт для тестов; биллинг — отдельный этап после юридических документов [[12-Журнал-решений#DR-009 — Юридическое оформление MVP]].

## Как перевести черновик в журнал

1. Утвердить формулировку с владельцем продукта.
2. Добавить в [[12-Журнал-решений]] новую запись **DR-0XX** с датой, как в принятых решениях.
3. Удалить или сократить соответствующий раздел здесь; при необходимости оставить одну строку «перенесено в DR-0XX от ДД.ММ.ГГГГ».
4. При необходимости обновить [[10-Каталог-функций-и-взаимодействий]], [[08-Архитектура]], [[11-Модель-данных-концептуально]].
