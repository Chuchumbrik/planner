# Каталог функций и взаимодействий

Этот документ — **операционное описание продукта**: что есть в MVP, как модули связаны, какие данные пересекаются.

---

## Онбординг и настройки профиля

**Назначение:** быстро зафиксировать систему работы пользователя без перегруза.

| Элемент | Поведение |
|---------|-----------|
| Wizard первого запуска | Выбор системы приоритетов (см. ниже), согласие на end-of-day, время ритуала |
| Настройки профиля | Смена языка (структура i18n), системы приоритетов, глобальных флагов ритуала, уведомлений |
| Выход / смена устройства | Связано с [[06-Приватность-и-безопасность]] (ключ, QR) |

**Связи:** задачи наследуют систему приоритетов из профиля; ритуал использует глобальный флаг и время.

---

## Системы приоритетов

Два режима (выбор при первом запуске, смена в настройках):

1. **Уровни 1–3** — простая шкала (или три категории — уточнение в [[99-Открытые-вопросы-к-команде]] при необходимости UX).
2. **Матрица Эйзенхауэра** — ось срочности × важности (четыре квадранта).

**Взаимодействие:** каждая задача имеет поля приоритета в активной системе; отчёты могут группировать по приоритету в зависимости от выбранной визуализации.

## Задачи, подзадачи, декомпозиция

| Сущность | Описание |
|----------|----------|
| Задача | Основная единица планирования на день/период |
| Подзадача | Вложенная единица для декомпозиции; наследует или переопределяет контекст (цвет/группа — правила в [[11-Модель-данных-концептуально]]) |
| Декомпозиция целей | UX-поток разбиения крупной цели на подзадачи без отдельной «игровой» сущности |

**Взаимодействие:** выполнение подзадач может влиять на прогресс родителя (логика прогресса — зафиксировать в реализации).

## Организация: цвета, группы/проекты

- **Цвет** — визуальная метка (список, календарные представления если появятся).
- **Группа/проект** — логическая кластеризация для фильтров и **отчёта по группам**.

**Взаимодействие:** отчёт «по группам» агрегирует метрики выполнения по этому полю.

## Двойное подтверждение

**Назначение:** снизить импульсивные «галочки» и ложноположительные закрытия.

| Параметр | Поведение MVP |
|----------|----------------|
| Включение | По **каждой задаче** отдельно |
| Задержка по умолчанию | **10 минут** после первого подтверждения до второго запроса |
| Настраиваемая задержка | Пользователь задаёт интервал при создании/редактировании задачи |
| Канал напоминания | **PWA push** (настройки частоты и тихих часов — в продуктовых решениях) |

**Цепочка:** напоминание → первое действие (выполнил/нет) → таймер → второе подтверждение → запись в историю для отчётов и [[#Эмоциональные анимации|анимации]].

## End-of-day ритуал

**Назначение:** короткая КПТ-рефлексия по дню.

| Параметр | Поведение |
|----------|-----------|
| Глобальный выключатель | В настройках профиля |
| Участие задачи | Флаг на задаче |
| Вопросы за сессию | **3 вопроса** из выбранных пользователем шаблонов |
| Базовый контент | **10 КПТ-вопросов** как стартовый пул |
| Время запуска | Из wizard / настроек |

**Взаимодействие:** подтягиваются только задачи с флагом ритуала и фактом участия в дне (выполнялись/обсуждались — правило отбора см. [[99-Открытые-вопросы-к-команде]]).

## Эмоциональные анимации

**Назначение:** эмоциональная регуляция без морализаторства.

| Тип | MVP |
|-----|-----|
| Успех | Радость / подбадривание (**пара анимаций**) |
| Неуспех | Мягкая мотивация (**пара**) |
| Freemium | Выбор из **4** вариантов на бесплатном плане ([[05-Freemium]]) |

**Триггеры:** завершение подтверждения задачи, возможно закрытие ритуала (не дублировать перегрузом).

## Уведомления (PWA)

- Напоминания о задачах.
- Второй шаг двойного подтверждения.
- Напоминание о времени **end-of-day** (если включено).

**Взаимодействие:** настройки уведомлений ограничивают спам; офлайн-очередь синхронизируется при подключении ([[08-Архитектура]]).

| Отчёт | Содержание MVP |
|-------|----------------|
| Процент выполнения | День / неделя / месяц |

## Админ-панель — ✅ Реализовано

**Как работает:** защищённый раздел `/admin/dashboard` (роль `admin` обязательна). Два таба: **Сводка** и **Пользователи**. Данные тянутся из Edge Function `admin-motivator-roles` через пять хуков с AbortController.

| Элемент | Описание |
|---------|----------|
| `AdminDashboardPage` | Точка входа, `RequireVault` + `RequireAdmin` |
| `AdminDashboardSummaryTab` | Hero-строка 4 KPI (total, reg 7d, MAU, churn) + вторичная сетка + графики |
| `AdminDashboardActivityChart` | Recharts BarChart — уникальные пользователи по UTC-дням; фильтр по периоду и роли |
| `AdminKpiChartZone` | Recharts AreaChart — месячная динамика выбранного KPI (total/reg/MAU/churn) |
| `AdminMotivatorRolePanel` | Таблица пользователей: role-бейджи, иконки vault/push/stale, модалка смены роли |
| `useAdminOverview` | Хук сводки: total, MAU, churn, vault, push, defects |
| `useAdminMotivatorUsers` | Хук списка: lazy-load (один раз), сбрасывается при смене supabase-клиента |
| `useAdminActivityChart` | Хук графика активности; перезагружается при смене days/role |
| `useAdminKpiTrend` | Хук KPI-тренда; активируется кликом по hero-карточке |
| `adminDashboardMetrics.ts` | Чистая логика: фильтрация сегментов, сортировка, isVaultStale |

**Ключевые инварианты:**
- Все хуки используют `AbortController` — нет race condition при быстрой навигации
- `ADMIN_ROLES_FN` — единственная константа в `adminMonitoringConstants.ts`
- Фильтры `inactive7`/`inactive30` не включают пользователей с `null` last_sign_in_at
- Churn показывает `—` при `total_users === 0` (не 100%)

**Связи:** [[08-Архитектура]], [[06-Приватность-и-безопасность]]

## Обсуждения (admin discussions) — ✅ Реализовано

**Как работает:** страница `/admin/discussions` (Phase 7.9–7.11) — внутренний инструмент-воркспейс для открытых вопросов и решений в виде тредов. Поверх Edge Function `admin-discussions` (10 actions, service role; таблицы `admin_discussions`/`_replies`/`_read`/`_subscribers`). Доступ: `admin` — всё; `beta_tester` — чтение + reply; `user` — 403 (см. [[20-Phase-7-План-краткой-сводки#4. Архитектурные решения]]). Источник истины для решений остаётся Obsidian-журнал — синхрон вручную через статус «К журналу».

| Элемент | Описание |
|---------|----------|
| `AdminDiscussionsPage` | Точка входа `RequireVault` + route под `RequireTesterPreview`; роутинг список ↔ тред по `:id` |
| `DiscussionList` | Список тредов: превью (`previewLine`), reply_count, relative time, unread-бейдж; сортировка open→pending-journal→synced→archived |
| `DiscussionThread` | Тред: markdown через `AiMarkdown`, ответы, reply-форма (Ctrl+Enter), кнопки переходов resolve/mark-synced/archive, тоггл подписки на уведомления, inline copy+sync блока «К журналу» |
| `CreateDiscussionModal` | Создание: заголовок + текст + опц. `linked_version` |
| `ResolveDiscussionModal` / `SyncDiscussionModal` | Переходы open→pending-journal (с резюме) и pending-journal→synced (с DR-XXX); модалка sync — резервный путь к inline-форме в треде |
| `useDiscussions` / `useDiscussionThread` | Хуки списка и треда с AbortController; тред делает `mark-read` fire-and-forget; отдаёт `subscribed` |
| `discussionStatusMeta.ts` | Чистая логика: маппинг статус→{цвет, иконка, label} (§8.4) |
| `discussionsApi.ts` | `invokeDiscussionsFn` + типы + `isStaleStatusError` (409 bad_status → friendly) |

**Ключевые инварианты:**
- Записи только через Edge (service role); прямой клиент — лишь SELECT по RLS (admin/beta)
- `list` отдаёт `body` для превью; `reply` бампает `reply_count`/`last_reply_at` триггером
- Возврат к списку (`backToList`) перезагружает его — снимает unread-бейдж после открытия треда
- Статусные переходы admin-only (create/resolve/mark-synced/archive); reply/read — и для beta

**Связи:** [[20-Phase-7-План-краткой-сводки]], [[12-Журнал-решений]], [[08-Архитектура]]

## Push-уведомления Discussions — ✅ Реализовано

**Как работает (Phase 7.10):** при новом ответе в треде БД-триггер `discussion_reply_notify` (`AFTER INSERT` на `admin_discussion_replies`) через `pg_net` (`net.http_post`) вызывает Edge `admin-discussions-notify`, которая рассылает web-push (VAPID) всем подписчикам треда (`admin_discussion_subscribers`), кроме автора ответа. Авто-подписка автора треда и каждого ответившего (idempotent upsert в `admin-discussions` actions create/reply). В UI треда — тоггл «Подписаться/Отписаться». In-app бейдж непрочитанных рядом с пунктом меню «Обсуждения».

| Элемент | Описание |
|---------|----------|
| `012_discussion_notify_trigger.sql` | `pg_net` + триггер-функция `notify_discussion_reply()` (SECURITY DEFINER); GUC `app.settings.supabase_url`/`service_role_key`, fire-and-forget |
| Edge `admin-discussions-notify` | Auth = `Bearer SUPABASE_SERVICE_ROLE_KEY` (строковое сравнение); читает title + подписчиков + `push_subscriptions`, шлёт payload `{title, body, url:/admin/discussions/{id}, tag}` (ru) |
| `UnreadDiscussionsBadge` | Берёт `supabase` синглтон, `action:'list'` → счёт `unread`; bell-shake (`animate-bell-shake`) при росте; рендерит `null` при 0 или без supabase |
| `ShellAdminNav` | Монтирует бейдж поверх иконки пункта `admin-discussions` |
| `DiscussionThread` тоггл | `subscribed` + `onSubscribeToggle` → Edge `subscribe`/`unsubscribe` + reload треда |

**Ключевые инварианты:**
- Push не блокирует сохранение reply (триггер fire-and-forget; нет секретов → тихо пропускает)
- Автор reply не получает уведомление о собственном ответе
- Бейдж не требует props и не ломает существующие тесты `ShellAdminNav` (при `supabase=null` → `null`)

**Связи:** [[20-Phase-7-План-краткой-сводки]], [[08-Архитектура]]

## Sync workflow «К журналу» (Discussions) — ✅ Реализовано

**Как работает (Phase 7.11):** при статусе `pending-journal` тред показывает amber-блок (pulse-чип статуса, §8.6) с подсказкой перенести итог в `obsidian-motivator/12-Журнал-решений.md` как DR-XXX. Рядом с `resolution_summary` — кнопка copy-to-clipboard (с fallback на `document.execCommand` в не-HTTPS). Inline-поле `linked_journal_entry` (DR-XXX) + кнопка «Перенести в журнал» шлёт `mark-synced` прямо из треда (модалка `SyncDiscussionModal` остаётся резервным путём). Список треды `pending-journal` поднимает вверх (сортировка в Edge `list`).

| Элемент | Описание |
|---------|----------|
| `DiscussionThread.inlineSync()` | `action:'mark-synced'` + `linkedJournalEntry` (trim); `bad_status` → `staleStatus`; success → `onChanged()` |
| `DiscussionThread.copySummary()` | `navigator.clipboard.writeText` с fallback; флеш «Скопировано!» 2 с |
| i18n | `copySummary`/`copiedToClipboard`/`linkedJournalEntryInline`/`syncInline` (ru/en) |

## Форма заведения дефекта — ✅ Реализовано

**Как работает:** модалка `FileDefectModal` для admin/beta_tester создаёт GitHub Issue через Edge Function `file-defect`. При открытии видны три ввода (тип — предвыбран `bug`, заголовок, описание) + зона вставки скриншотов (Ctrl+V / drag-drop / выбор файла, до 2 изображений ≤3 МБ) и кнопка отправки. Необязательные поля («шаги», «ожидаемое», «фактическое») и технический блок (маршрут/UA/устройство) скрыты за раскрывающимися `Collapsible`.

Прогрессивное раскрытие (UX-pass 0.7.30): `Collapsible` раскрывается/сворачивается плавно через `grid-template-rows: 0fr↔1fr` (220 мс, `prefers-reduced-motion` отключает), контент остаётся в DOM (введённые данные сохраняются между сворачиваниями), а в свёрнутом виде помечен `inert` — недоступен с клавиатуры и для ассистивных технологий; шеврон поворачивается; `aria-expanded` сообщает состояние.

Состояния отправки: `idle → submitting → success | error`. `success` показывает `role="status"`/`aria-live="polite"` блок с сообщением и кликабельной ссылкой на созданный issue (тач-таргет ≥44px), форма сама не закрывается — только по кнопке. `error` показывает `role="alert"` с переведённым сообщением, данные сохранены, доступен повтор. Идемпотентность: `onSubmit` и `handleClose` блокируются во время `busy` — двойной клик / повтор / закрытие во время отправки не создают второй дефект.

| Элемент | Описание |
|---------|----------|
| Компонент | `web/src/components/FileDefectModal.tsx` |
| Хук отправки | `web/src/hooks/useFileDefect.ts` → Edge `file-defect` |
| Анимация раскрытия | `.collapsible-region` в `web/src/index.css` (grid-rows, reduced-motion) |
| Цвета | токены дизайн-системы: `text-error`/`ring-error` (ошибки), `tertiary` (предупреждение перезаписи шаблона) |
| Тесты | `web/src/components/FileDefectModal.test.tsx` — валидация, идемпотентность (double-click + retry), success/error a11y, двуязычность лейбла раскрытия |

**Связи:** [[06-Приватность-и-безопасность]] (диагностический инструмент, не часть зашифрованного vault)
