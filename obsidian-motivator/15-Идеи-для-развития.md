# Идеи для развития

> Накопительный список **направлений и гипотез** для продукта. Здесь нет обязательств в релиз; приоритизация — отдельно. Связь с зафиксированным объёмом: [[03-Scope-MVP-и-бэклог]], принятые решения: [[12-Журнал-решений]].

**Статус:** концепт-заметка, 2026-05-12

---

## 1. Напоминания о неиспользованных функциях (feature discovery)

**Суть:** мягко подсвечивать пользователю возможности приложения, **которыми он ещё не пользовался** — на случай, что он о них не знает или не дошёл до сценария.

**Возможные направления реализации (для проработки):**

- Учёт **событий** в клиенте (feature flags / «уже открывал раздел X», «уже переключал язык») с хранением в vault или в `localStorage` (и политикой приватности).
- **Контекстные подсказки:** баннер, тост, карточка в пустом состоянии, один раз после онбординга — без навязчивости.
- **Цель:** снизить «скрытую сложность» (DnD, режим Эйзенхауэра, фильтр групп, подзадачи и т.д.) без спама.

**Риски:** раздражение от подсказок — нужны частота, отключение в настройках и эвристики «показали один раз».

---

## 2. Отдельная область «Привычки»

**Суть:** на экране задач (или отдельном разделе) выделить **«Привычки»** — сущность отличная от разовых задач: **повторяющиеся действия** с собственным потоком создания и, возможно, собственной моделью в vault (расширение схемы отдельным инкрементом).

**Черновые вопросы при создании привычки** (как ориентир для UX и данных):

| Вопрос | Зачем |
|--------|--------|
| **Каким человеком ты хочешь стать в рамках этой привычки?** | Связь с мотивацией и идентичностью (контекст для напоминаний и отчётов). |
| **Как часто ты готов ею заниматься** (например, в днях / интервал повторения)? | База для календаря, стрика привычки, «ожидаемых» отметок без смешения с задачами-проектами. |
| **Сколько времени ты готов тратить на неё в день?** | Ограничение реализма, планирование блоков времени, возможные будущие интеграции с расписанием дня. |

**Дальнейшие темы для продукта:** отличие от обычной задачи с повтором, связь с **end-of-day** и отчётами, напоминания (push), визуализация серии выполнений.

## 3. Поиск по задачам (после MVP)

**Суть:** поле **поиска** на экране планировщика (или в шапке), фильтрация списка задач и/или подсветка совпадений по заголовку, при необходимости — по группе, тегам, тексту чек-листа (объём уточнять при реализации).

**Зачем:** быстрый доступ к задаче при большом бэклоге и длинном списке без прокрутки фильтров.

**Заметки для реализации:** клиентский поиск по уже загруженному vault; при росте данных — дебаунс, опционально индекс в памяти; не отправлять запросы поиска на сервер в явном виде (конфиденциальность).

---

## 4. Интеграция с внешними календарями (после MVP)

**Суть:** связать планировщик с **системными и облачными календарями** (Google Calendar, Microsoft 365 / Outlook, Apple/iCloud и др.), чтобы не дублировать контекст вручную и видеть **задачи и внешние события** в одном временном контуре.

**Возможные направления (на выбор приоритизации, не как единый пакет):**

| Направление | Зачем |
|-------------|--------|
| **Экспорт / «подписка»** на ленту планов (например **ICS** по защищённой ссылке или выгрузка файла) | Показ задач с временем в стороннем календаре без полного доступа аккаунта |
| **Импорт событий** (read-only слой: занятость, встречи) | Планирование задач с учётом уже занятых слотов; минимизация пересечений «вслепую» |
| **OAuth + официальные API** (Google, Microsoft) | Двусторонняя синхронизация или расширенный импорт — только после отдельной продуктовой и **privacy**-оценки |

**Ограничения продукта:** клиентское шифрование vault — любая синхронизация с облаком календаря потребует **явной модели**: что уходит наружу (только время/заголовок vs полный текст), куда пишутся изменения, как разруливаются конфликты. Для MVP это **не входит** в объём [[03-Scope-MVP-и-бэклог]]; пункт фиксируется как **идея на v1.1+**.

## 5. Персонаж-тамагочи и привязка к группам (после MVP)

**Суть:** лёгкая **геймификация без давления** — настраиваемый **персонаж** (тамагочи), чьи потребности **кормление, гигиена, настроение** и т.д. отражают или усиливают мотивацию. Для **каждой группы задач** можно задать, **какое действие персонажа** «получает» выполнение задач этой группы (например: рабочие задачи → «опыт», домашние → «уют»).

**Черновые направления:** отдельный экран/виджет персонажа, редкие ненавязчивые напоминания, опциональное отключение в настройках. Сохранение состояния персонажа — в vault или локально; синхронизация между устройствами — отдельное решение.

**Границы:** не превращать планировщик в игру; не наказывать за пропуски; для MVP **не входит** в [[03-Scope-MVP-и-бэклог]].

## 6. MCP для AI (после MVP)

**Суть:** после стабилизации ядра продукт можно связать с экосистемой **AI-ассистентов** через **Model Context Protocol (MCP)** — стандартный способ отдавать модели **структурированный контекст** и **безопасные действия** (tools), не смешивая это с обходом клиентского шифрования без явного согласия пользователя.

**Зачем пользователю:** голосом или в чате IDE («разложи бэклог на неделю», «что на сегодня», «подготовь формулировки задач из заметки») — ассистент опирается на **реальные** задачи и календарный контекст приложения, а не на угадывание.

**Черновые направления проработки (не пакет обязательств):**

| Направление | Заметки |
|-------------|---------|
| **MCP-сервер как локальный/доверенный мост** | Чтение расшифрованного vault **только там, где уже есть ключ** (например десктоп или доверенная среда); сервер не хранит master-key на облаке. |
| **Набор tools** | Только то, что можно формализовать: список задач за день, черновики, группы; опционально «предложить черновик задачи» без автозаписи в vault до подтверждения в UI. |
| **Privacy и границы** | Явное включение в настройках; аудит того, что уходит в LLM; запрет полной выгрузки vault наружу по умолчанию. |
| **Интеграции** | Cursor / Claude / другие клиенты с MCP; возможно отдельный лёгкий адаптер поверх существующего `@motivator/core`. |

**Дополнение — ИИ через HTTP API (BYOK):** помимо MCP возможен сценарий **прямого** вызова моделей по API (пользовательский ключ, выбор провайдера / OpenAI-совместимый endpoint) для подсказок внутри UI. Те же ограничения: никакой автоматической отправки всего vault; только явно разрешённый пользователем контекст; отдельная продуктовая и privacy-оценка до реализации. Может сосуществовать с MCP как другой слой доступа к ИИ.

**Границы:** не входит в MVP [[03-Scope-MVP-и-бэклог]]; до реализации нужны отдельные **DR** по безопасности и UX подтверждений.

## 7. Режимы просмотра задач на «День» (после MVP)

**Суть:** сейчас вкладка «День» — преимущественно **список** (план + бэклог). На будущее — дать пользователю **переключаемые представления** одних и тех же задач на выбранную дату, без дублирования данных в vault.

**Примеры режимов (гипотезы):**

| Режим | Зачем |
|--------|--------|
| **Список** (как сейчас) | Быстрый обзор, сортировки, бэклог рядом |
| **По часам** | Вертикальная **шкала дня** (аналогично колонке недели в таймблокинге): задачи с временем начала/конца — в слотах; без времени — отдельная полоса «без времени» или верх/низ экрана |
| **Компактный** | Уменьшенные карточки / одна колонка для просмотра на маленьком экране |

**Заметки для реализации:** общая фильтрация по `taskOccursOnDate` / слотам уже есть в `@motivator/core`; UI — переключатель вида + сохранение предпочтения в `localStorage` или настройках; повторы и пересечения — те же правила, что на неделе.

**Границы:** не входит в [[03-Scope-MVP-и-бэклог]]; приоритет после стабилизации MVP и базового календаря.

## 8. Шаблоны создания задач: порядок и обязательные поля (после MVP)

**Суть:** развить идею **шаблонов задач** (сохранённые заготовки полей) так, чтобы форма быстрого создания можно было подстроить под привычный сценарий.

**Дополнения к концепции (не реализация в текущем MVP):**

| Направление | Зачем |
|-------------|--------|
| **Порядок полей** | Пользователь или шаблон задаёт, **в каком порядке** показываются поля в модалке создания (что сверху — заголовок, время, группа, чек-лист и т.д.), чтобы при повторяющихся типах задач сразу попадать в нужные поля. |
| **Обязательность** | Для выбранного шаблона или набора правил — отметить поля **обязательными к заполнению** до сохранения задачи; снижает «пустые» карточки там, где контекст важен. |

**Заметки для будущей проработки:** хранение порядка и флагов — в vault или в настройках профиля; нужны ли **глобальные** умолчания и **переопределение на шаблон**; как сочетается с «быстрым» созданием без шаблона. Не входит в [[03-Scope-MVP-и-бэклог]].

## 9. Тестовые аккаунты с простым seed (после MVP)

**Суть:** завести **отдельные учётные записи** для проверки приложения (ручные сценарии, регрессия, при необходимости автоматизация), у которых **seed** для клиентского шифрования vault **намеренно простой** — короткая известная команда фраза, зафиксированная только во внутренней документации команды.

**Зачем:** одинаково расшифровать vault на разных машинах и прогонах без раздачи личных бэкапов; быстрее входить в предсказуемое состояние данных для тестов.

**Риски и границы:** простой seed **неприемлем** как рекомендация обычным пользователям — это компромисс только для изолированных тестовых аккаунтов. Нужны явные правила: где хранится список таких учёток, как **ротировать** seed при утечке, что такие vault **не смешиваются** с реальными пользовательскими данными и не попадают в прод без отдельного решения.

Не входит в [[03-Scope-MVP-и-бэклог]].

---

## 10. Вход через сторонние сервисы — Яндекс, Google и др. (после MVP)

**Суть:** добавить к текущему сценарию **email + пароль** возможность войти через **федеративных провайдеров** — как минимум **Яндекс** и **Google**, в типичной связке с **Supabase Auth** (OAuth / social login).

**Зачем:** снизить трение при регистрации и входе; дать привычный способ тем, кто не хочет заводить ещё один пароль.

**Заметки для реализации:** включение провайдеров в проекте Supabase, редиректы и callback URL; кнопки на экране входа/регистрации; правило **связки** identity (один человек — несколько провайдеров + пароль); клиентское шифрование vault **остаётся** — seed по-прежнему вводит пользователь, OAuth только выдаёт сессию приложения. Нужны тексты согласия и политика данных при передаче минимального профиля от провайдера.

**Риски:** зависимость от доступности сервисов провайдера; дополнительная поверхность для поддержки («не могу войти через Google»). Не входит в [[03-Scope-MVP-и-бэклог]].

## 11. Кнопка «Обновить» после выхода новой версии (сброс кэша) — после MVP

**Суть:** после деплоя новой сборки пользователь может долго держать открытую вкладку и получать **устаревший JS/CSS** из кэша браузера или **старое поведение service worker**. Идея — при обнаружении **новой версии приложения** (semver или build-id из метаданных сборки) показывать ненавязчивый элемент UI — кнопку или тонкий баннер **«Обновить»** / «Доступна новая версия».

**По нажатию:** выполнить **перезагрузку с инвалидацией кэша** для статических ресурсов (hard reload / принудительное обновление SW), чтобы подтянулась актуальная оболочка приложения.

**Заметки для реализации:** откуда брать «эталон» версии (константа в бандле vs лёгкий запрос к серверу); что сохранять в `localStorage`; как не предлагать обновление посреди несохранённой работы с vault без явного подтверждения; для **PWA** — согласовать с политикой **service worker** (`skipWaiting`, перезапуск контроллера).

**Риски:** случайная перезагрузка в неудачный момент; раздражение от частых деплоев — нужны дебаунс и «напомнить позже». Не входит в [[03-Scope-MVP-и-бэклог]].

## 12. Достижения (после MVP)

**Суть:** добавить **достижения** — коллекцию значков или отметок за осмысленные успехи в использовании планировщика: например первая **серия дней** со стриком, закрытие задачи после долгого переноса, первый полный ритуал End-of-Day за неделю и т.д.

**Зачем:** мягкое позитивное подкрепление привычек — без обязательной «гонки» и без наказания за паузы.

**Заметки для реализации:** критерии разблокировки должны быть **прозрачны** или описаны в одном экране «Достижения»; опционально **отключить** раздел в настройках; хранение флагов — в vault или в профиле (не смешивать с чувствительными данными задач без необходимости). Связь с общей **геймификацией** и персонажем (§5 выше) — по приоритету продукта, чтобы не размножить конкурирующие системы наград.

Не входит в [[03-Scope-MVP-и-бэклог]].

---

## 13. Перенос незакрытой задачи при частично выполненном чек-листе (после MVP)

**Суть:** когда пользователь **переносит** на другой день (или в бэклог / новую дату) задачу, которая ещё **не закрыта целиком**, но в **чек-листе** уже есть **отмеченные пункты**, перенос должен **сохранять этот прогресс**: выполненные строки остаются **выполненными**.

**Дополнительно:** формулировки **уже выполненных** пунктов остаются **редактируемыми** в интерфейсе (например в модалке задачи) — чтобы можно было поправить текст без потери статуса «сделано».

**Заметки для реализации:** что считать «переносом» (смена даты плана, дубликат, отдельная команда); как не конфликтовать с **повторами** и **исключениями серии**; не сбрасывать чек-лист без явного действия пользователя.

Не входит в [[03-Scope-MVP-и-бэклог]].

---

## 14. Заведение дефектов из приложения (тестеры и админы) — после MVP

**Суть:** дать ролям **бета-тестер** и **администратор** возможность **отправить сообщение о дефекте** не выходя из планировщика: короткая форма (заголовок, описание, шаги воспроизведения), опционально **скриншот** или указание текущего экрана/маршрута.

**Зачем:** быстрее собирать обратную связь по сборкам; меньше трения, чем «напишите на почту»; единая точка входа для QA перед интеграцией с GitHub / Linear / другим трекером.

**Заметки для реализации:** тот же источник ролей, что и для раздела «Тестирование» (`motivator_role` в metadata); куда пишем на сервере — таблица Supabase + Edge Function, webhook в GitHub, или только email — по ресурсам команды; **конфиденциальность**: не тащить расшифрованный vault в тикет автоматически; версия клиента и локаль — добавлять автоматически.

Не входит в [[03-Scope-MVP-и-бэклог]].

## 15. Страница первичной настройки при первом входе (после MVP)

**Суть:** после **первого** успешного входа в приложение (и при необходимости — после разблокировки vault по seed) показать отдельный **экран первичной настройки**: не список задач сразу, а короткий мастер, который знакомит с ключевыми настройками и снимает «холодный старт».

**Примеры шагов (подбор приоритизации, не как единый обязательный пакет):**

| Шаг | Зачем |
|-----|--------|
| **Язык интерфейса** | Сразу снять неопределённость i18n; можно предложить смену позже в настройках. |
| **Названия уровней приоритета** | Пользователь видит осмысленные подписи вместо дефолтов vault. |
| **Ритуал End-of-Day** | Кратко объяснить и дать включить/выключить глобально (согласовать с текущими настройками). |
| **Мини-тур по вкладкам** | День / Неделя / Месяц / Отчёты — без длинного туториала; опционально «пропустить». |

**Заметки для реализации:** где хранить факт «мастер пройден» (`localStorage`, поле в профиле Supabase, флаг в vault); не блокировать сценарий «у меня уже есть seed» и повторный вход на новом устройстве; не дублировать обязательный онбординг seed из ТЗ — **дополнить** или **сцепить** по продуктовому решению.

**Риски:** лишнее трение перед первой задачей — нужны короткий путь и явный «Пропустить»; согласовать с будущими **feature discovery** (§1) и **входом через OAuth** (§10).

Не входит в [[03-Scope-MVP-и-бэклог]]; в модалке «Краткая сводка» — карточка в **`IDEAS_LATER_ENTRIES`** (`productRoadmap.ts`).
