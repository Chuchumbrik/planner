# Идеи для развития

> Накопительный список **направлений и гипотез** для продукта. Здесь нет обязательств в релиз; приоритизация — отдельно. Связь с зафиксированным объёмом: [[03-Scope-MVP-и-бэклог]], принятые решения: [[12-Журнал-решений]].

**Статус:** концепт-заметка, 2026-05-09

---

## 1. Напоминания о неиспользованных функциях (feature discovery)

**Суть:** мягко подсвечивать пользователю возможности приложения, **которыми он ещё не пользовался** — на случай, что он о них не знает или не дошёл до сценария.

**Возможные направления реализации (для проработки):**

- Учёт **событий** в клиенте (feature flags / «уже открывал раздел X», «уже переключал язык») с хранением в vault или в `localStorage` (и политикой приватности).
- **Контекстные подсказки:** баннер, тост, карточка в пустом состоянии, один раз после онбординга — без навязчивости.
- **Цель:** снизить «скрытую сложность» (DnD, режим Эйзенхауэра, фильтр групп, подзадачи и т.д.) без спама.

**Риски:** раздражение от подсказок — нужны частота, отключение в настройках и эвристики «показали один раз».

---

## 2. Отдельная область «Привычки»

**Суть:** на экране задач (или отдельном разделе) выделить **«Привычки»** — сущность отличная от разовых задач: **повторяющиеся действия** с собственным потоком создания и, возможно, собственной моделью в vault (расширение схемы отдельным инкрементом).

**Черновые вопросы при создании привычки** (как ориентир для UX и данных):

| Вопрос | Зачем |
|--------|--------|
| **Каким человеком ты хочешь стать в рамках этой привычки?** | Связь с мотивацией и идентичностью (контекст для напоминаний и отчётов). |
| **Как часто ты готов ею заниматься** (например, в днях / интервал повторения)? | База для календаря, стрика привычки, «ожидаемых» отметок без смешения с задачами-проектами. |
| **Сколько времени ты готов тратить на неё в день?** | Ограничение реализма, планирование блоков времени, возможные будущие интеграции с расписанием дня. |

**Дальнейшие темы для продукта:** отличие от обычной задачи с повтором, связь с **end-of-day** и отчётами, напоминания (push), визуализация серии выполнений.

## 3. Поиск по задачам (после MVP)

**Суть:** поле **поиска** на экране планировщика (или в шапке), фильтрация списка задач и/или подсветка совпадений по заголовку, при необходимости — по группе, тегам, тексту чек-листа (объём уточнять при реализации).

**Зачем:** быстрый доступ к задаче при большом бэклоге и длинном списке без прокрутки фильтров.

**Заметки для реализации:** клиентский поиск по уже загруженному vault; при росте данных — дебаунс, опционально индекс в памяти; не отправлять запросы поиска на сервер в явном виде (конфиденциальность).

---

## 4. Интеграция с внешними календарями (после MVP)

**Суть:** связать планировщик с **системными и облачными календарями** (Google Calendar, Microsoft 365 / Outlook, Apple/iCloud и др.), чтобы не дублировать контекст вручную и видеть **задачи и внешние события** в одном временном контуре.

**Возможные направления (на выбор приоритизации, не как единый пакет):**

| Направление | Зачем |
|-------------|--------|
| **Экспорт / «подписка»** на ленту планов (например **ICS** по защищённой ссылке или выгрузка файла) | Показ задач с временем в стороннем календаре без полного доступа аккаунта |
| **Импорт событий** (read-only слой: занятость, встречи) | Планирование задач с учётом уже занятых слотов; минимизация пересечений «вслепую» |
| **OAuth + официальные API** (Google, Microsoft) | Двусторонняя синхронизация или расширенный импорт — только после отдельной продуктовой и **privacy**-оценки |

**Ограничения продукта:** клиентское шифрование vault — любая синхронизация с облаком календаря потребует **явной модели**: что уходит наружу (только время/заголовок vs полный текст), куда пишутся изменения, как разруливаются конфликты. Для MVP это **не входит** в объём [[03-Scope-MVP-и-бэклог]]; пункт фиксируется как **идея на v1.1+**.

## 5. Персонаж-тамагочи и привязка к группам (после MVP)

**Суть:** лёгкая **геймификация без давления** — настраиваемый **персонаж** (тамагочи), чьи потребности **кормление, гигиена, настроение** и т.д. отражают или усиливают мотивацию. Для **каждой группы задач** можно задать, **какое действие персонажа** «получает» выполнение задач этой группы (например: рабочие задачи → «опыт», домашние → «уют»).

**Черновые направления:** отдельный экран/виджет персонажа, редкие ненавязчивые напоминания, опциональное отключение в настройках. Сохранение состояния персонажа — в vault или локально; синхронизация между устройствами — отдельное решение.

**Границы:** не превращать планировщик в игру; не наказывать за пропуски; для MVP **не входит** в [[03-Scope-MVP-и-бэклог]].

## 6. MCP для AI (после MVP)

**Суть:** после стабилизации ядра продукт можно связать с экосистемой **AI-ассистентов** через **Model Context Protocol (MCP)** — стандартный способ отдавать модели **структурированный контекст** и **безопасные действия** (tools), не смешивая это с обходом клиентского шифрования без явного согласия пользователя.

**Зачем пользователю:** голосом или в чате IDE («разложи бэклог на неделю», «что на сегодня», «подготовь формулировки задач из заметки») — ассистент опирается на **реальные** задачи и календарный контекст приложения, а не на угадывание.

**Черновые направления проработки (не пакет обязательств):**

| Направление | Заметки |
|-------------|---------|
| **MCP-сервер как локальный/доверенный мост** | Чтение расшифрованного vault **только там, где уже есть ключ** (например десктоп или доверенная среда); сервер не хранит master-key на облаке. |
| **Набор tools** | Только то, что можно формализовать: список задач за день, черновики, группы; опционально «предложить черновик задачи» без автозаписи в vault до подтверждения в UI. |
| **Privacy и границы** | Явное включение в настройках; аудит того, что уходит в LLM; запрет полной выгрузки vault наружу по умолчанию. |
| **Интеграции** | Cursor / Claude / другие клиенты с MCP; возможно отдельный лёгкий адаптер поверх существующего `@motivator/core`. |

**Дополнение — ИИ через HTTP API (BYOK):** помимо MCP возможен сценарий **прямого** вызова моделей по API (пользовательский ключ, выбор провайдера / OpenAI-совместимый endpoint) для подсказок внутри UI. Те же ограничения: никакой автоматической отправки всего vault; только явно разрешённый пользователем контекст; отдельная продуктовая и privacy-оценка до реализации. Может сосуществовать с MCP как другой слой доступа к ИИ.

**Границы:** не входит в MVP [[03-Scope-MVP-и-бэклог]]; до реализации нужны отдельные **DR** по безопасности и UX подтверждений.

## 7. Режимы просмотра задач на «День» (после MVP)

**Суть:** сейчас вкладка «День» — преимущественно **список** (план + бэклог). На будущее — дать пользователю **переключаемые представления** одних и тех же задач на выбранную дату, без дублирования данных в vault.

**Примеры режимов (гипотезы):**

| Режим | Зачем |
|--------|--------|
| **Список** (как сейчас) | Быстрый обзор, сортировки, бэклог рядом |
| **По часам** | Вертикальная **шкала дня** (аналогично колонке недели в таймблокинге): задачи с временем начала/конца — в слотах; без времени — отдельная полоса «без времени» или верх/низ экрана |
| **Компактный** | Уменьшенные карточки / одна колонка для просмотра на маленьком экране |

**Заметки для реализации:** общая фильтрация по `taskOccursOnDate` / слотам уже есть в `@motivator/core`; UI — переключатель вида + сохранение предпочтения в `localStorage` или настройках; повторы и пересечения — те же правила, что на неделе.

**Границы:** не входит в [[03-Scope-MVP-и-бэклог]]; приоритет после стабилизации MVP и базового календаря.
