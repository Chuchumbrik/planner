# Модель данных (концептуально, MVP)

Цель раздела — дать разработчикам общий язык до проектирования конкретной SQL-схемы в Supabase. Физическая схема может хранить **зашифрованные полезные нагрузки** как единый blob на запись.

## Основные сущности

### Пользователь / аккаунт

- Идентификатор в Auth (Supabase).
- Настройки профиля: язык, система приоритетов, глобальные флаги ритуала, время ритуала, настройки уведомлений.
- План (free/premium) для [[05-Freemium]].

### Задача

- Идентификатор, ссылка на пользователя.
- Заголовок, описание (в ciphertext при полном E2E).
- Подзадачи: либо вложенный список внутри ciphertext, либо отдельные строки с `parent_id` (решение влияет на запросы и синк).
- Цвет, группа/проект, приоритет (в выбранной системе).
- Флаги: двойное подтверждение, интервал минут, участие в end-of-day.
- Статусы и история выполнения за дни (для отчётов и стрика).

### Группа / проект

- Имя, цвет по умолчанию (опционально), порядок сортировки.

### Ритуал (шаблоны и ответы)

- Пул КПТ-вопросов (базовые 10 + премиум-расширение позже).
- Пользовательские выборы: какие 3 вопроса активны в ротации.
- Записи ответов по датам (привязка к дню и опционально к задаче).

### Уведомления

- Локальные подписки web push + серверные идентификаторы подписок (если нужны для Edge).

### Синхронизация и конфликты

- Версии сущностей или vector clocks — уточняется при выборе offline-библиотеки.

## Метаданные vs шифрование

**Решение для MVP:** вся аналитика и отчёты считаются **только на клиенте** после расшифровки; сервер — **только ciphertext** ([[12-Журнал-решений#DR-007 — Отчёты где считать аналитику]]). Любое изменение этой модели — новая запись в [[12-Журнал-решений]].
