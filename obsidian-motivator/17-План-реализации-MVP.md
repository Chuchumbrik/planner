# План реализации MVP (продукт v1.0.0)

> **Источники требований:** [[16-TZ-MVP-v1.0]], [[03-Scope-MVP-и-бэклог]].  
> **Журнал решений:** [[12-Журнал-решений]] (в т.ч. **DR-002**, **DR-004**, **DR-007**, **DR-008**, **DR-009**, **DR-010**, **DR-012**, **DR-013** и связанные записи).

**Текущее состояние кода:** веб-клиент `web/` — актуальный контракт возможностей и версия схемы vault см. в **`web/README.md`** и пакете `@motivator/core`. Ниже по фазам перечислены **все** элементы MVP из ТЗ и Scope; там, где функциональность уже реализована, это не исключает регрессионной проверки и доводки до критериев ТЗ перед релизом **1.0.0**.

**Принцип порядка работ:** сначала **инварианты данных и ядро задачи**, затем **День → Неделя → Месяц**, затем **повторы**, затем **отчёты на клиенте** (включая стрик и «часто проваленные»), затем **ритуалы** (двойное подтверждение, End-of-Day, анимации), затем **аккаунт и юридика** (включая экспорт seed / онбординг), затем **монетизация и уведомления** (если остаются в составе релиза), затем **offline-first** как усиление архитектуры, затем **формальный релиз 1.0.0**.

Ничего из обязательного Scope [[03-Scope-MVP-и-бэклог]] и §2–4 [[16-TZ-MVP-v1.0]] не выкидывается: элементы либо попадают в одну из фаз ниже, либо — в блок **«Непрерывно»** (сквозные активности).

## Фаза 0 — Фундамент: модель данных, шифрование, платформа

- Зафиксировать целевую **версию схемы vault** и перечень сущностей: план на дату / backlog, слоты времени, оценка длительности, повторы, шкала приоритетов **1–5** в настройках, чек-лист, группы, **черновики формы создания задачи** (если входят в утверждённую схему).
- **Миграции** между версиями схемы на клиенте (цепочка v1→…→актуальная; таблица соответствия полей для пользователей со старыми данными — по необходимости).
- **Клиентское шифрование** vault; политика ключа / seed — [[DR-005]], [[DR-006]]; паритет с контрактом `packages/motivator-core/docs/VAULT_CRYPTO_CONTRACT.md`.
- **Web + PWA:** сборка, manifest, service worker — по политике проекта ([[14-V1-минимальный-запуск-Vercel]], `web/README.md`).
- Каркас **локализации** (i18n; минимум **ru** и **en** согласно Scope).

## Фаза 1 — Ядро задачи и экран «День»

- Сущности **задача**, **группа**, **цвет** (в т.ч. **палитра** выбора цвета).
- **Разделение просмотра и редактирования:** мини-карточка vs полная модалка редактирования ([[16-TZ-MVP-v1.0]] §4).
- **Чек-лист** («план работы над задачей»): порядок пунктов, отображение на мини-карточке по правилам продукта.
- Флаг **backlog** vs **план на календарную дату**; локальная **дата устройства** для «Дня».
- Экран **День:** список задач и мини-карточки; секция **Backlog внутри Today** по ТЗ.
- **Приоритеты 1–5** с настраиваемыми названиями (экран настроек + использование в задаче).
- **Оценка времени** (в минутах / представление часы+минуты — как зафиксировано в реализации и ТЗ).
- **Время начала XOR время завершения** (взаимоисключение) или отсутствие времени суток.
- Базовое **создание задачи** (модалка), при необходимости — **черновики** в vault при закрытии без сохранения и UX при **нескольких черновиках** (кнопка со счётчиком, список в модалке).

---

## Фаза 2 — Неделя (таймблокинг)

- Вид **«Неделя»:** сетка дни × часы; задачи в **реальных временных слотах** по правилам ТЗ (начало + оценка → конец / правило для времени завершения — по выбранному режиму).
- При **завершении** задачи — **освобождение слота** по продуктовым правилам.
- **Пересечения** интервалов с другими задачами: проверка при планировании + **мягкое предупреждение** с возможностью продолжить (формулировка по ТЗ).

---

## Фаза 3 — Месяц, фильтры, информер

- Вид **«Месяц»:** календарная сетка с **отметками** по задачам / плотности — как определено в ТЗ и текущей реализации.
- **Фильтры** текущего представления: группы, приоритеты, повторы (и иные поля из ТЗ для месяца).
- **Иформер:** сводка активных фильтров / охвата ([[16-TZ-MVP-v1.0]] §3).

---

## Фаза 4 — Повторяющиеся задачи

- Типы повторов из ТЗ: **каждый день**; **каждые N дней**; **по дням недели**.
- Поля серии: якорь, исключения / отметки выполнения по вхождениям — по утверждённой модели в vault и журналу (включая преемственность с отчётами).
- Идентификатор серии / шаблона для **«часто проваленных»** и отчётности — [[DR-008]].

---

## Фаза 5 — Отчёты и аналитика (клиент)

- Отчёты считаются **на клиенте** ([[DR-007]]): доли, диаграммы по периодам, прочие метрики из ТЗ.
- **Стрик** — по согласованному правилу [[DR-013]] (связка с EOD и задачами плана дня).
- Таблица / блок **часто проваленных** задач — [[DR-008]].
- При необходимости: **производительность** тяжёлых расчётов (worker, батчи, отложенные пересчёты) — без нарушения отзывчивости UI.

**Состояние веб-реализации (первая итерация):** экран **`/app/reports`**, агрегаты в пакете **`@motivator/core`** (`lib/reporting/vaultAnalytics.ts`): окно 7/30 дней, KPI и столбиковый график отметок, таблицы пропусков по сериям и разовых с просрочкой; **полный стрик по DR-013** (с учётом End-of-Day) — после **фазы 6**. См. `web/README.md`.

---

## Фаза 6 — Двойное подтверждение, End-of-Day, анимации

- **Двойное подтверждение** по правилам задачи — [[DR-004]].
- **End-of-Day:** глобальный вкл/выкл; состав задач ритуала с учётом полей «план на день» и календаря — [[DR-002]]; при необходимости уточнение КПТ-банка под финальную модель.
- **Анимации:** минимум **две пары** тона (успех / мягко при «не сделал») — [[16-TZ-MVP-v1.0]] §2–3 и Scope.

---

## Фаза 7 — Настройки, аккаунт, seed, юридика

- Настройки **структурированы по группам** разделов (как в ТЗ §4): приоритеты, группы задач, язык, пароль и т.д.
- **Смена пароля входа** (email/password Supabase): текущий, новый, подтверждение на экране настроек.
- **Экспорт seed / онбординг:** отдельный экран или улучшенный поток «скопировать seed» — обязательное улучшение MVP ([[16-TZ-MVP-v1.0]] §4).
- **Удаление аккаунта** с возможностью **восстановления в течение 30 дней по ключу**; серверная политика и UX — по [[DR-009]] и технической спецификации (таблицы Supabase, Edge Functions или аналог).
- Юридические и учётные тексты — согласование с [[DR-006]].

## Фаза 8 — Монетизация и уведомления (если входят в состав релиза 1.0.0)

- **Freemium** — реализация по [[05-Freemium]] и [[DR-010]], после стабилизации ядра планирования и отчётов (чтобы не блокировать MVP-трек).
- **Настраиваемые PWA push:** в [[16-TZ-MVP-v1.0]] не детализированы; если закреплены для данного релиза в [[03-Scope-MVP-и-бэклог]] и решении спринта — планировать после базовых напоминаний/стабильности задач.

## Непрерывно на всём цикле (не отдельная «фаза», а обязанность)

- **Локализация:** поддержание ключей и качества **ru/en** (и расширение локалей по решению).
- **Безопасность данных:** аудит изменений, затрагивающих шифрование и хранение vault.
- **Документация для разработчиков:** `web/README.md`, при необходимости — заметки в `obsidian-motivator/` после каждой крупной завершённой фазы.
- Регрессия критических путей при каждом значимом merge.
