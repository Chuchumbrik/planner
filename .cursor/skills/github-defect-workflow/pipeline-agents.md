# Пайплайн дефекта — субагенты и шаблоны

Полные правила фаз A–D, документация и закрытие issue — в [SKILL.md](SKILL.md). Справка по комментариям и `gh` — в [reference.md](reference.md).

Slash-команда: `.cursor/commands/defect-squad-pipeline.md`. **Новые фичи** (планирование + реализация): **`/feature-delivery-pipeline`** — `.cursor/skills/feature-delivery-workflow/`.

**Права GitHub/git:** для этого пайплайна пользователь **предоставил права** (`gh`, MCP GitHub при наличии, issue/PR, push по сценарию). Оркестратор и субагенты **используют** их без избыточных отговорок, пока среда не вернула ошибку — см. [SKILL.md](SKILL.md) и [reference.md](reference.md).

## Контекст дефекта (шаблон для оркестратора)

Один и тот же блок передаётся в субагентов (после строки с этапом):

- **Issue:** `#N` и/или полный URL.
- **Суть:** что не так или что предлагают (1–3 предложения).
- **Шаги / данные:** воспроизведение или пометка «информативный / дорожная карта».
- **Окружение:** при необходимости `appVersion`, маршрут, локаль, роль.
- **Ограничения:** «не пушить», «только анализ», сухой прогон, лимит раундов QA и т.п.
- **Права GitHub/git:** по умолчанию выданы пользователем — агент использует `gh`/MCP/push по сценарию до фактической ошибки среды (см. [SKILL.md](SKILL.md), [reference.md](reference.md)).

## 3. Агрегатор / приниматель решений

Субагент **`defect-decision-aggregator`** сводит отчёты `## [SYSTEM] …` и `## [BA] …` в один блок **`[DECISION]`** по шаблону ниже. При конфликте ролей — явно указать, что принято и почему.

```markdown
## [DECISION] Дефект #N

### Итоговый выбор
`IMPLEMENT_NOW` | `DEFER_TO_PHASE` | `POST_MVP_IDEA` | `DOCS_AND_ROADMAP_ONLY` | `CLOSE_AS_DUPLICATE_OR_INVALID` | `BLOCKED_NEED_HUMAN` | (другое — по смыслу)

### Краткое обоснование (для таблицы «Почему такое решение»)
2–4 предложения: что сказали SYSTEM и BA, где сошлись / разошлись, почему выбран именно этот исход (риск, фаза плана, срочность, отсутствие бага в коде).

### Следующие шаги для оркестратора
- при `IMPLEMENT_NOW` + боевой push: **синхронизация git** (`fetch`, актуальный `main`) → ветка от `main` → доки до кода (BA → SYSTEM) → UX → FE → BE → QA → PR в `main` с `Fixes #N` в теле PR;
- при отложенном / только доках: что править в README / `productRoadmap` / локалях; нужен ли QA по докам.

### Риски и допущения
… или «нет существенных».
```

## Итоговое сообщение оркестратора (обязательно на любом завершении)

**Правило:** как только сценарий по дефекту **остановлен** (после `[DECISION]`, после лимита QA, после полного пайплайна, «только анализ» и т.д.), **ведущий агент в том же ответе пользователю** выдаёт:

1. **«Суть дефекта (кратко)»** — 2–5 предложений простым языком.
2. **Таблицу** — все строки ниже заполнены; неприменимое — **`—`** или короткая пометка.
3. **Ручная настройка** — только если после реализации (или merge) человеку **нужно** что-то сделать вне репозитория (деплой Edge, секреты, SQL в Dashboard, Vercel, cron, роли Auth и т.п.): **отдельным блоком** под таблицей — см. подраздел **«Шаблон: ручная настройка для человека (чеклист)»** ниже на этой странице. Если ручных шагов **нет** — в таблице явно **«не требуется»** и **третий блок не выводить**.

Шаблон таблицы (Markdown):

```markdown
## Итог по дефекту #N

### Суть дефекта (кратко)
…

| Поле | Значение |
|------|----------|
| **Решение** | значение из `[DECISION]` + при необходимости уточнение в скобках |
| **Почему такое решение** | 1–4 предложения: на чём основан выбор (выжимка SYSTEM/BA, ограничения из контекста, план фаз); при альтернативах — почему они отвергнуты или отложены |
| **Статус кода** | реализовано / частично / не реализовано + пояснение при необходимости |
| **Что сделано** | факт: разбор, коммиты, файлы, ссылки на PR — кратко |
| **План, если не код** | фаза / post-MVP / **—** если ветка с немедленным кодом |
| **Документация / сводка** | обновлено: да/нет — где; или «не обновлялась (режим …)» |
| **GitHub issue** | открыт / закрыт — полный URL issue |
| **Ветка / PR** | при боевом push: `fix/issue-…` + URL PR; иначе **—** |
| **QA** | `PASS` / `FAIL` / не запускался — число раундов или причина |
| **Ручная настройка** | **не требуется** / **требуется** — см. блок «Ручная настройка» ниже (кратко одна фраза, если требуется) |
```

Поле **«Почему такое решение»** не дублирует дословно весь `[DECISION]`, а даёт **читаемую причину** для человека вне чата: почему команда идёт этим путём.

**Пример строки «Почему такое решение»:** «SYSTEM указал отсутствие обработки paste в зоне вложений; BA подтвердил приоритет UX для отчётов об ошибках. Срочный point-fix в текущем компоненте дешевле, чем ждать фазу вложений целиком; откладывание оставили на отдельную задачу после MVP.»

### Шаблон: ручная настройка для человека (чеклист)

Использовать **только** если в строке таблицы **«Ручная настройка»** не «не требуется». Писать **простым языком**, без жаргона там, где можно заменить на действие «нажми / вставь / выбери». Пункты — в виде **чекбоксов Markdown** (`[ ]` не сделано; человек меняет на `[x]` у себя в копии или в issue при желании):

```markdown
### Ручная настройка после выката

**Зачем это нужно:** одно-два предложения связи с дефектом.

**Где подробности в репо (если есть):** ссылка на раздел `web/README.md` или другой согласованный документ — **не** подменять README длинным дублированием, если там уже полная процедура; тогда здесь — **краткий** чеклист «что отметить по дороге».

**Чеклист (отметь выполненное):**

- [ ] Шаг 1 — конкретное действие (куда зайти, что вставить).
- [ ] Шаг 2 — …
- [ ] Проверка: как убедиться, что настройка сработала (1–3 шага).

**Если что-то нельзя сделать:** что откладывается и какой симптом ожидать.
```

FE/BE при внедрении нового ручного сопровождения по правилам репозитория должны отразить процедуру в **`web/README.md`** (или явно указанном документе) в той же итерации — см. **`.cursor/rules/documentation-orientation.mdc`**; в ответе пользователю всё равно дать **этот короткий чеклист** для немедленной отметки прогресса.

## Ветка и PR от `main` (боевой цикл с push)

**Когда:** разрешён полный цикл с **push** в remote, нет ограничения «не пушить».

**Кратко:**

1. **Перед созданием ветки фикса — синхронизация с Git (обязательно):** не создавать `fix/issue-…` от устаревшего или «грязного» состояния.
   - Убедиться, что **нет неожиданных локальных правок** на текущей ветке: `git status`. Если есть несохранённые важные изменения — **commit** в соответствующую ветку или **`git stash push -m "wip before defect branch"`** (потом не забыть `stash pop` на своей ветке). Если мешают незакоммиченные файлы, не относящиеся к дефекту — не смешивать их с веткой фикса.
   - Подтянуть удалённое: **`git fetch origin`**.
   - Перейти на целевую базу и выровнять её с remote: **`git checkout main`** (или согласованная default-ветка) → **`git pull origin main`** (или **`git merge origin/main`**, если так принято в команде). Цель: **`main` локально = `origin/main`** на момент создания ветки.
   - **Затем** создать ветку: **`git checkout -b fix/issue-<N>-<slug>`** — все коммиты по дефекту только в ней. Этот порядок — **до** этапа «доки до кода» (`ДОКИ_ДО_КОДА`), если он используется.
2. После **`[QA]` → `PASS`** — `git push` и **PR в `main`**.
3. В **теле PR** обязательна строка с [ключевым словом GitHub](https://docs.github.com/en/issues/tracking-your-work-with-issues/linking-a-pull-request-to-an-issue): **`Fixes #<N>`** (или `Closes` / `Resolves`) — иначе issue не закроется автоматически при merge. Несколько issue — несколько номеров в теле PR.
4. **Фаза D:** комментарий в issue с URL PR.

Без push — в строке таблицы **«Ветка / PR»** указать **`—`**.

## Каталог субагентов (файлы в `.cursor/agents/`)

| Файл / `name` | Когда вызывать |
|---------------|----------------|
| `defect-system-analyst` | Технический разбор или техдоки до кода |
| `defect-ba-po` | Продукт / фазы / сводка или продуктовые доки до кода |
| `defect-decision-aggregator` | После параллельного SYSTEM + BA |
| `defect-ux-designer` | После `[DECISION]` с реализацией, если нужен UX-слой |

Если в UI настроек **Subagents** не видно всех файлов сразу: **Developer: Reload Window** в палитре команд или переоткройте папку проекта — Cursor перечитывает `.cursor/agents/*.md`.

## 1. Выход SYSTEM — `ЭТАП: РАЗБОР` (только отчёт)

Заголовок ответа: **`## [SYSTEM] Анализ дефекта #N`**. Блоки (по смыслу, пустые — «н/п»):

- **Факты из issue** — что подтверждено / что не хватает данных.
- **Гипотеза и зона кода** — файлы, поток данных, Edge vs web.
- **Инварианты** — auth, vault, приватность; не ослаблять.
- **Риски срочного фикса** vs отложенная фаза.
- **Открытые вопросы** — списком для человека.

**Запрещено:** правки файлов репозитория.

---

## 2. Выход BA/PO — `ЭТАП: РАЗБОР` (только отчёт)

Заголовок: **`## [BA] Анализ дефекта #N`**.

- **Суть для пользователя** и приёмочные критерии простым языком.
- **Продукт / MVP / фазы** — баг vs ожидание «не в scope» vs информативный issue.
- **Документация и сводка** — что должно измениться в README / `productRoadmap` / локалях (идея, без обязательных правок в РАЗБОР).
- **Рекомендация по решению** (не финальная — финал у агрегатора): ship / defer / docs-only / duplicate.

**Запрещено:** правки файлов в РАЗБОР.

---

## 3b. Выход UX — после `[DECISION]` (обычно без коммитов кода)

Заголовок: **`## [UX] Дефект #N`**.

- **Проблема UX** в 1–2 фразах.
- **Рекомендации** — layout, копирайт (ru/en), состояния, a11y.
- **Критерии приёмки** — нумерованный список для FE.
- **Не в scope UX** — явно написать, если шаг можно пропустить.

По умолчанию **не** менять `.tsx` самостоятельно — отдать в FE.

---

## 4. Выход Frontend

Заголовок: **`## [FE] Дефект #N`**.

- **Что сделано** — файлы, поведение.
- **Тесты** — что прогнано (`vitest` / ручная проверка).
- **i18n** — затронуты ли `ru`/`en`.
- **Ручная настройка** — если после merge человеку нужны шаги вне git (деплой, секреты, дашборд и т.д.): краткий список для оркестратора / итогового чеклиста; при новой процедуре — обновить **`web/README.md`** по правилам репозитория.

Зона: `web/`, клиентский TypeScript; **не** Edge handlers и **не** секреты сервера.

---

## 5. Выход Backend

Заголовок: **`## [BE] Дефект #N`**.

- **Что сделано** — Edge functions, SQL/migrations, RLS, env-зависимости.
- **Как проверить** — curl/логи/Supabase dashboard по шагам.
- **Ручная настройка** — миграции в Dashboard, deploy функций, секреты, cron и т.п.: перечислить для итогового чеклиста; при новой процедуре — **`web/README.md`** по правилам репозитория.

---

## 6. Выход QA

Заголовок: **`## [QA] Вердикт дефекта #N`**.

- **Раунд** — `k` из `M` если известно.
- **Покрытие** — white-box, `eslint`/`tsc`/`vitest`, доки (README, roadmap, локали) если менялись.
- **Вердикт:** `PASS` или `FAIL`.
- При **FAIL** — кому вернуть (`defect-frontend-dev` / `defect-backend-dev` / `defect-decision-aggregator` для только доков).
- **Ручная настройка** — если по фиксу человеку всё ещё нужны шаги вне git: в вердикте явно «требуется чеклист» + перечень пунктов для итогового сообщения оркестратора; если всё в коде/доках и деплой не меняет правил — **«ручная настройка: не требуется»**.

Оркестратор повторяет FE/BE и снова QA до **PASS** или лимита раундов (по умолчанию **3**).
