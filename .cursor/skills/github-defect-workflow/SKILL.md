# GitHub Issues: проработка дефектов (planner)

## Права GitHub и git для агента

По умолчанию для работы по этому скиллу и slash-команде **`/defect-squad-pipeline`**: пользователь **предоставил необходимые права** на операции с GitHub и репозиторием (**`gh`**, MCP GitHub, если сервер подключён, комментарии и закрытие issue, PR, **`git push`** — когда сценарий это разрешает). Агент **должен использовать** эти возможности и **не** пропускать шаги из‑за гипотетического «нет доступа», пока **реальная** попытка в среде не завершилась ошибкой. Тогда — запасной путь по [reference.md](reference.md).

## Когда пользоваться этим скиллом

**Включать и следовать этому скиллу**, когда:

- работа ведётся с **дефектами из GitHub**: конкретные **Issues** (номер, ссылка, список issue), в том числе созданные через **`file-defect`** / настройки **«Завести дефект»**, закрытие issue после фикса, комментарии и метки в репозитории GitHub, прогон по списку дефектов из GitHub;
- **или** в чате есть **явное** указание, что задача про **дефект, баг, регрессию, поломку** и т.п. (предмет работы сформулирован как ошибка продукта), даже без номера issue — те же фазы анализа и фикса; шаги, завязанные **только** на GitHub (комментарий в issue, **фаза D**), выполнять, когда issue есть или пользователь просит завести/привязать.

**Не считать обязательным** для размытых вопросов в чате **без** привязки к GitHub **и без** явной дефектной формулировки («что-то странное», общий рефакторинг и т.д.); тогда достаточно общих правил репозитория и при необходимости **`pre-commit-docs-roadmap`**.

**GitHub — только открытые:** в разбор и в общий прогон по списку попадают **только** issue в состоянии **`open`**. **Закрытые** не брать как текущие дефекты, если пользователь **явно** не просит повторный разбор, аудит причины закрытия, регрессию после закрытия или переоткрытие.

### Жёсткое правило: закрыть issue в GitHub после фикса

Если дефект **привязан к issue** (номер, ссылка, контекст «закрывает #N») и исправление **уже доставлено** (push в remote, merge в целевую ветку по договорённости команды) и выполнен критерий «готово» из issue — **обязательно** выполнить **фазу D** в **той же итерации работы**, пока нет пункта из списка **«Когда не закрывать issue сразу»** в [reference.md](reference.md).

**Оставить issue в состоянии `open` после доставленного фикса** без комментария с причиной и без ссылки на PR/коммит — **нарушение процесса**: пользователь и команда не видят в GitHub, что дефект снят. **Не считать проработку дефекта завершённой**, пока на GitHub не сделано одно из: **Close issue** (с комментарием), или **осознанное** оставление открытым с комментарием по исключениям из reference.

## Базовые правила (формулировка заказчика — дословно)

1. Обязательно, если формулировки не понятны уточняем информацию
2. Проводим глубокий анализ как локально кода по дефекту, так и архитектурно
3. Если вопросы простые, задавать их в чат в полном объеме, понятно на человеческом языке
4. Если все понятно, выполнить обновление документации и исправление дефекта
5. После исправления **закрываем дефект в GitHub** (issue), если фикс доставлен и нет согласованного исключения — см. [жёсткое правило](#жёсткое-правило-закрыть-issue-в-github-после-фикса) и [reference.md](reference.md).

И так по всему списку.

## Как интерпретировать базовые правила в работе

| Пункт | Практика |
|-------|----------|
| **1** | Не угадывать: если шаги воспроизведения, ожидание/факт, версия, роль, маршрут или контекст из issue не сходятся с кодом — **остановиться** и уточнить у человека (или в issue комментарием, если пользователь просит вести диалог там). Зафиксировать допущения в ответе/коммите. |
| **2** | **Локально:** точки входа (компоненты, Edge, `@motivator/core`), данные, состояния гонок, границы доверия (JWT, vault). **Архитектурно:** связь с `obsidian-motivator/` (журнал решений, ТЗ, архитектура), границы клиент/сервер, последствия для приватности и миграций. **Документация и фазы:** см. блоки [«план против срочного фикса»](#дефект-документация-и-будущие-фазы-план-против-срочного-фикса) и [«информативный»](#дефект-информативный-порядок-фаз-mvp-идеи) — не ограничиваться только кодом. |
| **3** | Один раз сформулировать **все** простые вопросы списком, без жаргона, с вариантами ответа где уместно; не дробить на десять коротких сообщений подряд без необходимости. |
| **4** | Сначала **понимание и план**, затем правки кода; документация — по [чеклисту документирования](#документация-и-коммит) в том же коммите/итерации, что и фикс. Если корневая причина относится к **ещё не наступившей фазе** плана — см. блок про **отложенный фикс** (обновление плана вместо или рядом с немедленным исправлением). Если issue **информативный** — «исправление» часто **только документы и данные сводки** (без кода приложения). |
| **5** | **Issue на GitHub закрыть**, когда фикс **доставлен** (push / merge по правилам команды) и критерий «готово» выполнен — с **комментарием** и ссылкой на PR или коммит; см. [фазу D](#фаза-d-закрытие-issue-пункт-5) и [reference.md](reference.md). Исключения («не закрывать сразу») — **только** с явным комментарием в issue. **Работу по дефекту не считать завершённой**, пока issue остаётся **`open`** без такого комментария. Либо явное решение «закрыть как неактуально / дубликат» без кода. |
| **«По всему списку»** | Вести **явный прогресс**: какой issue сейчас, что сделано, что следующий; не смешивать несколько несвязанных дефектов в одном коммите без согласования. При выборке из GitHub — **только открытые** issue в очереди. |

## Фаза A: приёмка issue (до кода)

- [ ] **Состояние:** issue **открыт** (`open`). Если **закрыт** — не продолжать как активный дефект по этому скиллу без **явного** запроса пользователя (повторный разбор, регрессия после закрытия, оспорить закрытие, переоткрыть); при обходе списка — пропускать закрытые.
- [ ] **Метаданные:** номер, заголовок, метки; есть ли дубликаты или связанные PR.
- [ ] **Воспроизведение:** шаги из тела issue + при необходимости уточнение окружения (браузер, локаль/prod, версия из **`appVersion`** в issue). Для **информативных** issue воспроизведение может быть неуместно — тогда зафиксировать **суть предложения** и ожидаемый результат (какие документы или блоки сводки должны измениться).
- [ ] **Ожидаемое / фактическое** — явно; критерий «готово» сформулировать одной фразой.
- [ ] **Границы:** это баг, регрессия, недоделанный scope, продуктовое решение или **информативное** замечание (см. [ниже](#дефект-информативный-порядок-фаз-mvp-идеи))? При конфликте с ТЗ/журналом — не «чинить молча»: отметить в чате и согласовать направление.

## Фаза B: анализ (глубина по пункту 2)

Если в [фазе A](#фаза-a-приёмка-issue-до-кода) issue уже **информативный**, нижний чеклист по коду выполнять **только** при подозрении на реальный дефект; иначе опираться на подразделы **документации, фаз и информативных** замечаний.

- [ ] Найти **реальные** участки кода (поиск по символам, маршрут из issue, стек из описания).
- [ ] Проследить **поток данных** (props, vault, Supabase, Edge) до конца сценария.
- [ ] Оценить **побочные эффекты:** i18n (`ru`/`en`), a11y, мобильная вёрстка, PWA/SW если касается.
- [ ] Для крипто/vault/ролей — не ослаблять инварианты; сверка с `packages/motivator-core` и заметками в `obsidian-motivator/06-*.md`, `08-*.md` при сомнении.

### Дефект: документация и будущие фазы (план против срочного фикса)

Часть отчётов выглядят как «баг», но при разборе упираются в **задел под следующие фазы** (офлайн, отчёты, push, роли и т.д. по [`obsidian-motivator/17-План-реализации-MVP.md`](../../../obsidian-motivator/17-План-реализации-MVP.md)) или в **намеренно отложенный** scope. Тогда анализ на уровне документации обязателен:

- [ ] Сверить формулировку issue с **ТЗ и планом фаз** (`obsidian-motivator/16-TZ-MVP-v1.0.md`, `17-План-реализации-MVP.md`, при необходимости журнал решений `12-Журнал-решений.md`).
- [ ] Сверить с **данными «Краткой сводки»**: `MVP_PHASES_PLANNED`, `IMPLEMENTED_MVP_PHASES`, при отложенном улучшении — `IDEAS_LATER_ENTRIES` и `obsidian-motivator/15-Идеи-для-развития.md` (правила переноса — в скилле **`pre-commit-docs-roadmap`**).
- [ ] Оценить **стоимость** немедленного костыля vs **честного** решения в рамках нужной фазы (долг техдолга, риск регрессий, приватность).
- [ ] Если **правильнее внести в план**, а не чинить сразу: явно сказать пользователю и зафиксировать в **комментарии к issue** (ссылка на фазу/документ, почему не point-fix сейчас). Обновить дорожную карту/README при согласовании (часто — уточнение формулировки фазы или открытого вопроса, без закрытия issue как «исправлено», пока не сделана работа — либо метка «отложено / план», либо дочерняя задача на фазу).
- [ ] Если **всё же срочный фикс** нужен (безопасность, блокер для пользователей, явное требование) — зафиксировать в комментарии, **почему** не откладываем, и не противоречить принятым DR в журнале.

### Дефект информативный (порядок фаз, MVP, идеи)

Часть issue **не про поломку**, а про **смысл плана**: обратная связь по дорожной карте, порядку работ или границам MVP.

Типичные запросы:

- **Поменять местами фазы** или иначе перестроить ожидаемый порядок внедрения — сверка с [`obsidian-motivator/17-План-реализации-MVP.md`](../../../obsidian-motivator/17-План-реализации-MVP.md); после согласования — согласованные правки в **`MVP_PHASES_PLANNED`** / при необходимости **`IMPLEMENTED_MVP_PHASES`** в `web/src/data/productRoadmap.ts` (и в Obsidian по правилам команды), без выдумывания «фактически сделано» в реализованных фазах.
- **Добавить идею после MVP** — запись в **`IDEAS_LATER_ENTRIES`** и выравнивание с [`obsidian-motivator/15-Идеи-для-развития.md`](../../../obsidian-motivator/15-Идеи-для-развития.md) (инварианты — в скилле **`pre-commit-docs-roadmap`**).
- **Встроить в MVP** (расширить обязательный scope до 1.0.0) — обязательно сверка с **`16-TZ-MVP-v1.0`**, **`03-Scope-MVP-и-бэклог`**, журналом **`12-Журнал-решений`**; при согласовании — обновить план фаз и при необходимости ТЗ/открытые вопросы (локали, `settings.roadmapOpenQuestion*`), не противореча зафиксированным DR без явного решения.

Чеклист:

- [ ] Классифицировать issue как **информативный** уже в [фазе A](#фаза-a-приёмка-issue-до-кода); не искать «баг в коде», пока не отвергнута эта гипотеза.
- [ ] **Глубокий анализ** здесь — продуктовая и документальная трассировка (последствия для версий, сводки, ожиданий пользователей); код приложения трогать только если меняется описанное в UI поведение или обнаружен реальный дефект по пути.
- [ ] Зафиксировать в **комментарии к issue**, что принято (с ссылками на изменённые документы/константы); шаблон — [reference.md](reference.md).

## Фаза C: исправление

Для **информативных** issue эта фаза может **не включать** правок в runtime-коде: достаточно `productRoadmap.ts`, `web/README.md`, локалей и при договорённости команды — заметок в `obsidian-motivator/`. Всё равно пройти **`pre-commit-docs-roadmap`** для коммита с изменениями сводки/README.

- [ ] Минимальный дифф под задачу; стиль проекта; без «заодно» рефакторинга.
- [ ] **Тесты:** добавить или обновить, если есть релевантный слой (`vitest` в `packages/motivator-core` / `web` — по месту).
- [ ] Прогнать релевантные команды (`npm test`, `npm run lint`, `npm run build` в затронутых пакетах) и устранить регрессии.

## Фаза D: закрытие issue (пункт 5)

- [ ] **Инвариант:** фикс по issue **уже в remote** (или смержен в целевую ветку) и это не входит в исключения из [reference.md](reference.md) → **закрыть issue на GitHub** в этой же сессии или сразу после merge, **не откладывая** «на потом» без причины.
- [ ] Если issue **отложен на фазу** плана (см. блок про документацию и фазы): оставить **прозрачный комментарий** в issue (шаблон — [reference.md](reference.md)); не закрывать как «баг исправлен», пока нет реализации по фазе или явного решения команды (закрыть как дубликат плана, «не баг» и т.п.). При отдельном PR только с обновлением плана/сводки — дать ссылку в том же issue.
- [ ] Комментарий в issue: **что** сделано (код и/или **план/сводка**), **где** в репозитории в двух словах, ссылка на **PR или коммит** (после появления в remote).
- [ ] Закрыть issue (`gh issue close`, MCP GitHub после чтения схемы инструмента, или UI) — см. [reference.md](reference.md). **Не** считать заранее, что доступа нет: права считаются выданными пользователем; **только если** попытка в среде дала ошибку (нет `gh`, нет MCP, отказ auth) — дать человеку готовый текст комментария / ссылку «Close with comment» в UI.
- [ ] Если фикс **не** влез в ожидаемый релиз или откладывается — **не** закрывать без метки/комментария и согласования.
