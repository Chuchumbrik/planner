# GitHub defect workflow — справка

## Права GitHub для агента

Для сценариев по **`github-defect-workflow`** и команды **`/defect-squad-pipeline`**: пользователь **предоставил права**, достаточные для работы с GitHub (**`gh`**, MCP GitHub при наличии, операции с issue/PR, push — если сценарий не запрещает). Агент **использует** их сразу, **не** откладывает действие из‑за предполагаемого отсутствия доступа, пока среда не вернула **фактическую** ошибку. Тогда — UI, текст для человека или обходной путь ниже.

## Закрытие issue после фикса

**Инвариант для агента и разработчика:** если по issue **уже доставлен** исправляющий код или согласованные правки плана/сводки (push в remote или merge), работу **не считать завершённой**, пока на GitHub не выполнено одно из: **issue закрыт** (с комментарием и ссылкой на PR/коммит), или **осознанно** оставлен открытым с комментарием по разделу **«Когда не закрывать issue сразу»** ниже. Закрытый PR при **открытом** issue без объяснения — типичный провал процесса.

1. Убедиться, что изменения **в remote** (push) и по возможности CI зелёный.
2. Комментарий в issue (шаблон, адаптировать):

```markdown
**Исправлено:** [кратко суть бага].

**Изменения:** [ссылка на PR] или коммит `https://github.com/OWNER/REPO/commit/HASH`.

**Поведение:** [как проверить вручную в 1–3 шага].
```

3. Закрыть issue:
   - **GitHub UI:** Close issue.
   - **CLI:** из корня репозитория  
     `gh issue close <N> --comment "Закрыто: <кратко>. См. <URL PR или commit>."`  
     Проверка: `gh auth status`.

## MCP GitHub

По умолчанию права на использование MCP считаются **выданными** пользователем. **Перед вызовом** любого инструмента прочитать JSON-дескриптор в `mcps/user-github/tools/` и вызывать строго по схеме. **Использовать** MCP для комментариев, закрытия issue и связанных операций, когда это ускоряет фазу D. **Только если** сервер не подключён, инструмента нет или вызов вернул ошибку авторизации/прав — **`gh`** или UI GitHub по шагам выше.

## Когда не закрывать issue сразу

- Нужно **деплоить** на среду, где воспроизводили баг — оставить открытым до подтверждения или завести подзадачу на выкат.
- Исправление только в **черновике** / локально без merge.
- Баг маскирует **продуктовый разрыв** — закрыть как «не баг» только после явного решения; иначе перевести в discussion / roadmap.
- Симптом относится к **ещё не реализованной фазе** плана MVP — не закрывать как исправленный до работ по фазе; см. основной скилл, блок «Дефект: документация и будущие фазы».

## Комментарий в issue: отложено на фазу плана (шаблон)

```markdown
**Разбор:** [в чём суть с точки зрения пользователя].

**Документация:** дефект пересекается с **фазой N** плана ([`obsidian-motivator/17-План-реализации-MVP.md`](../../../obsidian-motivator/17-План-реализации-MVP.md) / ТЗ / DR …) — **немедленный point-fix** даст [костыль | дублирование | конфликт с заделом].

**Решение:** внести в план / дорожную карту: [что именно добавили или предлагаете добавить в `MVP_PHASES_PLANNED` / `IDEAS_LATER_ENTRIES` / журнал].

**Issue:** оставляем открытым до [условие] **или** закрываем с меткой «перенесено в план» по согласованию с командой.
```

## Комментарий в issue: информативное замечание принято в план / сводку (шаблон)

```markdown
**Тип:** информативное замечание (дорожная карта / порядок фаз / MVP / идея после MVP).

**Принято:** [перестановка фаз | новая идея в `IDEAS_LATER_ENTRIES` | уточнение MVP scope — кратко].

**Изменения в репозитории:** [ссылка на PR] — `productRoadmap.ts` / README / локали / Obsidian по списку.

**Не баг в коде:** поведение приложения не менялось [или: менялись только тексты сводки].
```

## Связь с дефектами из приложения

Issues от **`file-defect`** часто содержат блоки Steps / Expected / Actual и Environment без email/vault. При анализе сверять **`appVersion`**, **`route`**, **`locale`**, **`motivator_role`**.

## Git: перед веткой фикса

Перед **`git checkout -b fix/issue-<N>-…`**: синхронизация с remote — **`git fetch origin`**, переход на **`main`**, **`git pull origin main`** (или merge `origin/main`), чистое дерево или **stash** постороннего WIP. Полный порядок — в [pipeline-agents.md](pipeline-agents.md) → **«Ветка и PR от main»**.

## Два issue — один корневой косяк

Предпочтительно **один PR** с одним коммитом/линией правок и закрытие **обоих** через `Fixes #N, Fixes #M` в описании PR (если принято в команде). Иначе — комментарий с перекрёстными ссылками и закрытие вручную после merge.
