## Нормативная база

### Иерархия

1. **ГОСТ 19.201-78** (ЕСПД) и **ГОСТ Р 59795-2021** — **ориентир структуры** и терминологии.
2. **Правила скилла** (язык, завершённость предложений, измеримость, wikilinks) — **строже** ГОСТа.

Полный текст стандартов в репозиторий **не** копировать. При сомнении в структуре раздела — сверяться с логикой соответствующего ГОСТа и адаптировать под Obsidian-markdown.

### ГОСТ Р 59795-2021 — качество требований

Ориентир для формулировок в таблице требований:

- требование **однозначно** интерпретируется;
- **проверяемо** (есть способ проверить выполнение);
- **выполнимо** в заданных ограничениях;
- **непротиворечиво** другим требованиям и DR;
- **прослеживаемо** через wikilinks и якоря DR.

## Девять критериев качества требований

| Критерий | Краткая проверка |
|----------|------------------|
| **Атомарность** | Одно требование — одна проверяемая мысль; составное — разбить |
| **Завершённость** | Нет «висящих» отсылок; предложение грамматически и по смыслу закончено |
| **Согласованность** | Нет противоречий с DR и другими разделами; конфликт → пользователю |
| **Однозначность** | Одна трактовка; нет «при необходимости», «по возможности» без условия |
| **Выполнимость** | Реализуемо в scope MVP/версии и известном стеке |
| **Проверяемость** | В колонке «Критерий» — наблюдаемый результат или условие «если … то …» |
| **Актуальность** | Соответствует текущим решениям журнала и scope |
| **Корректность** | Факты и термины верны; нет ложных обобщений |
| **Отслеживаемость** | Wikilink на DR или документ-источник; при изменении DR — обновить ссылки |

## Эталон: плохо и хорошо

**Не использовать** формулировки из существующих файлов `obsidian-motivator/` как образец до ревизии по запросу.

### Антипример (нейрослоп)

> **Порядок прохождения:** сначала **выполненные** (радостные анимации / позитивный тон), затем **проваленные** (мягкие КПТ-вопросы).

**Проблемы:** нет подлежащего у «сначала выполненные»; «позитивный тон» без объекта; «мягкие КПТ-вопросы» без действия системы.

### Исправление (эталон)

> Порядок прохождения ритуала завершения дня: сначала показывается эффект от выполненных задач (радостные анимации, позитивный тон информеров и уведомлений), затем — от проваленных (вывод КПТ-вопросов с возможностью ответа на них).

## Глоссарий переменных и методов

**Расположение:** `obsidian-motivator/18-Глоссарий/`

| Элемент | Правило |
|---------|---------|
| **Структура папок** | `переменные/` — поля и свойства; `методы/` — функции и алгоритмы |
| **Имя файла** | Совпадает с идентификатором в коде или модели: `scheduledLocalDate.md`, `tasksPlannedForEodRitual.md` |
| **Индекс** | `00-Индекс.md` — таблица всех записей; обновлять при добавлении |
| **Ссылка из текста** | `[[18-Глоссарий/переменные/scheduledLocalDate\|scheduledLocalDate]]` — **обязательно** вместо голого идентификатора |

### Шаблон страницы переменной

```markdown
# scheduledLocalDate

**Тип записи:** переменная
**Версия:** 1.0
**Дата:** YYYY-MM-DD

## Смысл

Локальная календарная дата (ключ `YYYY-MM-DD`), к которой привязана разовая задача в плане дня пользователя на устройстве.

## Принимаемые типы данных

| Аспект | Значение |
|--------|----------|
| Тип | `string` |
| Формат | `YYYY-MM-DD` (локальный календарь устройства) |
| Обязательность | Для разовой задачи с датой — обязательно; для задачи только в backlog — отсутствует |
| Связанные поля | [[18-Глоссарий/переменные/dateKey|dateKey]] |

## Применение — общее упоминание

- [[09-User-Flow-одного-дня#…]] — планирование задачи на выбранный день
- [[12-Журнал-решений#DR-013]] — правило «план на календарный день» для стрика и EOD

## Применение — техническое

- [[11-Модель-данных-концептуально#…]] — поле сущности Task
- [[16-TZ-MVP-v1.0#…]] — критерий отбора задач в ритуал EOD
```

### Шаблон страницы метода

```markdown
# tasksPlannedForEodRitual

**Тип записи:** метод
**Версия:** 1.0
**Дата:** YYYY-MM-DD

## Смысл

Правило отбора задач, входящих в план на календарный день ритуала End-of-Day: совпадает с логикой вкладки «День» для заданного `dateKey`.

## Принимаемые типы данных

| Параметр | Тип | Описание |
|----------|-----|----------|
| `tasks` | `Task[]` | Полный список задач пользователя |
| `dateKey` | `string` | Ключ дня `YYYY-MM-DD` |
| **Возвращает** | `Task[]` | Задачи плана на `dateKey` (разовые и повторяющиеся по правилу повтора) |

## Применение — общее упоминание

- [[09-User-Flow-одного-дня#…]] — какие задачи показываются в вечернем ритуале
- [[12-Журнал-решений#DR-002]] — состав списков «Сделано» / «Не закрыто»

## Применение — техническое

- [[10-Каталог-функций-и-взаимодействий#…]] — вызов при открытии модалки EOD
- [[08-Архитектура#…]] — слой `@motivator/core`, чистая функция без I/O
```

```markdown
# Глоссарий — переменные и методы

## Переменные

| Идентификатор | Кратко | Страница |
|---------------|--------|----------|
| scheduledLocalDate | Дата разовой задачи в локальном календаре | [[18-Глоссарий/переменные/scheduledLocalDate]] |

## Методы

| Идентификатор | Кратко | Страница |
|---------------|--------|----------|

```

### Пример в тексте документа (хорошо)

> Задача входит в план на календарный день, если [[18-Глоссарий/переменные/scheduledLocalDate|scheduledLocalDate]] совпадает с [[18-Глоссарий/переменные/dateKey|dateKey]] ритуала **или** правило повтора включает этот день; отбор выполняет [[18-Глоссарий/методы/tasksPlannedForEodRitual|tasksPlannedForEodRitual]].

### Антипример

> Задача была активна сегодня по правилу: createdAt или updatedAt или дедлайн.

**Проблемы:** идентификаторы полей не вынесены в глоссарий; нет wikilinks; правило не совпадает с зафиксированным методом отбора.

### Проверка глоссария (дополнение к wikilinks)

1. Страница существует в `18-Глоссарий/переменные/` или `…/методы/`.
2. Заполнены разделы **Смысл**, **Принимаемые типы данных**, **Применение — общее**, **Применение — техническое**.
3. Каждый документ из разделов «Применение» содержит **обратный** wikilink на эту запись.
4. Запись есть в `00-Индекс.md`.

## Проверка wikilinks

Перед финализацией документа:

1. **Файл** из `[[NN-Имя]]` или `[[NN-Имя#якорь]]` **существует** в `obsidian-motivator/`.
2. **Якорь** `#DR-NNN` или заголовок **есть** в целевом файле (заголовок `## DR-NNN — …` даёт якорь `DR-NNN` в Obsidian).
3. **Содержимое** по ссылке **относится** к утверждению в тексте (не ссылка «для галочки»).
4. При **новом DR** — предложить добавить запись в `12-Журнал-решений.md` и при необходимости строку в `00-Главная-MOC.md`.
5. При **новой переменной или методе** — страница в `18-Глоссарий/`, строка в `00-Индекс.md`, обратные ссылки в разделах «Применение».

## Раздел для примеров после ревизии (фаза 2)

По запросу пользователя сюда добавляются пары «антипример → исправление» из реальной ревизии `obsidian-motivator/`. До ревизии единственный встроенный эталон — блок [Эталон: плохо и хорошо](#эталон-плохо-и-хорошо) выше.

### Ревизия DR-001 (2026-05-30)

**Антипример (фрагмент до ревизии):**

> **Стрик ломается** только если пользователь **пропустил End-of-Day** (не открыл приложение вечером в ожидаемом сценарии завершения ритуала).

**Проблемы:** утверждение противоречит собственным условиям успешного дня и [[12-Журнал-решений#DR-013]] — день с EOD, но без выполненных задач при непустом плане тоже не засчитывается; «ожидаемый сценарий» не проверяем; нет явного субъекта у «ломается».

**Исправление:**

> **Неуспешный день** (не входит в цепочку подряд идущих успешных дней) наступает, если пользователь **не завершил** End-of-Day за календарный день **или** в плане на день **были** задачи, но **ни одна** из них не отмечена выполненной (даже при завершённом EOD).

---

**Антипример (фрагмент до ревизии):**

> Для ЦА с СДВГ и выгоранием важно **не наказывать** за провалы задач; критичнее регулярность рефлексии и честность ритуала, чем процент закрытия.

**Проблемы:** «критичнее … чем …» без подлежащего; «провалы задач» размыто (отдельная задача или весь день?).

**Исправление:**

> Целевая аудитория продукта (СДВГ, выгорание) нуждается в модели, которая **не наказывает** за день без запланированных задач. Продукт **приоритизирует** регулярность вечерней рефлексии (End-of-Day) и честное завершение ритуала над процентом закрытия всех задач плана.

### Ревизия DR-002 (2026-05-30)

**Антипример (фрагмент до ревизии):**

> Задача была **активна сегодня** по правилу: **создана сегодня** **или** **изменена сегодня** **или** **имела дедлайн на сегодня**.

**Проблемы:** правило неоднозначно; противоречит реализации и [[12-Журнал-решений#DR-013]] — в коде отбор через **план на календарный день** (`tasksPlannedForEodRitual`), а не по `createdAt` / `updatedAt`.

**Исправление:**

> Задача входит в **план на календарный день** ритуала — **то же правило**, что для вкладки «День» и подсчёта стрика ([[DR-013]]): разовая — `scheduledLocalDate === dateKey`; повтор — вхождение по правилу повтора на `dateKey`.

---

**Антипример (фрагмент до ревизии):**

> **Порядок прохождения:** сначала **выполненные** (радостные анимации / позитивный тон), затем **проваленные** (мягкие КПТ-вопросы).

**Проблемы:** нет подлежащего; «позитивный тон» без объекта; КПТ смешаны с порядком блоков списка задач.

**Исправление:**

> Затем блок **«Сделано»** — задачи плана с главной отметкой выполнения за этот день; блок оформляется **радостной** анимацией и позитивным тоном заголовков и информеров. Затем блок **«Не закрыто»** — задачи плана без главной отметки; при нуле незакрытых модалка подсвечивает блок как успех, иначе применяет **мягкую** анимацию. **КПТ-рефлексия** (3 вопроса из набора шаблонов) — отдельный шаг после обхода списков.

---

### Ревизия DR-003 (2026-05-30)

**Антипример (фрагмент до ревизии):**

> Сочетает структуру матрицы с низким трением входа (inbox), без противоречия с wizard выбора системы.

**Проблемы:** предложение без подлежащего («сочетает» — кто?); «низкое трение» без указания действия пользователя; не зафиксирован конфликт с [[12-Журнал-решений#DR-012]], который заменил модель DR-003 для MVP v1.0.

**Исправление:**

> Матрица Эйзенхауэра задаёт структуру «срочность × важность»; Inbox снижает порог входа при быстром захвате задачи без немедленной классификации. … **Актуальность для MVP v1.0:** [[DR-012]] принимает единую шкалу 1–5; текст DR-003 сохраняет исходное решение и правила миграции v3→v4.

---

**Антипример (фрагмент до ревизии):**

> При выборе системы **Эйзенхауэр**: задача **обязательно** находится в одном из **четырёх квадрантов** **или** в отдельном **Inbox** (без квадранта).

**Проблемы:** нет имени поля данных; «находится» не задаёт проверяемый критерий; Inbox описан как UI-ярлык, а не `eisenhowerQuadrant === null`.

**Исправление:**

> При `prioritySystem: "eisenhower"` у каждой задачи `eisenhowerQuadrant ∈ {q1,q2,q3,q4}` **или** `null` (Inbox).

---

### Ревизия DR-004 (2026-05-30)

**Антипример (фрагмент до ревизии):**

> Через **30 минут после второго пинга** (интервал **настраиваемый**) задача **автоматически** получает статус **Failed** («не сделал»).

**Проблемы:** в модели задачи нет статуса `Failed`; в коде (`applyExpireStaleDoubleConfirm`) снимается только `doubleConfirmPending` без записи выполнения; «второй пинг» как отдельное событие не смоделирован — есть единое окно до `confirmDeadlineIso`.

**Исправление:**

> Если момент позже `confirmDeadlineIso`, система снимает `doubleConfirmPending`; выполнение за день **не записывается** — задача остаётся невыполненной; отдельного поля «Failed» нет.

---

**Антипример (фрагмент до ревизии):**

> Дополнительно: мягкое **push** «Не забыли подтвердить задачу?» через **10 минут после дедлайна второго подтверждения**.

**Проблемы:** push по второму шагу DR-004 в MVP v1.0 **не реализован**; формулировка «после дедлайна» противоречит смыслу напоминания «до автозакрытия».

**Исправление:**

> **Целевое** поведение — push в окне ожидания (каталог функций); в сборке MVP v1.0 отдельный push по DR-004 **не реализован** — явно вынесено в gap / scope уведомлений.
