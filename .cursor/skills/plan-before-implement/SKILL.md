# Планирование перед реализацией

**Обязательное правило:** агент **не начинает писать код**, пока не прошёл фазы **0 → 1 → 2** ниже.
Исключения — только «тривиальные правки» (см. §Исключения).

Связь: детали качества кода — [[engineering-craft]]; доки и договорённости — [[documentation-orientation]].

---

## Фаза 0 — можно ли декомпозировать?

Перед любым «сделай X» задай себе:

1. **X — одна атомарная задача или портфель?** Если в формулировке «и», «плюс», несколько файлов/слоёв/фич — **декомпозируй**.
2. **Есть ли зависимости?** (схема БД → API → фронт; core → web; ADR → vault).
3. **Что можно сделать параллельно, что — строго последовательно?**

**Выход фазы 0:** список подзадач с границами (каждая — один осмысленный PR или один логический блок в ответе):

```markdown
## Декомпозиция
1. [ ] … — слой/файлы; критерий готовности
2. [ ] …
```

Если подзадач **больше одной** — в ответе пользователю **сначала** покажи декомпозицию и план; не прыгай в Implement всего объёма сразу.

**Эвристика «одна подзадача»:** один слой, один модуль, один user-visible сценарий, один гейт/тест-контур закрывается за итерацию.

---

## Фаза 1 — глубокий проработанный план

Для **текущей** подзадачи (или всей задачи, если она атомарна) построй план **до** правок файлов.

### 1.1 Контекст (прочитать, не угадывать)

- [ ] Релевантный код (соседние файлы, существующие паттерны).
- [ ] Документация: `web/README.md`, Obsidian по смыслу, `packages/motivator-core/docs/`, DR в журнале решений.
- [ ] Skills по зоне: [[layer-boundaries-and-ports]], [[vault-and-crypto-invariants]], [[react-ui-conventions]] — что применимо.

### 1.2 План реализации (конкретный)

Минимум:

| # | Шаг | Файлы / артеfact | Риск |
|---|-----|------------------|------|
| 1 | … | … | … |

Плюс явно:

- **Критерии приёмки** (как пользователь/CI поймёт, что готово).
- **Тест-контур:** кто пишет тесты ([[test-contour-orchestrator]]), какие команды.
- **Доки:** README / roadmap / Obsidian — нужно ли ([[pre-commit-docs-roadmap]]).
- **Ручные шаги** вне репо (Amvera, SQL Dashboard) — если есть.

### 1.3 Альтернативы (кратко)

Если решение неочевидно — **1–2 альтернативы** и почему выбран вариант в плане (trade-off в 2–3 предложениях).

**Выход фазы 1:** план, который другой разработчик мог бы выполнить без переспрашивания «а зачем так».

---

## Фаза 2 — перепроверка противоречий

План **не утверждён**, пока не пройден чеклист:

### Против документации и решений

- [ ] Не противоречит `web/README.md`, ТЗ MVP, активным фазам roadmap.
- [ ] Не ломает DR-019 / vault DR-005 без явного ADR.
- [ ] Не дублирует уже реализованное (grep / поиск по репо).

### Против архитектуры

- [ ] Слой и порты — [[layer-boundaries-and-ports]]; нет «Supabase в UI» / «decrypt на сервере».
- [ ] Crypto/vault — [[vault-and-crypto-invariants]], если релевантно.

### Против процесса

- [ ] Scope PR не раздувается (refactor + feature — разделить).
- [ ] Тесты: автор фичи не пишет первичный набор — [[tests-by-independent-agent]].
- [ ] Коммит потребует README/сводку? — заложено в план.

### Против формулировки пользователя

- [ ] План **решает заданную задачу**, а не «удобную соседнюю».
- [ ] Неоднозначности — **вопрос пользователю**, не скрытый допуск в коде.

**Если найдено противоречие:** обнови план или задай вопрос; **не** начинай Implement.

**Выход фазы 2:** одна строка «План согласован с …» или явный вопрос к человеку.

---

## Фаза 3 — только теперь реализация

- Иди **по шагам плана**; отклонение — осознанное (зафиксируй в ответе, почему).
- После кода — [[engineering-craft]] (self-review) и test-contour.
- Если декомпозиция дала несколько подзадач — **закрой текущую**, отчитайся; следующую — снова фазы 0–2 при необходимости.

---

## Исключения (можно без полного плана)

| Ситуация | Достаточно |
|----------|------------|
| Опечатка, одна строка, без смены поведения | кратко: что меняется |
| Явная команда «только ответь / только объясни» | без кода |
| Пользователь дал **готовый пошаговый план** и просит только выполнить | фаза 2 (противоречия) — да; фаза 0–1 — сжато |
| Pure docs без кода | план = список файлов и правок |

Всё остальное — **полный цикл 0 → 1 → 2 → 3**.

---

## Шаблон ответа агента (рекомендуемый)

```markdown
## Декомпозиция
- …

## План
1. …

## Проверка противоречий
- Доки: …
- Архитектура: …
- Процесс: …

## Реализация
(далее — код/правки по плану)
```

Для **малой** атомарной задачи блоки можно сжать до 3–5 буллетов, но **пропускать нельзя**.
