# Инженерное мастерство — как писать код

Этот skill — **мета-уровень**: как думать и писать, чтобы код был достойен лучшей команды в индустрии.
**Сначала** — [[plan-before-implement]] (декомпозиция, план, противоречия); **потом** код по этому skill.
Детали по слоям — [[layer-boundaries-and-ports]], vault — [[vault-and-crypto-invariants]], UI — [[react-ui-conventions]].
Тесты — [[tests-by-independent-agent]] + [[test-contour-orchestrator]].

---

## Мантра (прочитай перед каждой задачей)

> Я не «закрываю тикет строками». Я **улучшаю систему**: читаемость через год, безопасность пользователя,
> предсказуемое поведение, тестируемый дизайн, честная документация. Минимальный diff — не минимальное
> качество. Умный код — тот, который **легко удалить** и **легко проверить**.

---

## Промпт «Перед написанием» (обязательный внутренний диалог)

**Предусловие:** пройдены фазы 0–2 skill [[plan-before-implement]] (декомпозиция, план, противоречия).

Сформулируй себе (можно вслух в плане ответа):

1. **Что меняется для пользователя или контракта?** Одним предложением.
2. **Где живёт логика?** (core / web lib / API / UI only) — см. [[layer-boundaries-and-ports]].
3. **Какие инварианты нельзя сломать?** (vault, auth, i18n, a11y, offline) — перечисли явно.
4. **Как проверить, что готово?** Конкретные команды: `npx vitest run`, сценарий в UI, e2e если риск сквозной.
5. **Что НЕ делаю в этом PR?** Отсечь scope creep — вынести в follow-up.

Если пункт 1 неясен — **спроси**, не кодь.

## Промпт «Во время написания»

Держи в голове:

```
Пишу как инженер, которому завтра ревьюит Staff:

- Сначала читаю существующий код и соседние файлы — стиль, абстракции, ошибки.
- Имена — документация: функция говорит ЧТО, комментарий — только ПОЧЕМУ неочевидно.
- Ветвления — из бизнес-правил, не из формы UI.
- Ошибки — явные типы/коды; не глотаю catch без действия.
- Нет мёртвого кода, TODO без issue, закомментированных блоков «на потом».
- Diff маленький: каждая строка защищает пользователя или упрощает сопровождение.
- Безопасность по умолчанию: секреты, PII, ciphertext — не в логах и не в git.
- Доступность и i18n — не «потом», если трогаю UI.
```

## Промпт «После написания, до сдачи»

Self-review как жёсткий ревьюер:

1. **Прочитай diff глазами новичка** — что непонятно за 30 секунд?
2. **Инверсия:** как сломать эту фичу одной строкой? Есть ли тест на это?
3. **Удаление:** если удалить новую функцию — останется ли код чище? (нет лишних слоёв)
4. **Производительность:** нет лишних O(n²), re-render, full vault upload без нужды?
5. **Команды:** `cd web && npx vitest run` (и core при правках packages) — **зелёные**.
6. **Доки:** README/roadmap/Obsidian — если менялось поведение ([[documentation-orientation]], [[pre-commit-docs-roadmap]]).
7. **Тест-контур:** первичные тесты на новую логику — **не ты**, если ты автор фичи → [[test-contour-orchestrator]].
8. **PR:** перед merge — [[pr-and-code-review]]; нетривиальный diff — [[code-reviewer]] в отдельном контексте.

## Принципы архитектуры (кратко)

| Принцип | Практика |
|---------|----------|
| **Single responsibility** | один модуль — одна причина для изменения |
| **Dependency inversion** | зависим от портов/интерфейсов, не от Supabase/fetch |
| **Explicit boundaries** | core без React; API без UI; UI без SQL |
| **Fail fast** | invalid input → ранняя ошибка с понятным кодом |
| **Boring technology** | проверенные паттерны репо > модная абстракция |
| **Evolution, not revolution** | расширяем контракт; breaking change — осознанно + ADR |

## Принципы качества кода

- **Читаемость > краткость.** Три строки понятнее одной «умной».
- **Копипаста 3+ раз** — только тогда extract (не раньше).
- **Типы — документация.** `unknown` + narrow лучше `any`.
- **Pure functions** где можно — проще тесты.
- **Side effects** — на краях (adapters, providers).
- **Комментарии** — *why*, не *what*; не дублируют типы.

## Тесты (мышление автора, не только гейт)

Гейт [[tests-for-new-code]] ловит **наличие** теста. Мастерство — **смысл**:

- Тест = **спецификация поведения**, не зеркало `if/else` реализации.
- Имена тестов — предложения: «when X, then Y».
- Краевые случаи: empty, null, max, race, permission denied, network fail.
- Один assert-поведение на тест (AAA).
- Флак = баг процесса — фиксить, не `@retry`.

Автор фичи **не** пишет первичный набор тестов на новую логику — [[tests-by-independent-agent]].

## Безопасность и приватность (в каждом PR)

- Секреты — env / Amvera LK, **не** репозиторий.
- Vault/crypto — [[vault-and-crypto-invariants]] без исключений.
- User input — validate length, type, authz (JWT role, owner user_id).
- Admin endpoints — проверка роли на сервере, не только скрытие кнопки.
- Логи — structured, без ciphertext/seed/tokens.

## Рефакторинг vs фича

| Можно в одном PR | Лучше разделить |
|------------------|-----------------|
| rename + мелкий фикс рядом | refactor 500 строк + новая фича |
| extract helper для текущей фичи | смена архитектуры + UX |

Рефакторинг **без** смены поведения — сохраняй зелёные тесты; не требуй «новых» тестов на перестановку.

---

## Когда эскалировать человеку

- Продуктовая неоднозначность (free/paid vault, cutover).
- Breaking change API/core/schemaVersion.
- Новая зависимость с лицензией/весом bundle.
- Компромисс security ↔ UX.

Зафиксируй вопрос; **не** угадывай продукт кодом.
