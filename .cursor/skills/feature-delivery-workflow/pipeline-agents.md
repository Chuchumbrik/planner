## Контекст фичи (шаблон для оркестратора)

Передаётся субагентам (после строки с этапом, если есть):

- **Название / одна строка:** как называем фичу.
- **Цель:** что пользователь получит (поведение, данные, интеграция).
- **Не в scope:** явные границы этой итерации.
- **Источник:** чат / issue #N / Obsidian / другое.
- **MVP и фазы:** что уже зафиксировано в плане; не придумывать «уже сделано» в сводке.
- **Ограничения:** срок, «не пушить», только анализ, без Edge и т.п.
- **Права GitHub/git:** по умолчанию выданы — см. [SKILL.md](SKILL.md).

## Обязательный блок: уточняющие вопросы

**Когда:** в **начале** работы по фиче (до или вместе с первым параллельным разбором), либо сразу после получения «Контекста фичи».

**Кто:** ведущий оркестратор **или** **`feature-product-strategist`** (если он первый отвечает).

**Формат ответа:**

```markdown
## [ВОПРОСЫ] По фиче

### Статус уточнений
`НЕ ТРЕБУЮТСЯ` | `НУЖЕН ОТВЕТ ПОЛЬЗОВАТЕЛЯ`

### Если НЕ ТРЕБУЮТСЯ
Одно-два предложения: почему контекста достаточно для разбора и критериев приёмки.

### Если НУЖЕН ОТВЕТ
1. [Вопрос простым языком]? **Варианты:** A … / B … / своё: …
2. …
3. …

### Зависимости для решения
Какие пункты **обязательно** должны быть решены до `[FEATURE_DECISION]` с веткой реализации (если применимо).
```

Если статус **`НУЖЕН ОТВЕТ ПОЛЬЗОВАТЕЛЯ`** — агрегатор при необходимости выдаёт **`BLOCKED_NEED_ANSWERS`** до получения ответов в чате.

## Параллельный разбор (`ЭТАП: РАЗБОР`)

Запускать **`feature-product-strategist`** и **`feature-system-architect`** параллельно; в **первой строке** промпта: **`ЭТАП: РАЗБОР`**, затем **«Контекст фичи»** и при наличии — **`## [ВОПРОСЫ] По фиче`** с ответами пользователя.

Шаблоны выходов — раздел **«Шаблоны ответов субагентов»** ниже.

## Агрегатор решений по фиче

Субагент **`feature-decision-aggregator`** объединяет `## [PRODUCT] …` и `## [SYSTEM] …` в **`[FEATURE_DECISION]`**:

```markdown
## [FEATURE_DECISION] Фича: <короткое имя>

### Итоговый выбор
`IMPLEMENT_NOW` | `SPIKE_OR_PROTOTYPE_FIRST` | `DEFER_TO_PHASE` | `DOCS_AND_ROADMAP_ONLY` | `POST_MVP_IDEA` | `BLOCKED_NEED_ANSWERS` | `BLOCKED_NEED_HUMAN`

### Краткое обоснование (для таблицы «Почему такое решение»)
2–5 предложений: продукт + техника; конфликты и компромиссы.

### Критерии приёмки (кратко)
Нумерованный список, проверяемый после реализации.

### Следующие шаги для оркестратора
- при `IMPLEMENT_NOW` + push: синхронизация git → ветка от `main` → **ДОКИ_ДО_КОДА** (продукт → система) → UX → FE/BE → QA → PR с `Fixes #…` при наличии issue;
- при spike — только согласованные артефакты / ветка прототипа;
- при `DOCS_AND_ROADMAP_ONLY` — правки roadmap/README/локалей + QA по докам.

### Риски и допущения
…
```

## Документирование до кода (`ЭТАП: ДОКИ_ДО_КОДА`)

При **`IMPLEMENT_NOW`** и боевом пути с коммитами: **после** `[FEATURE_DECISION]` и ветки от актуального `main` — сначала **`feature-product-strategist`** (`ЭТАП: ДОКИ_ДО_КОДА`), затем **`feature-system-architect`** (`ЭТАП: ДОКИ_ДО_КОДА`). Зоны ответственности как у дефект-пайплайна: продукт — `productRoadmap`, продуктовый README, локали; система — техника README, согласованность данных сводки. Скилл **`pre-commit-docs-roadmap`**.

## Ветка и PR от `main` (боевой цикл с push)

**Когда:** реализация с доставкой в remote разрешена.

1. **Синхронизация git (обязательно):** `git status` → при необходимости stash/commit постороннего WIP → **`git fetch origin`** → **`git checkout main`** → **`git pull origin main`** → **`git checkout -b feature/<короткий-slug>`** (или `feat/…` по стилю команды).
2. После **`[QA]` PASS** — push и **PR в `main`**.
3. В **теле PR** — при привязке к issue: **`Fixes #N`** (или `Closes` / `Resolves`).
4. **Ручная настройка** — если нужна вне git: см. шаблон чеклиста в разделе **«Итоговое сообщение»** ниже.

---

## Итоговое сообщение оркестратора (обязательно на любом завершении)

1. **«Суть фичи (кратко)»** — 2–5 предложений.
2. **Таблица** — все строки; неприменимое — **`—`**.
3. **Ручная настройка** — отдельный блок с `- [ ]` только если требуется (как в дефект-пайплайне); иначе в таблице **«не требуется»**.

Шаблон таблицы:

```markdown
## Итог по фиче: <имя>

### Суть фичи (кратко)
…

| Поле | Значение |
|------|----------|
| **Решение** | из `[FEATURE_DECISION]` |
| **Почему такое решение** | 1–4 предложения |
| **Статус реализации** | не начато / в работе / готово в PR / в `main` + пояснение |
| **Что сделано** | коммиты, PR, файлы — кратко |
| **MVP / фаза** | где фича в плане или «вне текущего MVP» |
| **Документация / сводка** | да/нет — где |
| **GitHub** | issue/PR URL или **—** |
| **Ветка / PR** | `feature/…` + URL или **—** |
| **QA** | `PASS` / `FAIL` / не запускался |
| **Ручная настройка** | не требуется / требуется — см. блок ниже |
| **Уточняющие вопросы** | не применялись / закрыты в чате / **ожидают ответа:** … |
```

### Шаблон: ручная настройка (чеклист для человека)

Как в `github-defect-workflow/pipeline-agents.md`: заголовок **«Ручная настройка после выката»**, зачем, ссылка на README при полной процедуре, пункты `- [ ]`, проверка.

|--------|------|

## Шаблоны ответов субагентов

### PRODUCT — `feature-product-strategist`, `ЭТАП: РАЗБОР`

Заголовок: **`## [PRODUCT] Анализ фичи: <имя>`**.

- Ценность для пользователя, **критерии приёмки** простым языком.
- Влияние на **MVP / фазы / сводку**; конфликты с ТЗ — явно.
- **Уточняющие вопросы** (если ещё не выведены оркестратором): дополнить или сослаться на `[ВОПРОСЫ]`.

Без правок файлов в РАЗБОР.

### SYSTEM — `feature-system-architect`, `ЭТАП: РАЗБОР`

Заголовок: **`## [SYSTEM] Анализ фичи: <имя>`**.

- Варианты реализации, затрагиваемые модули (web, Edge, `motivator-core`).
- Инварианты: auth, vault, приватность.
- Риски и зависимости (миграции, флаги, версии API).

Без правок файлов в РАЗБОР.

### UX — `feature-ux-designer`

Заголовок: **`## [UX] Фича: <имя>`** — рекомендации, критерии для FE, a11y, ru/en.

### FE — `feature-frontend-dev`

Заголовок: **`## [FE] Фича: <имя>`** — файлы, тесты, i18n; зона `web/`.

### BE — `feature-backend-dev`

Заголовок: **`## [BE] Фича: <имя>`** — Edge, SQL, RLS; как проверить.

### QA — `feature-qa-tester`

Заголовок: **`## [QA] Вердикт фичи: <имя>`** — покрытие, `PASS`/`FAIL`, кому вернуть при FAIL; ручная настройка — перечислить для итога.
