# Справочник: релиз-ноты и краевые случаи

Дополнение к [SKILL.md](SKILL.md). Читай при сомнениях в формате данных или при нестандартном коммите.

## Структура `RoadmapReleaseNoteItem`

```ts
{
  releasedInVersion: { ru: '0.6.31', en: '0.6.31' },
  changes: [
    { ru: 'Кратко **что** в коде/UX (допустимы бэктики и пути).', en: 'Same in English.' },
  ],
  plainBullets: [
    { ru: 'Что увидит человек без чтения репозитория.', en: 'User-facing summary.' },
  ],
}
```

- **`changes`**: можно несколько элементов массива — отдельные осмысленные изменения одного выпуска.
- **`plainBullets`**: опционально, но **обязательно** для заметного пользовательского эффекта (новая кнопка, изменённый ритуал, новый текст в UI).

## Несколько релизов за один календарный день

Сценарий: утром патч `0.6.30`, вечером ещё патч `0.6.31`, оба в один `dateLabel`.

- Один блок `RoadmapReleaseNoteBlock` с этой датой (или несколько блоков с **одинаковой** `dateLabel` подряд — в UI день склеивается).
- Два (или больше) объекта в **`items`**: первый — более новый semver (как принято в проекте для сортировки в UI).

## Только документация в `.cursor` или `obsidian-motivator`

- Если **нет** изменений в `web/src/**` и версии продукта: можно ограничиться пунктом в **`RELEASE_NOTES_BLOCKS`** за сегодняшний день (техн. / процесс) **и** при необходимости правкой README, если там описан процесс.
- Если правило/скилл меняют **ожидания к коммитам** в веб-пакете — зафиксировать в релиз-нотах за день работ (как внутренний/процессный пункт), чтобы история сводки не расходилась с git.

## Типичные ошибки

1. **Неверная дата**: `dateLabel` = день merge в main, **или** «завтра»/IDE-«сегодня» вместо дня коммита, **или** желаемая дата релиза — неверно; ориентир — **`git log`** и день фактических коммитов с правкой в этот репозиторий.
2. **`releasedInVersion` с `+git`** — неверно для этого поля.
3. **Пустые `plainBullets`** при смене видимого поведения приложения — ухудшает модалку для не-разработчиков.
4. **Коммит только `productRoadmap.ts`** без README при изменении таблицы возможностей в README — если таблицу нужно обновить, сделай оба файла.

## Согласованность с модалкой

- Фазы **«Уже реализовано»** должны отражать реально включённые в сборку вехи; не обещать в `IMPLEMENTED_MVP_PHASES` то, что спрятано за флагом и выключено по умолчанию, без пояснения.
- **«Открытые вопросы»**: тексты в i18n + дублирующий список в `ProductRoadmapModal.tsx` — держать в паре.

## Сообщения коммитов (рекомендация)

- `docs(web): …` — README, roadmap, скиллы, правила.
- `feat(web): …` / `fix(web): …` — всегда вместе с обновлением сводки, если есть пользовательский эффект.
