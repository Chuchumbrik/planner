# Pre-commit: README и «Краткая сводка»

## Обязательность

**Перед выполнением `git commit` или `git push`** (включая «только один файл», «быстрый фикс», merge-коммиты) агент **обязан**:

1. **Прочитать этот `SKILL.md` целиком** (или открыть skill `pre-commit-docs-roadmap` в Cursor), выполнить чеклист ниже и отразить результат **в том же коммите**, который содержит продуктовые правки.
2. Не полагаться только на краткий текст правила `.cursor/rules/pre-commit-docs-roadmap.mdc`: правило задаёт **требование**, этот скилл — **порядок действий и краевые случаи**. Обход скилла при коммите с изменениями вне «только docs-файлы по списку» — нарушение процесса.

Если пользователь просит коммит **без** прохождения чеклиста — напомнить о правиле репозитория; для исключений нужно явное человеческое решение и коммит только разрешённых файлов.

## Цели

- **`web/README.md`**: версия и таблица «Возможности (сводка)» соответствуют сборке.
- **`web/src/data/productRoadmap.ts`**: содержимое модалки «Краткая сводка» не противоречит сборке; релиз-ноты привязаны к календарным дням работ и к semver из `web/package.json`.
- **Локали** `web/src/i18n/locales/ru.json` и `en.json` — при смене продуктовых формулировок в «Открытых вопросах» и связанных ключах.

Подробные примеры, шаблоны формулировок и разбор ошибок: [reference.md](reference.md).

## Шаг 0. Классификация диффа

Определи, что попадает в коммит:

| Сценарий | Минимум действий |
|----------|------------------|
| **Только** `web/README.md`, `web/src/data/productRoadmap.ts`, при необходимости `web/src/i18n/locales/*.json` | Выполни релевантные пункты чеклиста для этих файлов; сверь версию README ↔ `web/package.json`. |
| Любой код/UX в `web/src/**`, пакет **`@motivator/core`**, версия в `package.json`, поведение приложения | README + `productRoadmap.ts` (см. ниже) + локали при необходимости. |
| «Внутреннее» (рефакторинг, тесты, CI, `.cursor`) с **не** видимым эффектом в UI | Всё равно пункт в **`RELEASE_NOTES_BLOCKS`** за **день работ** (`YYYY-MM-DD`): кратко, метка уровня «техн. / внутреннее» или дополнение к блоку этой даты. |

## Шаг 1. `web/README.md`

- [ ] Строка **текущей версии** совпадает с `"version"` в `web/package.json`, если версию меняли или README ещё не синхронизирован.
- [ ] Таблица **«Возможности (сводка)»** и разделы **«Краткая сводка»**, **«Перед коммитом»** отражают поведение после правок (не оставлять противоречий с кодом).
- [ ] Новые процессы (например, скиллы в `.cursor/skills/`) при пользовательской или командной значимости — **кратко** упомянуты там, где читатель ищет «как сопровождать продукт» (раздел про сводку / коммиты).

## Шаг 2. `web/src/data/productRoadmap.ts`

### 2.1 `RELEASE_NOTES_BLOCKS` (главный источник правды для «что изменилось»)

Выполни **все** применимые пункты:

- [ ] **`dateLabel`**: календарный **день коммита(ов)** с этой правкой в репозиторий (`YYYY-MM-DD`); при расхождении с `user_info` — **`git log`**, не «завтра» относительно IDE и **не** дата позже фактического коммита. Поле «сегодня» в user_info — подсказка только если коммит делается в тот же календарный день. Новый день без блока → новый элемент **`RoadmapReleaseNoteBlock`** (обычно **в начало** массива `RELEASE_NOTES_BLOCKS`, чтобы в UI новее было выше).
- [ ] За один день несколько отдельных «выпусков» → несколько объектов в **`items`** этого дня. Внутри одного выпуска несколько правок → несколько строк в **`changes`**.
- [ ] У **каждого** элемента `items` заполнен **`releasedInVersion`** (`ru` и `en`): semver из **`web/package.json` на момент отдачи изменений пользователю**, **без** суффикса `+git` из production-сборки.
- [ ] Для заметных пользователю изменений добавь **`plainBullets`** (раскрывашка «Подробности простым языком»): суть без жаргона; не дублировать дословно весь техтекст.
- [ ] Сортировка в UI внутри дня — по **убыванию semver** в `releasedInVersion`; порядок в массиве `items` должен ей соответствовать (см. комментарий над константой в файле).
- [ ] Не «прячь» старые даты: интерфейс не фильтрует прошлое; не подменяй день ради «красивого порядка», если работа была в другой день; **не** ставь `dateLabel` на **будущий** календарный день относительно дня коммита.

### 2.2 Остальные константы

- [ ] **`IMPLEMENTED_MVP_PHASES`** / **`MVP_PHASES_PLANNED`** — обновить, если изменился реально поставленный или запланированный scope MVP.
- [ ] **`IDEAS_LATER_ENTRIES`** — при новых идеях после 1.0.0; при переносе из Obsidian сверять `obsidian-motivator/15-Идеи-для-развития.md`.
- [ ] Комментарий в шапке файла и типы — по необходимости синхронизировать с правилом (не дублировать простыню, но ключевые инварианты держать актуальными).

## Шаг 3. i18n

- [ ] Если менялись **`settings.roadmapOpenQuestion*`** или другие строки модалки дорожной карты / сводки — обновить **и** `ru.json`, **и** `en.json`.
- [ ] **`ProductRoadmapModal.tsx`**: при добавлении нового открытого вопроса — не забыть список абзацев в коде (см. README).

---

## Шаг 4. Финальная проверка перед `git commit`

- [ ] Нет запрещённой комбинации: изменения в приложении **без** правок README и/или `productRoadmap.ts`, если на пользователя это влияет (см. правило «Запрещено» в `.cursor/rules/pre-commit-docs-roadmap.mdc`).
- [ ] Контент дорожной карты **не противоречит** текущему поведению сборки.
- [ ] Сообщение коммита — нормальный язык; при необходимости одна строка отсылает к обновлённой сводке («docs: … roadmap + README»).

После коммита, если пушишь: убедись, что в пуш не попал коммит только с кодом без сопроводительной документации.

|------|-----------------|
